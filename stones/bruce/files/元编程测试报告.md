## Issues

### MEDIUM-001: activateTrait 对不存在的 trait 静默成功

- 操作: `activateTrait("nonexistent_trait_xyz")`
- 预期: 返回错误或抛异常
- 实际: 返回成功消息，无任何报错
- 影响: 开发者拼错 trait 名称时不会收到任何反馈，可能导致难以排查的"trait 没生效"问题
- 建议: activateTrait 应检查 trait 是否存在，不存在时返回错误

### LOW-001: createTrait 输入输出不对称

- 操作: `createTrait("test", { when: "never", readme: "纯内容" })` → `readTrait("test")`
- 预期: readTrait 返回的 readme 与传入内容一致
- 实际: readTrait 返回的 readme 包含 frontmatter 头（`---\nwhen: never\n---\n纯内容`）
- 原因: when 存储在 readme.md 的 frontmatter 中，readTrait 返回了原始文件内容
- 影响: 不影响功能，但 API 使用者需要自行解析 frontmatter 才能拿到纯内容
- 建议: readTrait 返回的 readme 应剥离 frontmatter，或文档中明确说明

### LOW-002: API 错误处理风格不一致

- 现状:
  - createTrait 无效输入 → 返回 `"[错误] ..."` 字符串
  - readTrait 不存在 → 返回 `null`
  - activateTrait 不存在 → 静默成功（MEDIUM-001）
- 影响: 调用方需要针对每个 API 写不同的错误处理逻辑
- 建议: 统一错误处理约定（推荐：错误返回 `{ error: string }` 或统一返回 null + 错误日志）

## 结论

元编程 API 的核心路径（创建→读取→编辑→列表→激活）工作正常，边界防护到位（无效名称拦截、kernel trait 保护）。主要问题集中在错误处理的一致性上，其中 activateTrait 静默成功是最值得优先修复的。
