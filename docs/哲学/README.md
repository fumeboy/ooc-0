# 哲学 — 元层 · 思想侧

> 这一层回答的问题是：**为什么这样设计**。
> 它是 OOC 其他所有文档的基础——`对象/` 的每一处结构都能在这里找到理由。

## 核心主张

OOC 的设计哲学可以浓缩为一个等式：

```
Object = 存在 ∩ 认知 ∩ 行动 ∩ 成长
```

传统系统中，这四者是分离的：数据库管存在，提示词管认知，函数管行动，没有人管成长。
OOC 中，它们是同一个对象的四个面。改变任何一个面，其他三个面同时改变——因为它们**共享同一个持久化目录**。

## 两个对偶

本目录的组织围绕两组对偶关系：

### 基因 ↔ 涌现

```
基因（微观）              涌现（宏观）
──────────              ──────────
13 条基因     组合      12 条涌现能力
G1-G13        →→→       E1-E12
单个规则                多规则的 emergent behavior
                 ←←←
                 验证
```

- **[genes/](genes/)** — 13 条基因：系统最小的、不可再分的原则集合
- **[emergences/](emergences/)** — 12 条涌现能力：基因组合后"长出来"的高阶能力
- **[emergence.md](emergence.md)** — 涌现总表：验证状态与基因映射

### 展开 ↔ 沉淀

```
展开循环（正向）：
  身份 → 能力 → 计划 → 思考 → 行动
                                    │
沉淀循环（逆向）：                   │
  身份 ← 沉淀 ← 反思 ← 记录 ← 经验 ←┘
```

- **[两个循环.md](两个循环.md)** — 展开循环与沉淀循环的完整描述
- **[统一性.md](统一性.md)** — 为什么四个维度是同一个对象

## 如何阅读

**如果你第一次接触 OOC**：先读 `genes/g01-数据即对象.md`，再读 `emergence.md` 总表，最后根据感兴趣的涌现追溯到相关基因。

**如果你要做设计决策**：读 `统一性.md` + `两个循环.md`，判断新设计是否破坏这两层统一。

**如果你要验证某条涌现**：在 `emergences/<能力>.md` 查看依赖的基因和已做的验证实验。

## 与代码的关系

哲学层没有直接对应的代码目录——它是代码的**约束条件**。
每次基因变更，`对象/` 下对应领域的实现都需要同步更新；反之，实现层的重大重构（如线程树替换认知栈）应该在 `discussions/` 里先行讨论并反写到相应基因。
