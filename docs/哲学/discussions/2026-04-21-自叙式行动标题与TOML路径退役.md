# 自叙式行动标题 + TOML 兼容路径退役

> 讨论日期：2026-04-21  | 状态：已结论（本日落地为代码）
> 相关基因：G5（注意力与遗忘）、G11（UI 即自我表达）
> 发起人：Alan Kay
> 相关迭代：`工程管理/迭代/all/20260421_feature_工具调用title参数.md`

## 问题陈述

### 问题 1：LLM 密集工具调用时，外部观察者难以一眼看懂"现在在做什么"

当对象（Stone）通过 `open/submit/close/wait` 与系统交互时，前端的 TuiAction 卡片只能展示 `toolName(args)` 摘要——这是机器语义，不是面向人的叙述。

Thinking 通道虽然能表达意图，但：
- 不是每次 tool call 都有对应 thinking；
- Thinking 通常在 tool call 之前，跨度较长，难与某一次具体调用一一对齐。

人类用户、其他协作对象（如 Bruce/D1）、supervisor 都会卡在"参数摘要看不出意图"这一步。

### 问题 2：thread 架构内部仍保留着 TOML 解析 + 兼容路径

在 thread-tree 架构上线后，`src/thread/parser.ts` 和 `src/thread/thinkloop.ts` 仍负责把 LLM 的 TOML 文本输出翻译为 thread 指令。这条兼容路径：
- 让 `src/thread/engine.ts` 中 `runWithThreadTree` / `resumeWithThreadTree` 各自多出 300+ 行的 else 分支；
- 意味着同一语义（比如 talk 后的自动 ack）要在两条路径上各写一遍，一旦忘了同步就出现行为不一致的 bug；
- 阻碍了新协议（如 title 参数）的干净落地——如果 TOML 路径继续被保留，就得考虑"TOML 里也给 title 一个对应表示"，徒增复杂度。

## 关键观点

### 观点 A — title 应该进 tool schema，而不是只挂在 thought 里

**论点**：tool schema 是 LLM 必然遵守的结构化协议。把 `title` 作为所有 tool 的顶层参数（open/submit 更 `required`），LLM 就会在每一次调用中**稳定生成**一句话的行动说明。

这与"只在 thinking 里写意图"形成对比：
- thinking 是自由文本，粒度不对齐；
- title 强制的字段，粒度与单次 action 严格对应。

**收益**：
- 外部观察：前端 TuiAction 用 title 作为行首主标题，扫一遍时间线就能读懂整段协作过程。
- 自我对齐：LLM 每次显式复述意图，是一种"低成本的自我监督"——降低 agent 走偏后仍在发力的情况。

### 观点 B — 不考虑旧版本兼容，砍掉 TOML 路径后再建

**论点**：按 `CLAUDE.md` 的"不考虑旧版本兼容"原则，在引入新协议字段时，应先把旧协议的残留清理干净，再在干净的底座上做新设计。

具体操作：
- 删除 `src/thread/parser.ts`、`src/thread/thinkloop.ts` 及其测试；
- 删除 `engine.ts` 中两处 TOML else 分支；
- 在 tool-calling 路径上补齐之前只在 TOML 路径实现的行为（如 talk 的自动 ack），让新旧语义完全统一。

**未一并清理的范围**：`src/world/scheduler.ts` + `src/flow/*` 是**旧 Flow 架构**（thread-tree 之前的调度器）。world.ts 通过 `_useThreadTree` 开关保留双模式。彻底清理这部分意味着几乎重写 world/session 模块——本次迭代判断这已超出范围，作为独立迭代处理。

### 观点 C — title 与 create_sub_thread 的"子线程名" 的命名冲突怎么办

旧 submit schema 里 `title` 字段语义是"create_sub_thread 的子线程标题"。新设计要让 `title` 成为通用的"tool call 行动标题"。

**解法**（已落地）：
- submit schema 同时声明 `title`（顶层行动标题，required）和 `child_title`（子线程名，optional）；
- engine 处理 create_sub_thread 时 `args.child_title ?? args.title`——向后兼容 LLM 的老用法；
- 新的 LLM prompt 教材应引导使用 `child_title`，避免混淆。

这是 **"schema 演化与向后兼容并行"** 的一个案例：新字段先设可选，老字段保留 fallback，等待 LLM 行为收敛后再强制。

## 已达成的共识

1. **所有 tool call 应具备自叙式行动标题**。open/submit 强制 required；close/wait 由于意图自明（关闭/等待），保持 optional（实际 close/wait 本次未加 title 字段，留待观察需求）。
2. **title 的位置是 tool call 顶层，不是 args 子字段**——前端用 `ThreadAction.title` 展示，不从 args 里扒。
3. **TOML 兼容路径立即退役**（thread 模块内部），旧 Flow 架构单独处理。
4. **schema 演化采用渐进式**：create_sub_thread 保留 `title → 子线程名` 的 fallback 通路，教材引导 `child_title`。

## 未决问题

1. **close / wait 是否终将加上 title required**？留待前端观察人类是否真的"看 close 看得懂"。如果 close 也出现"一堆卡片不好追"的情况，再补。
2. **title 会不会催生新的涌现**？LLM 是否会把多个连续 title 串联成"自我叙事"？这需要放到实验（`工程管理/验证/实验/`）中观察。预期观察点：
   - 长对话中 title 是否保持连贯（不跳话题、不与 thought 脱节）；
   - LLM 自己是否会基于前面几条 title 自检（"我上一步说要 X，这一步却 Y，是否跑偏了？"）。

## 与基因的关系

- **G5 注意力与遗忘**：title 是对象对自身行动的"意图摘要"，是主动保留下来的注意力痕迹。未来经验沉淀时，按 title 检索行动历史比按参数检索更自然。
- **G11 UI 即自我表达**：对象在每次 tool call 时显式声明"我现在在做什么"，前端把这一自我表达直接展示给观察者——这是 G11 的一个具体落地。
