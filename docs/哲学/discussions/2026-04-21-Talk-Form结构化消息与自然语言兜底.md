# Talk Form：消息协议支持结构化选项但保留自然语言兜底

> 讨论日期：2026-04-21 | 状态：已结论（本日落地为代码）
> 相关基因：G8（消息投递）、G11（UI 即自我表达）、G6（社交网络）
> 发起人：Alan Kay
> 相关迭代：`工程管理/迭代/all/20260421_feature_talk_form.md`

## 问题陈述

线程树架构下 `talk` 只能发纯文本。但现实场景里，发起方经常心里已有几个候选回复：

- Supervisor 问 user："这个需求按哪种方式实现？A/B/C"
- Bruce 问 user："我遇到阻塞，你倾向于哪条路径？"

纯文本的选项列表有三个缺陷：
1. **接收方需要猜 parsing 规则**——"写 'A' 还是 '方案A' 还是 'a' 可以？"
2. **前端没法渲染交互组件**——只能展示成普通 bubble
3. **LLM 回复容易跑偏**——它可能既没选 A 也没明确"自由文本"，说了一大段模棱两可的分析

我们需要一种"结构化但不强制"的消息协议扩展。

## 关键观点

### 观点 A — 结构化表单是「消息协议的可选扩展」，不是「新消息类型」

我们没有给 talk 再加一个 `talk_form` 原语。**表单只是 talk 消息的可选字段 `form`**：

```
submit(
  form_id=...,
  target="user",
  message="这个需求按哪种方式实现？",
  form={ type: "single_choice", options: [...] }
)
```

字段缺失时，talk 退回为普通纯文本消息——**零破坏的向后兼容**。

这符合 G8（消息投递）的设计原则：消息协议尽量少的核心原语，尽量多的可选 metadata。

### 观点 B — 前端渲染分化，协议层保持透明

用户视角的前端（MessageSidebar）收到带 form 的消息时，**换一种渲染**：

- 没 form → TuiTalk（普通 markdown bubble）
- 有 form → TuiTalkForm（option picker：编号按钮 + 键盘导航 + 自由文本兜底）

后端只负责**正确落盘 form payload + 生成 formId**。前端按 `(threadId, messageId)` 反查到 message_out action 时自然拿到 form 字段，不需要额外 API。

这体现了 G11（UI 即自我表达）：同一份数据，因展示形态不同，呈现出不同的"交互表达"。

### 观点 C — 自由文本兜底永远不关闭

我们显式在 schema 里保留了 `allow_free_text` 字段，**但业务上恒为 true**。不暴露关闭它的配置。

为什么？因为：
1. **用户不可预测**：发起方预设的选项可能都不是用户想要的，强制选会逼出假回答
2. **LLM 的 form 不完备**：LLM 经常遗漏重要选项（"选 A/B" 漏了 C）
3. **哲学坚持**：form 是**降低摩擦的工具**，不是**约束用户的锁**。用结构化辅助，但不用结构化禁锢。

体验上：TuiTalkForm 的"Something else"输入框始终可用，Esc 也能跳过。

### 观点 D — formResponse 必须让 LLM 能机读

用户点完选项后，后端需要把结构化信息传回给发起方 LLM。我们的做法：

**POST /api/talk body 扩展** `formResponse: { formId, selectedOptionIds, freeText }`，server.ts 把它以单行 JSON 前缀注入消息体：

```
[formResponse] {"formId":"form_xxx","selectedOptionIds":["A"],"freeText":null}

米饭（中式米饭套餐）
```

LLM 在 inbox 里看到这样的消息，既能机读选项（第一行 JSON），也能看到人类可读的回复（第二段）。

这是「结构化数据 + 自然语言」并存，不是「结构化 vs 自然语言」二选一。

## 设计取舍

### 为什么不改 world.ts？

迭代约束上本来是"不碰 world.ts"（并行 agent 在动）。但从设计上讲，这个决定**也是对的**：

- formResponse 是**协议层面的辅助前缀**，不是**世界状态的变化**
- server.ts 做字符串预处理就够了——越靠外层处理越容易调试
- 保持 world.ts 和 engine.ts 专注于"对象如何协作"的核心逻辑

### 为什么不让 engine 把 formResponse 作为 inbox message 的结构化字段？

我们类型上扩展了 `ThreadInboxMessage.formResponse?`，但 server.ts 实际走的是"前缀注入 message 正文"路径——LLM 读 inbox.content 时就能看到 formResponse。

原因：
- inbox.content 注入已经满足"LLM 机读"需求
- 额外的 inbox.formResponse 字段只是给前端/工具调试用，本迭代没有调用方
- 字段留着，给未来留接口

### 为什么 formId 由 engine 生成而不是 LLM 给？

- LLM 生成 ID 容易冲突、格式不稳定、偶尔漏掉
- engine 生成：`form_<ts>_<rand>`，天然唯一 + 格式稳定
- LLM 不需要关心 formId 的生成细节（这是协议层事务）

### 为什么不做 `allow_free_text=false` 的硬模式？

- 哲学上坚持"自由文本是用户的基本权利"
- 工程上减少分支复杂度
- 如果未来某场景真需要"必选"——再加新字段，不要复用现有字段做双语义

## 落地

1. **协议层**（`kernel/src/thread/tools.ts`）：submit tool args 增加 optional `form`
2. **类型层**（`kernel/src/thread/types.ts`）：新增 `TalkFormPayload` / `FormResponse`；`ProcessEvent` / `ThreadInboxMessage` 扩展 `form?` / `formResponse?`
3. **Engine**（`kernel/src/thread/engine.ts`）：`extractTalkForm` 标准化 args.form；talk 分支生成 formId 写入 action.form
4. **Server**（`kernel/src/server/server.ts`）：`POST /api/talk` body 解析 `formResponse`，注入 `[formResponse] ...\n\n` 前缀
5. **前端组件**（`kernel/web/src/components/ui/TuiTalkForm.tsx`）：option picker 实现
6. **集成**（`kernel/web/src/features/MessageSidebar.tsx`）：timeline 匹配 form，已提交态持久化到 localStorage
7. **Trait 文档**（`kernel/traits/talkable/TRAIT.md`）：新增 form 使用章节 + 代码示例
8. **用例验证**：supervisor 发 form → user inbox 索引正确 → 模拟 formResponse 回复 → LLM 正确识别选项 A

## 涌现预期

- **对象主动用 form 降低协作摩擦**：有候选答案时倾向用 form，协作效率提升
- **user 的回复变结构化**：点选项后 inbox 里是 `{selectedOptionIds:["A"]}`，不再是"A 啊""方案 A""第一个"三种表达
- **前端可扩展**：后续可加 text_input / rating / date_picker 等 form type，不改协议骨架
