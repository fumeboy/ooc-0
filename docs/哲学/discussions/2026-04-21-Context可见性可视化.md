# Context 可见性可视化 — G5/G13 的具象化

> 讨论日期：2026-04-21  | 状态：已结论（实现为 feature）
> 相关基因：G5（注意力与遗忘）、G13（线程树）
> 发起人：Alan Kay

## 问题陈述

在 OOC 中，每个线程的 Context 由 `context-builder.ts` 基于线程在线程树中的位置构造：
完整 actions（自身）+ 祖先链摘要 + 直接子摘要 + 同级兄弟摘要。

但这种"看得到什么"的结构对外部观察者（debug 中的自己、用户）是**隐形的**——
必须心算线程树 + 重走一遍 context-builder 的三个 render 函数，才能回答
"当前 focus 线程的世界边界到底在哪里"。

## 为什么这很重要

"可见性"是 OOC 两条核心基因的直接体现：

- **G5（注意力与遗忘）**：Context 不是一段永不增长的文本，而是按需组装的"当前视野"。
  不可见的东西从不出现在线程的世界里 —— 这是遗忘的结构形态。
- **G13（线程树）**：每个线程的视野由它在树中的位置决定，而不是由全局状态决定。
  这是组件化的认知空间。

过去这两条基因在代码里成立、在 Context 里生效，但在 UI 里**看不见**。
观察者只能看到"线程树结构"，看不到"视野边界"。

## 方案

1. **后端分类器**：`kernel/src/thread/visibility.ts#classifyContextVisibility`
   给定 focus 节点 ID，返回每个节点的四类分类：
   - `detailed`：focus 自身（process 区段完整 actions）
   - `summary`：祖先 / 子 / 兄弟，拥有 `summary` 字段
   - `title_only`：祖先 / 子 / 兄弟，没有 `summary` 字段
   - `hidden`：其他所有节点
2. **HTTP endpoint**：`GET /api/flows/:sessionId/objects/:name/context-visibility?focus=:threadId`
3. **前端视图**：`ThreadsTreeView` 新增 "Ctx View" 切换按钮：
   - 整棵树按 focus 视角着色（独立于 status 圆点色）
   - 顶部四色图例 + focus 名
   - 单击节点 → 切换 focus → 全树重算

## 一致性铁律

分类器规则**必须**对齐 `context-builder.ts` 的三个 render 函数：

| Render 函数 | 收集范围 | 可见性 |
|---|---|---|
| `renderAncestorSummary` | `getAncestorPath(..., focusId).slice(0, -1)` | summary / title_only |
| `renderChildrenSummary` | `focus.childrenIds` | summary / title_only |
| `renderSiblingSummary` | `parent.childrenIds \ focus` | summary / title_only |
| （其他） | —— | hidden |

这意味着：

- **uncle（父节点的兄弟）不可见** —— siblingSummary 只看 focus 的父的直接子，不看 focus 的祖父的其他子
- **孙节点不可见** —— childrenSummary 只深入一层
- **堂兄弟不可见** —— 他们是 uncle 的子，uncle 已经不在 Context 里了

这条边界对外部观察者**反直觉**（人类习惯"家族图谱"，会觉得 uncle 也能看得到），
但正是 OOC 的设计选择：**让视野最小化**，让认知栈真正隔离。可视化把这条设计
原则从"必须读代码才能知道"变成"用眼睛看就懂"。

## 结论

该 feature 已实现并合入主线，测试 20 pass。
后续若 `context-builder.ts` 的收集规则调整（如支持 scope chain 深度限制），
`visibility.ts` 必须同步更新——测试文件中的"可见集合 = 祖先+子+兄弟"断言会
守护这条一致性。
