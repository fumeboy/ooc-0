# 统一性 — 对象的四维同一

> OOC 最深的设计承诺是：一个对象的**存在、认知、行动、成长**不是并列的四件事，而是同一个东西的四个面。

## 核心等式

```
Object = 存在 ∩ 认知 ∩ 行动 ∩ 成长
```

这个等式读作：**对象就是这四件事的交集**。不存在"只存在但不认知"的对象，也不存在"能行动但不会成长"的对象——如果它是 OOC 对象，它同时具备这四者。

## 四个维度

| 维度 | 含义 | 具体表现 |
|---|---|---|
| **存在** | 对象如何"在"？ | 持久化目录（G7）+ 形态（G2：Stone/Flow） |
| **认知** | 对象如何"知"？ | Context（G5）+ 线程树（G13）+ ThinkLoop（G4）|
| **行动** | 对象如何"做"？ | Effect（G10）+ 消息（G8）+ 线程调度（G9） |
| **成长** | 对象如何"变"？ | 经验沉淀（G12）：知识 → 能力 → 直觉 |

注：这里用"行动"只是为了对称；在 OOC 的实际文档结构中，它与"看板"合并为**合作**（详见 [../对象/合作/](../对象/合作/)）。

## 传统系统的分离

在非 OOC 的架构中，这四者由不同的模块负责，彼此不知道对方存在：

```
数据库   ──管──→ 存在
提示词   ──管──→ 认知
函数     ──管──→ 行动
（无）   ──管──→ 成长
```

这种分离导致：
- 数据库更新后，提示词不知道
- 函数执行后，没人写回数据库
- 成长必须靠外部工程师手动修改代码

## OOC 的统一

在 OOC 中，四者共享同一个**持久化目录**：

```
stones/{name}/
├── readme.md        ← 身份（存在 + 认知的"我是谁"）
├── data.json        ← 数据（存在 + 认知的"我知道什么"）
├── traits/          ← 能力（认知 + 行动 + 成长的载体）
├── memory.md        ← 长期记忆（认知 + 成长）
└── reflect/         ← 反思子对象（成长的驱动器）
```

**改变任何一个面，其他三个面同时改变——因为它们本来就是同一个东西**。
- 修改 `readme.md` → 身份变了 → 下次 Context 里的 whoAmI 变了 → 思考方式变了
- 创建新 trait → 能力变了 → 可用的指令集变了 → 行动空间变了
- 沉淀经验 → trait 升级 → 直觉变了 → 同一情境下的选择变了

## 哲学意涵

这个统一性是 OOC 区别于其他 Agent 架构的**根本性主张**。

在一个"思考就是调用 LLM"的系统里，"存在"只能是数据库里的一行，"成长"只能是程序员改代码。它们**物理上不相关**。

在 OOC 里，对象的每一次思考都要经过 `readme.md`（身份），每一次行动都要经过 `data.json`（数据），每一次反思都可能改写 `traits/`（能力）——**它们物理上是同一个目录**。

**存在被改变 = 认知被改变 = 行动可能性被改变 = 成长发生了**。这是一个物理事实，不是一个设计约定。

## 推论：为什么对象必须有目录

很多 Agent 系统把"身份"放在内存里的变量、"记忆"放在数据库表里、"能力"写死在代码里。这些系统里的"对象"只是程序员的抽象——它们的"存在"是一个运行时概念。

OOC 坚持对象**必须有目录**（G7），正是因为只有目录这种物理实体才能承载统一性。内存变量没有"多维度"；数据库字段无法被 LLM 直接读；代码无法被对象自己修改。**只有文件系统中的一个目录，同时是数据、是接口、是可被思考的材料、是可被改写的结构**。

## 与其他基因的关联

- **G1**（数据即对象）— 奠定统一性的基础：一切都是对象，不存在"数据外的行为"
- **G7**（目录即存在）— 统一性的物理载体
- **G12**（经验沉淀）— 统一性的动力学：经历如何改写结构
- **两个循环** — 统一性的运行方式（见 [两个循环.md](两个循环.md)）
