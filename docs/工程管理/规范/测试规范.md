# 测试规范

> 测试是沉淀循环的入口——经过测试验证的行为才有资格进入沉淀。

## 两种测试

OOC 项目实践两种测试：

| 类型 | 目的 | 工具 | 谁做 |
|---|---|---|---|
| **单元测试** | 验证代码逻辑 | `bun test` | 代码作者 |
| **体验测试** | 验证用户视角 | spawn Bruce agent | 独立视角 |

两者**都必要**——缺了任何一种都会漏掉关键问题。

## 单元测试

### 位置

```
kernel/tests/
├── thread/
│   ├── context-builder.test.ts
│   ├── scheduler.test.ts
│   └── ...
├── persistence/
│   └── ...
└── ...
```

### 运行

```bash
cd kernel/
bun test
```

### 覆盖原则

- **核心逻辑**必须有测试（如 Engine、Scheduler、Context Builder）
- **边界条件**必须有测试（nil、空数组、超大输入）
- **并发场景**必须有测试（serializedWrite、线程复活）

### TDD 循环

```
RED   — 写一个失败的测试
GREEN — 最简实现让测试通过
IMPROVE — 重构，测试保持 pass
```

TDD 不是可选。核心逻辑**先写测试**。

### 0 fail 规则

提交前必须 0 fail。不要用 `.skip` 绕过失败测试——**失败的测试说明代码或测试有问题，必须搞清楚**。

## 体验测试

### 为什么需要

单元测试验证"代码对不对"。体验测试验证"**用户感受对不对**"。

例：
- 单元测试：`createIssue` API 返回正确的 issue 对象 ✓
- 体验测试：用户点"新建 Issue"，界面能否引导他填清楚、提交后是否看到反馈 ?

单元测试不能代替体验测试。

### Bruce Agent

Bruce 是**用户视角的观察者**。spawn 方式：

```
请扮演新用户，不看代码只看 UI 和行为，体验新功能 X。
报告：
- 能用吗？
- 有没有意外行为？
- 界面是否清晰？
- 什么需要说明但没说明？
```

Bruce 不该看到内部实现——他只从用户角度说话。

### 体验测试的时机

- **新功能完成后**：Step 8 体验验证
- **重大重构后**：验证用户流程未被破坏
- **Release 前**：整体体验检查

### 体验测试不做的事

- **不验证代码正确性**（那是单元测试的事）
- **不深入内部细节**（保持用户视角）

## 测试数据

### 不要 mock 核心逻辑

```
❌ 测试 Context Builder 时 mock 整个 ThreadTree
✅ 用真实的 mini ThreadTree，只 mock LLM 调用
```

mock 太多 → 测试通过但真实代码不 work。

### 用例数据

```
kernel/tests/fixtures/
├── stones/
│   └── test-object/
│       ├── readme.md
│       └── data.json
└── ...
```

测试用的 fixture 放在 `tests/fixtures/`，避免污染真实数据。

## 性能测试（可选）

对**性能关键**的路径做性能测试：

- Context 构建时间（应 < 200ms）
- 调度 tick 时间（应 < 50ms）
- 序列化写入吞吐（> 100 ops/s）

可选——当性能成为问题时补上。

## 测试与反模式

### 反模式 1：测试写配合实现

TDD 的价值是"先写测试让你想清楚要什么"。如果**先写实现再补测试**，容易写出"测试跟着实现走"的代码——只要代码改，测试也跟着改，永远 pass。

**正确**：RED → GREEN → IMPROVE。

### 反模式 2：mock 到 mock 测 mock

如果测试里一半以上是 mock，考虑：
- 架构是否过度耦合？
- 是否该直接写集成测试？

### 反模式 3：测试跑得慢

快速的测试（几秒内跑完全部）鼓励频繁跑；慢测试（几分钟）让人"跳过"。

如果测试慢：
- 减少 I/O（不要每个测试都写磁盘）
- 并行跑（bun 原生支持）
- 拆分 unit vs integration

## 与基因的关联

- **G10**（行动记录不可变）— 测试是 actions 的"验证层"，确保记录与预期一致
- **G12**（经验沉淀）— 通过测试的行为才有资格沉淀
- **verifiable trait**——代码世界的 `when_finish` 门禁；测试规范是人类世界的同类约束
