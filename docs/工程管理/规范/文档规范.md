# 文档规范

> 本次重构后的文档组织原则、写作风格、交叉引用格式。

## 目录结构

```
docs/
├── meta.md                   ← 全局概念树（唯一入口）
├── README.md                 ← 一页导航
├── 哲学/                     ← 元层 · 思想
├── 对象/                     ← 本体层
└── 工程管理/                 ← 元层 · 实践
```

详见 `docs/README.md`。

## 写作原则

### 1. 一个概念一个文件

每个 md 文件对应一个**独立概念**。不要把多个不相关的概念塞进一个文件。

判断"是否应该拆分"：
- 文件长度 > 500 行 → 通常该拆
- 内部标题 `##` 之间的跳跃逻辑很大 → 该拆
- 两个 `##` 小节互不引用 → 该拆

### 2. README.md 作为导航

每个目录下的 README.md **仅作为目录索引**，不是内容主体。

好的 README 包含：
- 这个目录是什么
- 每个子文件 / 子目录的一句话介绍
- 阅读顺序建议

不好的 README：
- 把子文件的内容都写在 README 里
- 没有清晰的"去哪里读什么"

### 3. 源码锚点必备

每份**讲实现**的文档末尾必须有**源码锚点**小节：

```markdown
## 源码锚点

| 概念 | 实现 |
|---|---|
| X 类型 | kernel/src/types/x.ts |
| X 加载 | kernel/src/x/loader.ts |
| Y 处理 | kernel/src/x/engine.ts |
```

让读者能快速从文档跳到代码。

### 4. 基因关联必备

每份**在对象/下**的文档末尾必须有**与基因的关联**小节：

```markdown
## 与基因的关联

- G1（数据即对象）— 本章的基础
- G5（Context 即世界）— 某机制的依据
```

让读者能快速理解"这个概念源于哪些基因"。

## 交叉引用

### 相对路径链接

跨目录引用用**相对路径**：

```markdown
详见 [../../哲学/genes/g01-数据即对象.md](../../哲学/genes/g01-数据即对象.md)
```

好处：
- 文档可以整体迁移
- IDE 支持跳转

坏处：
- 重构目录结构时需要更新路径

### @ref 标签（可选）

某些情况下用 `@ref` 注释更简洁：

```markdown
<!--
@ref docs/对象/认知/线程树/README.md — 线程树架构
@ref kernel/src/thread/tree.ts — 实现
-->
```

这让不支持相对路径跳转的阅读器（如 grep）也能看到引用。

### [[双向链接]]（Obsidian 风格）

在**哲学/** 下部分文件使用。让 Obsidian 等工具能建立知识图谱。

## 格式约定

### 5. 标题层级

- `#` 用于**文件标题**（全文唯一）
- `##` 用于主要章节
- `###` 用于小节
- 尽量不用 `####` 及以下（太深的层级说明该拆文件）

### 6. 代码块语言标签

```markdown
```typescript
// 代码
```
```

标签用途：
- 语法高亮（渲染器）
- 分类（grep 时用 `language=typescript` 过滤）

### 7. 表格对齐

表格用于**结构化对比**：

```markdown
| 维度 | 选项 A | 选项 B |
|---|---|---|
| 速度 | 快 | 慢 |
| 占用 | 低 | 高 |
```

不用于**长文本列表**（表格里文字换行难看，不如用列表）。

### 8. 代码引用内联

文件名、类型名、函数名用 \`backticks\` 包围：

```markdown
在 `kernel/src/thread/engine.ts` 中，`handleToolCall()` 处理...
```

## 文件命名

### 9. 中英混合

- **中文名**：面向人类读者的文档（如 `九大组成.md`）
- **英文名**：对应代码的文档（如 `reflect-flow.md`、`engine.md`）
- 数字开头用于**编号**（如 `g01-xxx.md`）

### 10. 避免空格

文件名用**连字符 `-`** 或 **下划线 `_`**，不用空格（命令行处理友好）。

## 特殊文件

### 11. meta.md

全局概念树。**不动它的结构**——只修正与代码不一致处。

### 12. README.md

每个目录**必须**有 README.md。除非目录只有一个文件（如 `哲学/discussions/` 暂只有 README）。

## 文档与代码同步

### 13. 提交前检查

```
改了代码 → 检查文档
```

每次 commit 前问自己：
- 我改的代码在哪些文档里被提到？
- 这些提及是否还准确？
- 如果不准确，一起改了吗？

**没改** → 这个 commit 不完整。

### 14. 删代码时也要删文档

删除某个 API / trait / 概念时：
- 直接删对应文档（不要保留"已废弃"警告）
- 删除所有引用（grep 确认）
- meta.md 里的引用也要清理

## 与 OOC 本身的对称

文档结构和 OOC 的对象结构**同构**：

| 文档 | OOC 对象 |
|---|---|
| 目录结构 | 对象目录结构 |
| README.md | readme.md |
| 子文件 | data / traits / memory |
| 跨目录引用 | 对象间的 relations |

**我们用 OOC 的方式组织文档**——这让文档本身是 OOC 哲学的一个佐证。
