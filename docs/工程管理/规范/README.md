# 规范 — 按什么标准做

> 工程管理元循环的**约束层**。已沉淀的做事方法。

| 文档 | 内容 |

## 为什么叫"规范"而非"规则"

**规则**：强制遵守，违反会被打回。
**规范**：推荐做法，有**理由支持**。违反也行，但需要说明"为什么这次不遵守"。

OOC 的规范是**已沉淀的经验**——它们曾经是某个具体情境下的经验教训，被抽象为"通用建议"。在新情境下可能不适用，但应该先默认遵守，有特殊理由再偏离。

## 规范的维护

### 新增规范

发现某个反模式反复出现 → 形成规范：

```
多次看到 "LLM 跑通测试但没跑就说完成"
  ↓
形成 verifiable trait（代码层的约束）
  ↓
同步到 docs/工程管理/规范/代码规范.md（人类层的约束）
```

### 淘汰规范

如果某规范被证明**成本 > 收益**，应该**移除**：

```
规则："所有函数必须有中文注释"
  ↓
实际：简单 getter 加注释反而增加噪音
  ↓
调整：规则改为"复杂函数必须有中文注释"
```

## 与基因的关联

规范是 G12（经验沉淀）在**工程管理层面**的表达：

- 知识：某次发现 X
- 能力：反复验证后形成 "遇到 X 就做 Y" 的规范
- 直觉：规范被内化到团队习惯，不需要主动查阅

OOC 的 trait 系统和本目录的规范系统**同构**——前者是对象的规范，后者是团队的规范。
