# 代码规范

> 源于 `user/CLAUDE.md` 中的 "工作约定" 部分，扩展细化。

## 核心约定

### 1. 中文注释

所有文件头部、函数、参数、字段**必须有清晰的中文注释**。

**动机**：本项目是中文团队的作品，中文注释让新人快速理解。英文注释要求"翻译"，增加阅读成本。

**例**：

```typescript
/**
 * 构建 Context 数据。
 * 从线程树当前节点沿作用域链向上收集 trait 和知识。
 */
export async function buildContext(threadId: string): Promise<ContextData> {
  // ...
}

interface ThreadData {
  /** 线程唯一 ID，格式 "th_xxxxxx" */
  id: string;
  /** 父线程 ID，根线程为 null */
  parentId: string | null;
  /** 当前状态 */
  status: "running" | "waiting" | "done" | "failed";
}
```

### 2. 测试驱动（TDD）

核心逻辑**必须编写配套的单元测试**，提交前确保所有测试通过。

**TDD 循环**：

```
RED   — 写测试，期望失败
GREEN — 最简实现让测试过
IMPROVE — 重构，测试保持 pass
```

**通过的标准**：

```bash
cd kernel/
bun test
# 所有测试 pass，0 fail
```

**不要**：
- 跳过失败的测试（`.skip` / `.only`）
- 在测试中用 mock 绕过核心逻辑
- 事后补测试（容易写"配合实现"的测试，失去 TDD 价值）

### 3. 及时更新文档

编写或修改代码后，**立即更新相关文档**。

**规则**：改了代码但没改文档，这个 commit 不完整。

**例**：
- 改了 `kernel/src/thinkable/engine/engine.ts` 的 tool 处理 → 检查 `docs/对象/人机交互/ooc-protocol.md`
- 新增 kernel trait → 更新 `对象/结构/trait/kernel-traits/`
- 改了 API 字段 → 更新对应文档 + meta.md 中的引用

### 4. 不考虑旧版本兼容

**永远以最优雅的代码方式实现新功能**。

**动机**：
- OOC 还在早期，没有"稳定 API"概念
- 兼容代码会快速腐化
- 读代码的人看到 "// v1 兼容" 的分支会困惑

**但**：
- 迁移期可以短暂保留切换策略作为过渡手段，但一旦切换完成，立即删除分支（示例：线程树架构迁移期曾有 `OOC_THREAD_TREE=0` 回退，2026-04-21 旧 Flow 退役后一并移除）
- 清理旧代码时要确保所有调用处已迁移

## 架构风格

### 5. LLM 做判断，代码做记账

**复杂决策交给 LLM，代码只做数据管理**。

**例**：
- 决定"是否值得沉淀 memory.md" → LLM 判断
- 写入 memory.md → 代码机械执行

**反例**（错的）：
```typescript
// 用正则/关键词匹配判断"是否值得沉淀"
if (message.includes("重要") || message.length > 100) {
  saveToMemory(message);
}
```

**正确**：
```typescript
// 把判断交给 LLM，代码只执行
const shouldSave = await llm.judge("这个值得沉淀吗？", message);
if (shouldSave) saveToMemory(message);
```

### 6. 示例 > 规则

**给 LLM 看具体例子比抽象描述有效 10 倍**。

**例**：

❌ 抽象：
```
trait 的 TRAIT.md 应该结构清晰、层次分明。
```

✅ 具体：
```
trait 的 TRAIT.md 遵循以下模板：
---
name: ...
description: ...
---
# 能力名

## 你能做什么
[列出主要 API]

## 何时使用
[触发场景]

## 示例
[完整代码示例]
```

这一原则同样适用于：
- Trait 设计
- 文档写作
- System prompt 构建

### 7. 最小改动

**不做不需要的改进，不加不需要的抽象**。

**例**：
- 修 bug → 只改造成 bug 的那几行
- 不要顺手重构周边代码（除非明确同步）
- 不要为"未来可能需要"添加抽象层

**动机**：
- 改动越大，reviewer 越难 review
- 被动重构容易引入意外 regression

### 8. 最小化条件分支

**过多 if-else 是设计没想清楚的信号**。

**例**：

❌ 条件分支爆炸：
```typescript
if (type === "command") {
  if (command === "program") { ... }
  else if (command === "talk") { ... }
  else if (command === "return") { ... }
} else if (type === "trait") { ... }
```

✅ 用 registry：
```typescript
const commandHandlers: Record<string, CommandHandler> = {
  program: handleProgram,
  talk: handleTalk,
  return: handleReturn,
};

if (type === "command") {
  await commandHandlers[command]?.(args);
}
```

### 9. 源码路径按能力表达

源码路径应该优先表达 OOC 具有什么能力，而不是只表达实现细节或历史归属。

**动机**：读目录时应该能直观看见系统能做什么，并能从能力反向找到相关代码。路径本身就是架构说明的一部分。

**风格**：
- 顶层目录回答"这类代码让 OOC 具备什么能力"
- 二级目录回答"这个能力下有哪些子能力"
- 可枚举的同类能力用"目录 + 一个能力一个文件 + index.ts 聚合"

**推荐的顶层能力词**：

```
kernel/src/
  app/            # 启动、CLI、运行装配
  executable/     # 执行：tools、commands、program sandbox、forms、protocol
  thinkable/      # 思考：LLM、context、engine、thread tree
  storable/       # 存储：stone、session、thread、memory、edit plans、inbox
  collaborable/   # 协作：talk、relation、super、kanban、inbox
  extendable/     # 扩展：trait、knowledge、skill、activation
  observable/     # 观察：HTTP/SSE、debug、visibility、test runner
  object/         # 对象模型和身份
  world/          # World 根对象和对象发现
  shared/         # 跨能力共享类型、工具和集成
```

**二级目录示例**：

```
kernel/src/executable/
  tools/
    open.ts
    refine.ts
    submit.ts
    close.ts
    wait.ts
    index.ts
  commands/
    program.ts
    talk.ts
    do.ts
    compact.ts
    plan.ts
    index.ts
  sandbox/
  forms/
  protocol/
```

**规则**：
- 每个能力一个同名文件，目录列表就是能力清单
- `index.ts` 只做类型导出、统一聚合、排序/注册，不承载单项能力实现
- 新增能力时必须新增对应文件，并在 `index.ts` 注册
- 如果一个文件位置不好判断，先问"它服务哪种能力"，再决定放入哪个能力目录

**例**：新增一个 command 时，优先新增对应文件并在聚合入口注册。

```
kernel/src/executable/commands/
  program.ts
  talk.ts
  do.ts
  index.ts
```

这样读者无需打开一个大表，就能直接从文件目录看出系统当前具有哪些 commands。

## 命名约定

### 10. 中文命名权衡

OOC 允许文件名和目录名用**中文**：

- 好处：团队阅读友好
- 代价：命令行 / 跨平台略麻烦

**规则**：
- 目录名可以用中文（如 `docs/哲学/`、`docs/对象/`）
- 文件名可以中文 / 英文混合（选**更能表达语义**的）
- 代码标识符（函数名、变量名、类型名）**必须英文**

### 11. 具体 > 抽象

变量名、函数名尽量**具体**：

❌ 抽象：`process()`, `handle()`, `data`
✅ 具体：`buildContext()`, `handleToolCall()`, `threadData`

## 提交约定

### 12. 提交信息格式

```
<type>: <subject>

<body（可选）>
```

type：feat / fix / refactor / docs / test / chore

subject：简短（< 50 字符），描述**做了什么**。

body（可选）：详细说明 **为什么**、**如何**、**影响范围**。

### 13. 一个 commit = 一个逻辑改动

不要把不相关的改动混在一个 commit。这让：
- 回滚更精准
- review 更高效
- 历史更可读

## 例外

每条规范都有"可偏离"的场景——**只要你能说清为什么**。

如果发现某条规范**频繁偏离**，考虑更新这条规范。
