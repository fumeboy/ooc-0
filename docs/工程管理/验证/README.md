# 验证 — 做对了吗

> 工程管理元循环的**反馈层**。有了验证才有沉淀。

## 两个子目录

| 目录 | 内容 | 性质 |
|---|---|---|
| [用例/](用例/) | 16 个端到端体验用例 | 常驻，持续维护 |
| [实验/](实验/) | 探索性实验记录 | 历史快照，归档为主 |

## 为什么"验证"是独立分支

工程管理的其他分支（目标、组织、流程、规范）都是**做什么 + 怎么做**。
验证回答的是**做得怎么样**——是闭环的最后一步。

没有验证 → 没有真实反馈 → 沉淀的经验可能是幻觉。

## 两种验证的互补

### 用例（常驻）

- **什么**：典型场景的端到端流程
- **粒度**：一个场景 = 一个目录
- **用途**：每次大改动后 checklist，防止 regression

### 实验（一次性）

- **什么**：探索性问题的答案
- **粒度**：一个实验 = 一份报告
- **用途**：验证涌现能力、验证哲学主张、探索未知

## 验证与沉淀

```
做事 → 验证 → 有效？
             ↓
           有效 → 沉淀（写入 memory / trait / 规范）
             ↓
           无效 → 调整或放弃
```

**没有通过验证的主张不应沉淀**——这是 verifiable trait 的精神在工程管理层的体现。

## 与基因的关联

- **G12**（经验沉淀）— 验证是沉淀的前置
- **G10**（行动记录不可变）— 用例和实验的结果是不可改的历史
