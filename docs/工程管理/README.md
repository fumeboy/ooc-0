# 工程管理 — 元层 · 实践侧

> 这一层回答的问题是：**我们如何做 OOC**。
> 与 `哲学/`（为什么这样设计）对称——一个是思想元层，一个是实践元层。

## 核心主张

**我们用 OOC 的方式做 OOC**。

工程管理本身是一个"Object"：它有身份（目标）、结构（组织）、认知（流程 + 规范）、合作与成长（验证）。

## 五个分支

```
目标 → 组织 → 流程 → 规范 → 验证
 │                              │
 └──────────── 反哺目标 ─────────┘
```

五个分支构成一个**元循环**，与 OOC 哲学中的展开-沉淀循环同构：

| 分支 | 问题 | 对应展开循环 | 对应 OOC 哲学 |
|---|---|---|---|
| **目标** | 我们要做什么？ | 起点：身份 | 对象的 whoAmI + parentExpectation |
| **组织** | 谁来做？ | 能力 | 对象的 trait 分工 |
| **流程** | 怎么做？ | 计划 | 展开循环 |
| **规范** | 按什么标准？ | 约束 | 沉淀的"能力"层 |
| **验证** | 做对了吗？ | 反馈 | 沉淀循环的入口 |

## 与哲学层的对称

```
哲学/          ← 关于思想的思想
  ↓
对象/          ← 系统本身
  ↑
工程管理/      ← 关于设计的设计
```

哲学讨论"为什么"；工程管理讨论"怎么做"；对象是这两层的产物。

## 读者指引

**新加入的协作者**：先读 `组织/README.md` 了解分工，再读 `流程/工作节奏.md` 了解节奏。

**想做某项工作**：
- 涉及"为什么" → 先读哲学，做设计 → `流程/` + `规范/文档规范.md`
- 涉及后端代码 → `规范/代码规范.md` + `规范/测试规范.md`
- 涉及前端代码 → `规范/代码规范.md` + `对象/人机交互/`
- 涉及扩展能力 → `组织/nexus.md` 了解生态层
- 做体验验证 → `验证/用例/README.md` 了解用例结构

**做年度规划**：`目标/` 下三个文档一起读。

## 与基因的关联

工程管理层的设计本身遵循 OOC 哲学：

- **G1**（数据即对象）— "我们"作为一个元对象，有自己的身份和能力
- **G12**（经验沉淀）— 规范是沉淀的"能力"层，验证是沉淀循环的入口
- **G6**（关系即网络）— 组织（1+3）是关系网络的具象化
