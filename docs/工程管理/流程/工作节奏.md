# 9 步工作节奏

> 每个工作项按以下节奏推进。不是每一步都必须完整走——但漏掉哪一步，要意识到"为什么这里可以跳过"。

## 九步

```
Step 1: 思考     → 阅读哲学文档，明确"为什么做"和"做什么"
Step 2: 调研     → 阅读代码，理解现状和约束
Step 3: 设计     → 输出技术方案（可 spawn Sophia/Kernel 做第二视角）
Step 4: 实现     → 编码，TDD（红→绿→重构）
Step 5: 自测     → bun test，确认 0 fail
Step 6: 文档     → 检查文档一致性
Step 7: 提交     → git commit
Step 8: 体验验证  → spawn Bruce 以用户身份体验新功能
Step 9: 反思     → 更新 discussions.md / emergence.md
```

## 每步详解

### Step 1: 思考（哲学先行）

**目的**：明确"为什么做"和"做什么"。

**动作**：
- 阅读相关的 `哲学/genes/` 和 `哲学/emergences/`
- 检查当前需求是否与已有基因冲突
- 如果是新主张，在 `哲学/discussions/` 开一份草稿

**产物**：对"要做什么 + 为什么"的清晰陈述。

**不要跳过的场景**：
- 涉及新哲学主张时
- 涉及 G13 等重大基因时
- 拿不准"这是不是对的方向"时

### Step 2: 调研（工程现状）

**目的**：理解现状和约束。

**动作**：
- 阅读相关 `对象/` 下的架构文档
- grep 源码确认当前实现
- 找到与本次改动**可能冲突**的地方

**产物**：一份"现状图"——知道当前代码、文档是什么状态。

### Step 3: 设计

**目的**：输出技术方案。

**动作**：
- 写设计文档（可以是 scratch pad，不必 commit）
- spawn Sophia 或 Kernel agent 做第二视角（可选）
- 列出方案、tradeoff、决策理由

**产物**：设计方案。包含：
- 改哪些文件
- 加哪些字段
- 旧接口如何过渡（如有）

### Step 4: 实现（TDD）

**目的**：编码。遵循 TDD（测试驱动开发）。

**循环**：
```
RED   — 写一个测试，期望它失败
GREEN — 写最简实现让测试通过
IMPROVE — 重构，保持测试通过
```

**TDD 的价值**：
- 先写测试强制你把"要什么"想清楚
- 测试覆盖率自然高
- 重构时有安全网

### Step 5: 自测

**目的**：确认 0 fail。

**动作**：
```bash
cd kernel/
bun test
```

**通过标准**：所有已有测试 + 新加测试都通过。

**禁止**：跳过失败测试（`skip` / `.only`）。

### Step 6: 文档

**目的**：检查文档一致性。

**动作**：
- 改了 API → 更新相关 `对象/` 下文档
- 改了 trait → 更新 `对象/结构/trait/kernel-traits/` 或对应路径
- 改了流程 → 更新 `工程管理/流程/`
- 改了 meta.md 引用的概念 → 同步改 meta.md

**规则**：**改了代码但没改文档，这个 commit 不完整**。

### Step 7: 提交

**目的**：git commit。

**格式**：
```
<type>: <subject>

<body>
```

type 可以是：feat / fix / refactor / docs / test / chore

**一个 commit = 一个逻辑改动**。不要把多个不相关的改动塞进一个 commit。

### Step 8: 体验验证

**目的**：用户视角的验证。

**动作**：spawn Bruce agent：
```
请扮演新用户，不看代码只看 UI 和行为，体验新功能 X。
报告：
- 能用吗？
- 有没有意外行为？
- 界面是否清晰？
- 有什么需要说明但没说明的？
```

**为什么需要**：
- 单元测试只验证"代码对不对"
- 体验测试验证"用户感受对不对"
- LLM 自己做的功能，LLM 自己容易漏掉"新手视角"的问题

### Step 9: 反思

**目的**：沉淀经验。

**动作**：
- 本次任务有新学到的？→ 写 `哲学/discussions/`
- 本次验证了某个涌现？→ 更新 `哲学/emergences/xxx.md` 的验证状态
- 本次有反模式被识别？→ 更新 `规范/代码规范.md`

**时机**：
- Commit 后、下一个任务前
- 不需要每次都写——只在有值得沉淀时

## 可跳过的场景

| 步骤 | 什么时候可以跳过 |
|---|---|
| Step 1 | 改 typo / 文档错别字 |
| Step 2 | 改单一已知文件的局部逻辑 |
| Step 3 | 改动极小（< 10 行）且方向明确 |
| Step 4 | 纯文档改动 |
| Step 5 | 无代码改动 |
| Step 6 | 纯内部重构无文档变更 |
| Step 7 | — 永远不跳过 |
| Step 8 | 纯后端改动无 UI 影响 |
| Step 9 | 无新经验可沉淀 |

## 节奏的快与慢

**大型任务**：9 步全走，可能 1-2 周。
**中型任务**：7-8 步，几天。
**微型任务**：3-5 步，半小时到几小时。

不要为小改动强行走完 9 步——过度流程会消耗工程节奏。

## 与基因的关联

- **G12**（经验沉淀）— Step 9 是沉淀循环在工程层的体现
- **G4**（对象通过输出程序来行动）— Step 4 是 "output program" 的工程翻译
