# 任务分发 — 按部门职责分发

> Supervisor 收到任务后，按部门职责拆分。本文档规定分发规则。

## 分发规则

### 涉及"为什么" → Sophia

典型特征：
- 讨论"XX 设计应该是什么"（哲学层面）
- 涉及基因（G1-G13）的修改或解读
- 未决的根本性问题

例：
- "Effect 是否应该支持反向流？"
- "G13 的'帧 0'是否该扩展为'帧 -1'（模拟 pre-thought）？"

### 涉及 `kernel/src/` 后端代码 → Kernel

典型特征：
- 实现某个哲学主张
- 修复后端 bug
- 性能优化
- 测试

例：
- "线程树的 scheduler 死锁检测优化"
- "Context 构建器支持新的字段"
- "修复 Inbox 并发写入的竞态"

### 涉及 `kernel/web/` 前端代码 → Iris

典型特征：
- UI / UX 设计与实现
- 前端状态管理
- 体验优化
- SSE / 动态加载等前端机制

例：
- "ChatPage 的输入框支持 @ mention"
- "SessionKanban 抽屉改成可记忆高度"
- "StoneView 在自渲染失败时有更友好的 fallback"

### 涉及扩展能力、新对象、外部集成 → Nexus

典型特征：
- 创建新的 library trait
- 集成外部系统（飞书、Slack 等）
- 新对象设计（如某个工作流对象）
- 扩展生态层

例：
- "添加 lark-doc trait 支持飞书文档读写"
- "Library 对象的 skill 市场机制"
- "创建 'code-reviewer' 对象"

### 涉及多个部门 → Supervisor 自己协调

典型特征：
- 单一任务横跨 2+ 个部门
- 需要特定顺序（如先哲学后实现）
- 需要协同步调（如后端和前端并行推进）

例：
- **线程树架构重构** → Sophia（设计）+ Kernel（后端）+ Iris（前端）
- **新用例实现** → Kernel + Iris + Nexus

## 分发流程

```
用户请求 → supervisor.root 线程
    ↓
Supervisor 分析：涉及哪些部门？
    ↓
创建 Issue（ISSUE-001）— 记录讨论与推进
    ↓
创建若干 Task（每部门一个）
    ↓
talk(alan-sophia, "请做 X 的哲学设计")  ← 并行/串行
talk(alan-kernel, "等 sophia 出方案后开始实现")
...
```

Supervisor 使用 [../../对象/合作/结构化/](../../对象/合作/结构化/) 的机制：

- `createIssue` — 记录总体需求
- `createTask` — 拆分给各部门
- `assignee` — 分配到对应部门的"代理对象"

## 并行 vs 串行

**并行**：任务间**没有依赖**。各部门独立推进。

```
Sophia 做设计 A
Kernel 做底层 B（不依赖 A）
Iris 做界面 C（不依赖 A 或 B）
  ↓
全部完成后 supervisor 集成
```

**串行**：后续依赖前面的产物。

```
Sophia 先做哲学设计
  ↓ (Sophia 的方案作为 Kernel 的输入)
Kernel 基于设计做实现
  ↓ (后端的 API 作为 Iris 的输入)
Iris 基于后端做前端
```

## 分发的注意事项

### 不要过早分发

如果需求**本身还不清楚**，先自己（或 spawn Sophia）做一轮需求澄清，再分发。

否则部门收到的是模糊任务，产出不对齐。

### 分发时带足够上下文

每个部门看不到全局（它们只看自己领域）。Supervisor 要把：
- 总体目标
- 本部门的责任
- 与其他部门的接口

都写清楚在 Task description 里。

### 用 Issue 保持多方可见的讨论

如果任务需要跨部门讨论（"这个设计是否合理？"），用 Issue，不用私下 talk。这样**所有部门都能看到完整讨论**。

## 与基因的关联

- **G6**（关系即网络）— 部门就是关系网络中的节点
- **G8**（Effect 与 Space）— Session + Issue 是分发的 Space
