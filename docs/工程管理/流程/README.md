## 核心主张

### 流程不是模板，是节奏

OOC 的工作流程不要求严格按步骤。它是**节奏参考**——让你知道一个完整 cycle 包含哪些环节，避免漏掉关键步骤。

### 实现前必须有思考和设计

9 步里前 3 步是**思考、调研、设计**——这让"着急写代码"的冲动被节制。

错误示范：
```
用户提需求 → 立即写代码 → 发现设计没想清楚 → 返工
```

正确流程：
```
用户提需求 → 思考"为什么" → 调研现状 → 输出设计 → 实现 → 测试 → 文档
```

### 完成后必须有验证和反思

9 步的最后 2 步是**体验验证**和**反思**。这让"做完就下个任务"的惯性被制约，让经验有机会沉淀。

## 与"哲学/两个循环"的对应

```
工作节奏 = 展开循环在工程层的具象化：
  思考（身份）→ 设计（计划）→ 实现（思考+行动）→ 反思（沉淀起点）
```

## 九步速览

```
Step 1: 思考     → 阅读哲学文档，明确"为什么做"
Step 2: 调研     → 阅读代码，理解现状和约束
Step 3: 设计     → 输出技术方案
Step 4: 实现     → 编码，TDD（红→绿→重构）
Step 5: 自测     → bun test，确认 0 fail
Step 6: 文档     → 检查文档一致性
Step 7: 提交     → git commit
Step 8: 体验验证  → spawn Bruce 以用户身份体验
Step 9: 反思     → 更新 discussions.md / emergence.md
```

详见 [工作节奏.md](工作节奏.md)。
