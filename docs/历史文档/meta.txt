# OOC (Object-Oriented Context) Agent 系统思维导图

OOC 旨在将 Agent 系统的 Context 从静态文本升级为动态的、可演化的“对象生态”。
通过引入面向对象的封装、元编程、并发思考（Bias）及活性依赖（Relation），构建一个具备自适应能力的智能体协作网络。

object
    具有
        角色定义 role
            对自身的说明，对内，只有自己可见
        名片 intro
            对外可见
        字段 field
            持有的数据、对象
        方法 method
            可以执行的程序
        思维方式 bias
            陈述如何思考某一方面问题的 prompt
        关系 relation
            object 与其他对象的关系
        影响/事实 thing
            object 做的事情

    行为方式
        通过“输出Python程序"来进行行动
        执行程序时可以访问
            self
                可以修改 self
                    包括字段、方法，可以为自己编写自己的方法, 具有“元编程能力”
                可以调用 self 的方法、读写 self 的字段

            其他对象的 public method
                world 是特殊的对象，具有和底层系统交互的 api
                    可以创建对象
                    可以注册回调、事件监听
                    可以调用 LLM 接口
                    可以提交要执行的代码

    以上组合可以表达出
        思考能力
        对话能力
        记忆能力
        组织能力
        计算能力、编程能力
        管理能力
        自我学习能力
        自我优化能力 等

    生命周期
        创建
            object 被加载或派生时进入创建阶段
        活跃
            可对话、可执行 method、可参与协作
        存档/销毁
            object 在结束或退出系统时进入收尾阶段
            此时需要触发清理与通知类 bias
                - 通知关联对象（relation 维护者、协作者）
                - 归档过程记忆与关键状态
                - 解除或更新 relation registry
                - 输出审计记录与最终摘要

    思考能力
        object 的 bias 负责一种视角的思考

        bias
            是
                特殊的 field
                表示 object 的思考角度

            按照定位分为两类
                main bias
                    核心 bias, 对应于 object 的素质、重要事项或者核心任务
                    每一次思考，核心 bias 都会参与
                extend bias
                    扩展 bias, 对应于 object 的其他方面、其他视角
                    按照激活条件、按需参与思考

            按照思考环节分为三类
                - pre_thinking
                    负责进行联想、信息查找
                - thinking
                    负责输出计划、具体行动
                - post_thinking
                    负责修饰、检查、约束

            的上下文
                会作为 LLM 接口的输入
                包含了
                    - bias 的 prompt
                    - 和所关注的事情相关的 self 的 field、method, bias 的 prompt 会说明如何使用这些数据、函数 （就像 agent skill 一样）
                    - 通过程序计算看到的信息
                        bias 可以为自己编写程序来获取需要的信息

            的触发与检索
                extand bias 具有 when 条件，当上下文满足 when 条件时，会触发 bias 进行思考 （会有一个 bias 负责判断 biases 的 when 条件、会有一个 bias 负责对大量 biases 进行分组）

                - 分组管理
                    如果 extend bias 数量特别多、或者定位相似，那么可以进行分组管理，每个分组都有一个 index bias 负责检查该组成员是否需要触发
                    例如：都在任务结束时触发的一组 review 工作的 bias 可以分到一组、都做编程工作的 go/java/py 等编程 bias 可以分到一组。同一个 bias 允许分到多组。
                    并且初始时总是有一个 root index bias 负责最初始的一轮 bias 检索工作
                    并且初始时总是有一个 manager bias 负责意识到“bias 数量太多，应当分组” 并进行分组操作

            的思考过程
                使用一个 flow 对象进行表示

                class BiasFlow 是特殊的 Flow 对象，它的 context 会包含 parent Flow 的 context
                并且具有 refresh 方法，可以让它基于最新的 context 重新运行 Bias 的思考过程

        完整的思考模式说明
            thinkloop {
                1. 基于 context 构建 LLM 输入
                2. 基于 context 进行思考（向 LLM 输入）
                3. 根据 LLM 的输出结果进行执行
                    3.1 执行结果会更新到 context 用于下一次思考
                    3.2 如果执行产生了新的 flow 对象，那么会自动添加到当前 context
            }

            context:
                object 的上下文分为 public 和 private 两部分:
                public 即所有 bias 都可以看见的公共上下文
                private 即只有这一个 bias 可以看见的上下文
                    例如，有一个 bias 负责查看通讯录并从中选择，那么这个通讯录，就是这个 bias 的私有上下文，其他 bias 不需要看见（因为会干扰它们的判断）

                public 和 private 的搭配呈现出“一主多辅”的思考结构。

                当 flow 对象实例化时，才具有 上下文；stone 对象没有上下文这个概念。

                context 包含了这个 flow 的请求、执行过程、子 flow 的状态、结果 以及 当前对象的 核心 bias，以及 plan 信息
                plan 字段可以被编辑

            context 的基本结构如下(伪代码表示，仅供参考):
                <context>
                    <role> <!-- 这里列出核心 bias -->
                        <bias name="self">
                            <!-- 注意，对于核心 bias，在public上下文中的表现为思考的 prompt 而不是思考的结果 -->
                            i'm ...
                            i could ...
                        </bias>
                        <bias name="xxx">
                            ...
                        </bias>
                    </role>
                    <request> <!-- 这里是 flow 的任务目标 -->
                        ...
                    </request>
                    <plan>
                        ...
                    </plan>
                    <process> <!-- 这里列出 flow 的过程 -->
                        <event type="execute">
                            <code>
                                self.methodExample("a")
                                browser := get_object("browser")
                                webpage := browser.open("https://www.example.com")
                                webpage.talk("intro your page content")
                            </code>
                            <output>
                                sub flow ooc://webpage_talk_bhwu21/ created
                            </output>
                        </event>
                        <event type="callback" from="webpage_talk_bhwu21">
                            page content is ...
                        </event>
                    </process>
                    <flows>
                        <!-- 这里会列出当前 flow 派生的 子 flow 的状态、结果-->

                        <flow id="webpage_talk_bhwu21" status="running" description="Talk about the webpage content"/>
                    </flows>
                    <biases>
                        <!-- 这里和其他 flow 做区分，单独列出当前 flow 派生的 子 bias flow 的状态、结果-->
                        <bias name="root_index_bias" phase="before_thinking" description="search for relevant">
                            ...
                        </bias>
                        <flow id="bias_search_docs" phase="before_thinking" status="finished" description="Search for relevant documents">
                            i searched ...
                        </flow>
                    </biases>
                </context>



    relation
        是
            特殊的 field
            和特定对象、或特定的一群对象 相关的 bias
        管理与登记
            关系登记表（relation registry）
                每个 object 维护一份 relation 列表（可持久化）
                记录字段包含
                    - target_id
                    - relation_type
                    - strength / trust
                    - evidence (触发原因与片段)
                    - created_at / updated_at
                    - status (active / archived)
            去重与合并
                以 (target_id + relation_type + evidence) 作为唯一键
                若已存在则更新 strength 与 evidence
            失效与解除
                关联对象被销毁或不可达时，解除 relation 并归档
                关系可被主动解除并归档
            可见性
                public relation 仅暴露必要摘要
                private relation 仅本 object 可见
        的检索
            当当前上下文提起这些对象时，会触发 relation 的 bias 进行思考 （会有一个 bias 负责检索相关的 relation）
        的创建
            机械方式：object 的 method 的程序中可以配置：当 object 的 method 被执行时，与调用方绑定一个固定的 relation
            人工方式：当 object 在思考时，可以手段与 其他 object 创建 relation （这也会有一个 bias 负责“意识到”应该创建 relation ）
            主动登记触发条件
                - talk 交互超过阈值（例如连续 N 次协作）
                - 任务协作产生明确产出（如共同完成报告/文件）
                - 出现高价值信息交换（被引用或被复用）
                - caller/receiver 被频繁提及（context 显著关联）
                - 发生显式引用 ooc://{id}/
            触发策略
                由 Relationship-Awareness Bias 扫描上述条件
                达到阈值后，向 relation registry 写入或更新
        作用
            当一方发生改变时，通知 relation 的另一方
        对象引用
            所有 object 都具有 id，并分配 URL = ooc://{id}/ 来引用这个 Object
            在对话过程中、创建 flow、创建 bias 等地方时，都可以通过这个 URL 来表示对某个 object 的引用
            需要注意，当 object 在进行引用时，需要主动创建和其他 object 的 relation

    thing
        是
            一段过程(process)
            object 做的一件事情
                object 的一切行动都通过 thing 表达
            object 的“偏特化”
            object 的派生
                thing 继承自某一个 object, thing 本身也是一种特殊的 object
                    thing 可以继续派生子 thing
        具有
            过程记忆
                thing 的执行过程
                object 的思考结果
                object 的行动结果
                object 的上下文
            状态标志
                初始
                进行中
                完结
                    成功
                    失败
                暂停

    method
        是
            object 的一段可复用、含义明确的静态程序
        区分
            readonly // 只否只读（不修改 object 的数据）
            mutable // 是否可能会修改数据
            with_llm // 是否涉及 llm 的调用
            public // 是否对其他 object 可见
        统计
            调用RT // 这一信息也会用于判断这个 method 是否是一个长耗时行为

    field
        区分
            public // 是否对其他 object 可见
        数据类型
            使用与 JSON 基本一致的数据类型，但支持 bigint(int64)

    可以分为两类
        stone 型
            stone 型的 object 是静态的，是数据、是状态、是定义
        flow 型
            flow 型的 object 是动态的，是过程、是行动，比如上述的 thing 就是 flow 型的 object
            flow 初始从 stone 的派生，从 stone 继承
            flow 可以派生更多 flow 对象，具有 stone 为根节点的 flow 派生树
            flow 是 stone 的私有对象，对外不可见

    支持持久化
        object 可以持久化到本地文件系统
        如果一个 object 的字段里持有对其他 object 的引用，持久化时只记录其他 object 的 id；从文件还原时，再根据 id 取回对象实例
        Ref[T] 泛型类型
            OOC 提供 Ref[T] 类型来显式管理对象引用
            序列化时：Ref[T] 会被序列化为目标对象的 ID 字符串
            反序列化时：ID 字符串会被还原为 Ref[T] 对象，此时内部尚未加载真实对象实例
            运行时：调用 Ref[T].get() 方法时，会自动通过 Kernel 获取或加载目标对象实例，并返回 T 类型的对象
            作用：
                1. 解决循环引用导致的序列化死循环问题
                2. 实现对象的延迟加载，提高系统启动速度
                3. 配合 IDE 提供完整的类型提示支持

        持久化时
            字段定义和方法定义以文件 code.py 的形式存储
            数据内容以 data.json 的形式存储


    支持分布式
        ownership lock
            object 具有 名为 ownership 的 lock
            当需要修改 object 的 source code、field value 时，需要对 object 的 ownership 加锁

    支持并发
        每个 flow 都是独立的进程/线程执行，flow 之间有可能会相互冲突
        因此
            需要考虑加锁
                object 在为自己编写方法时，需要考虑并发安全(由一个 bias 负责注意这个)
            需要考虑如何避免做冲突的事情
                具有一个 bias，能够查找、检索和当前 flow 所做的事情相关的其他 flow 并输出是否存在冲突、是否重复或者多余
                检索范围是 同一个 stone 下的 flow；不同 stone 之间的 flow 相互隔离，无权访问

    对话能力
        所有 object 都会从 base object 继承 talk 方法用于实现对话能力
        初始时，talk 会创建一个 flow 对象，来进行思考、行动和回复
        对同一个 flow 对象的 talk 不会重复创建 flow 对象，而是补充信息到这个 flow 对象

        难题
            1
                假设 stoneA 的 flowA 对象，向 stoneB 进行提问，stoneB 派生了 flowB 对象来思考并回答
                flowB 又向 stoneC 提出更具体的问题，于是又有 flowC 进行思考和行动
                但是这时，flowC 想反问 stoneA 获取更多信息，但是 stoneA 派生了 flowA ～ flowA100 等多个对象
                flowC 如何能够找到对应的 flowA 对象?
                    目前考虑应当这样
                        从 flow 的派生链路上向上查找，因为 flowC 在 flowA 的派生树上，因此可以向上查到对应的 flowA

            2
                假设 A 和 B 对话，A 和 C 对话，然后 B 和 C 对话
                这时 C 思考时的上下文，是否只包含 B~C 而不包含 A~C ?
                    这一点和 关于并发 一节中 “需要考虑如何避免做冲突的事情” 提到的办法一样，
                    通过一个 bias，查找、检索和当前 flow 所做的事情相关的其他 flow 并输出是否存在冲突、是否重复或者多余

            3
                stone 是否可以看见其他 stone 的 flow?
                    flow 应当是只有 stone 内部可见
                    也就是说， stoneA 的 flowA 向 stoneB 进行提问，stoneB 只知道是 stoneA 的问题，但不知道来源于 flowA
            4
                假设 A 和 B 对话，A 和 C 对话，然后 B 和 C 对话，然后 B 和 A 提起了 C, 然后 A 再和 C 对话
                这时 stoneC 先后共有三个 flow对象，flowC_A1, flowC_B, flowC_A2
                这时 flowC_A2 如何进行思考，上下文是怎么样的?
                    思考时，会由上述提到的 bias 检索出相关的 flowC_A1, flowC_B ，
                    flowC_A2 可以通过 talk 的方式向 flowC_A1 flowC_B 进行询问
            5
                如何进行回复?
                    回复通过 调用 caller 的 talk 方法的形式回复
                    对同一个 flow 对象的 talk 不会重复创建 flow 对象，而是补充信息到这个 flow 对象


    继承关系
        Stone 元类提供基础的 Object 定义
        Flow 元类提供与思考、执行过程相关的方法、Bias

        从 Stone 元类派生出 OOC 世界的 对象成员，例如 class Supervisor(Stone) 就表示是一个 stone 类型的 ooc 对象
        从 Stone 具体实例可以派生出 Flow 对象，这个对象会多继承自所属的 Stone 对象以及 Flow 元类，例如 class TaskA(Supervisor, Flow)


    Flow Tree 
        是
            从一个 Flow 派生出的 树形 Flows 组织（而 Flow 又是从 Stone 派生的）
            每个 Flow 都可以继续派生出子 Flow, 这很容易：和其他对象进行 talk 或者对自己进行 focus 都会创建子 Flow
            Flow 对象也可以进行元编程（因为它继承了Stone的能力），Flow 在元编程时，默认新增的字段、方法，只会对当前 Flow 生效，而不会对 Stone 以及 parent Flow 生效
            子 Flow 可以把自己认为 parent Flow 需要的新字段、新方法提供给 parent Flow（仅限于 父子 Flow 都从属于同一个 Stone） 或者 stone 
                注意，子 Flow 只能提供建议，具体是否接受由 parent Flow 决定
                当 子 Flow 完结时，进行一次这样的思考（这个思考点由 bias 实现）
                如果 Flow 没有 parent Flow, 它可以自己决定是否将新字段、新方法应用到所属的 stone 对象上

world
    是
        OOC 系统的容器系统，向 object 提供了一组底层 API
            create_object // 创建对象
            edit_object // 编辑对象（新增字段、方法； 废弃字段、方法）
            llm_query // 请求 LLM 接口
            exec_program // 执行一段程序

        world 也是一个 object


extension
    是
        OOC 系统的扩展
    作用
        在 OOC 系统初始化时，extension 可以向系统提供初始的 object
        例如
            base extension 提供 world、supervisor 对象
            filesystem extension 提供 filesystem 对象
            browser extension 提供 browser 对象
            shell extension 提供 shell 对象


项目实现
    编程语言
        Python3.13
    使用模式
        - kernel
            内核层
                包含了 OOC 系统的核心机制的实现
                包含了 OOC 系统的元类(Stone\Flow\Chat)的定义
                包含了 OOC 系统的内置 Object 以及基础 Bias 的定义
            以 Python 本地包的形式提供，通过 setup 工具打包，提供给用户层系统
            同时也会提供一个 CLI 工具来启动 OOC 系统
        - user
            用户层
                包含了 OOC 系统创建的对象实例，每个对象实例在文件系统中的表现都是一个 code.py 文件以及一个 data.json 文件
                其中 code.py 中的文件可以引用 kernel 中的类型定义、函数
    文件目录结构
        - kernel
            - meta
                包含了 OOC 系统的元类(Stone\Flow\Chat)的定义
                - stone.py
                    定义 Stone 类
                - flow.py
                    定义 Flow 类
                - chat.py
                    定义 Chat 类
                - bias.py
                    定义 Bias 类
                - biases 目录
                    目录下每个 xxx.py 对应一个内置的基础 bias
            - thinkable
                实现和 LLM API 相关的代码
            - storable
                实现和 持久化 相关的代码
            - executable
                实现和 执行程序 相关的代码（对应上述的 通过“输出Python程序"来进行行动）
            - world
                实现 OOC 系统的核心 API
    OOC 系统扩展
        - filesystem
        - browser
        - terminal

    OOC CLI 工具
        参数
            -world_dir=<path>
                指定 OOC 持久化对象文件的位置
            -work_dir=<path>
                指定 OOC 系统的工作目录，默认为当前目录
