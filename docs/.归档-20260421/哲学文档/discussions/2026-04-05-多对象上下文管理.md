# 多对象上下文管理设计讨论

> 讨论时间：2026-04-05
> 参与者：Alan Kay (Supervisor), User
> 状态：进行中

---

## 背景

OOC 系统支持多个 Object 之间相互通信，但当前设计在多对象、多轮次、多事情的上下文管理方面存在不足。

### 当前机制回顾

**消息投递**：
- `talk(target, message)` — 同步投递到目标的 `pendingMessages`
- `reply(message)` — 回复消息
- `talkToSelf(message)` — 与 SelfMeta 对话

**Session 管理**：
- 同一 Stone 在同一 Session 中只有一个 Flow（sub-flow 复用）
- Session 跟踪所有参与对象的 Flow
- `captureReply()` 机制捕获回复

**消息存储**：
- 每个 Flow 的 `messages[]` 记录收发消息
- `replyTo` 字段关联请求-响应

---

## 问题识别

### 六种典型场景

| 场景 | 当前支持 | 缺失能力 |
|------|---------|---------|
| 1. 简单请求-响应 | ✅ 完整支持 | - |
| 2. 多轮对话 | ⚠️ 部分支持 | **对话线程** — 将相关消息分组为一个对话 |
| 3. 协作链 | ⚠️ 部分支持 | **协作上下文共享** — 下游对象能看到上游的原始意图 |
| 4. 多方会议 | ❌ 不支持 | **共享上下文空间** — 多个对象共同可见的信息区域 |
| 5. 并行任务 | ⚠️ 部分支持 | **任务分组** — 将相关的并行任务标记为一组 |
| 6. 长期协作 | ⚠️ 部分支持 | **协作模式沉淀** — 跨 Session 的协作经验积累 |

**用户优先级**：多轮对话、多方会议、并行任务

---

## 设计原则确认

### Context 粒度选择

**问题**：对象在思考时需要"看到"多少上下文？

**用户决策**：**倾向于完整信息**

这意味着：
- **多轮对话**：对象能看到整个对话历史
- **多方会议**：所有参与者能看到会议中的所有发言
- **并行任务**：发起者能看到所有并行任务的进展

**哲学依据**：符合 OOC 的 G5 — **Context 是对象的全部世界**，完整信息让对象做出更好的决策。

## 方案探索

### 方案 1：对话线程（Conversation Thread）

**核心思想**：引入 `ConversationThread` 作为一等公民。

```typescript
interface ConversationThread {
  id: string;
  sessionId: string;
  participants: string[];
  messages: ThreadMessage[];
  type: "dialogue" | "meeting" | "parallel";
  parentThreadId?: string;
  createdAt: number;
  updatedAt: number;
}
```

**优点**：
- ✅ 清晰的对话边界
- ✅ 支持嵌套对话
- ✅ 完整历史追溯

**缺点**：
- ❌ 需要新增数据结构和持久化
- ❌ Context 可能膨胀
- ❌ 需要显式管理线程生命周期

---

### 方案 2：共享上下文空间（Shared Context Space）

**核心思想**：在 Session 级别创建共享的 `ContextSpace`。

```typescript
interface ContextSpace {
  sessionId: string;
  sharedMessages: Message[];
  sharedData: Record<string, unknown>;
  taskGroups: Map<string, TaskGroup>;
}
```

**优点**：
- ✅ 简单直观（Session = 共享空间）
- ✅ 自动同步
- ✅ 支持结构化数据共享

**缺点**：
- ❌ 缺少对话边界
- ❌ Context 膨胀更严重
- ❌ 难以区分消息的目标对象

---

### 方案 3：元数据标记（Metadata Tagging）

**核心思想**：在现有 `FlowMessage` 上增强元数据。

```typescript
interface FlowMessage {
  // ... 现有字段
  metadata: {
    conversationId?: string;
    meetingId?: string;
    taskGroupId?: string;
    replyTo?: string;
    participants?: string[];
  };
}
```

**优点**：
- ✅ 最小改动
- ✅ 灵活（可以同时标记多个维度）
- ✅ 向后兼容

**缺点**：
- ❌ 需要手动管理 ID
- ❌ Context 构建逻辑变复杂
- ❌ 缺少显式的"对话对象"

## Claude Code Team 机制启发

### 关键设计

1. **Team 配置文件**：`~/.claude/teams/{team-name}/config.json`
   - 定义 team 成员、角色、prompt

2. **Inbox 消息机制**：`~/.claude/teams/{team-name}/inboxes/{agent-name}.json`
   - 每个 agent 有独立的 inbox
   - 消息格式：`{from, text, timestamp, color, read}`
   - 支持结构化消息（`permission_request`, `idle_notification`）

3. **Task List 共享**：`~/.claude/tasks/{team-name}/`
   - 所有 agent 共享同一个 task list

4. **关键特性**：
   - ✅ 异步消息（通过 inbox 投递）
   - ✅ 角色分工
   - ✅ 状态通知
   - ✅ 权限请求

### 对 OOC 的启发

| 维度 | Claude Code Team | OOC 当前设计 | 可借鉴之处 |
|------|------------------|--------------|-----------|
| 消息投递 | Inbox（文件系统） | pendingMessages（内存） | ✅ 持久化 inbox |
| 对话历史 | 每个 agent 的 inbox 独立 | 每个 Flow 的 messages 独立 | ⚠️ 需要跨 agent 的对话视图 |
| 共享上下文 | Task List（共享文件） | Session（共享目录） | ✅ 结构化的共享数据 |
| 状态同步 | idle_notification | Flow.status | ✅ 可以增强状态通知 |

---

### 方案 4：Session Inbox + Conversation View（混合方案）

**核心思想**：借鉴 Claude Code 的 Inbox 机制 + 共享 Task List，但保持 OOC 的哲学。

```typescript
interface SessionContext {
  sessionId: string;

  // 1. Inbox 机制（借鉴 Claude Code）
  inboxes: Map<string, Message[]>;

  // 2. Conversation View（新增）
  conversations: Map<string, Conversation>;

  // 3. 共享数据区
  sharedData: Record<string, unknown>;

  // 4. 任务分组
  taskGroups: Map<string, TaskGroup>;
}

interface Conversation {
  id: string;
  participants: string[];
  messages: ConversationMessage[];
  type: "dialogue" | "meeting";
  createdAt: number;
  updatedAt: number;
}
```

**工作方式**：

**多轮对话**：
1. A `talk(B, msg)` → 自动创建 Conversation，写入 B 的 inbox
2. B 回复 → 消息写入 A 的 inbox，同时追加到 Conversation
3. Context 构建时，注入完整的 Conversation.messages

**多方会议**：
1. A 调用 `startMeeting([B, C])` → 创建 meeting 类型的 Conversation
2. A/B/C 的消息都写入同一个 Conversation
3. 所有参与者的 Context 都能看到完整对话

**并行任务**：
1. A 调用 `createTaskGroup("搜索任务", [B, C, D])`
2. 后续 talk 时传入 `taskGroupId`
3. Context 中按 taskGroup 分组展示消息

**优点**：
- ✅ 借鉴成熟的 Inbox 机制
- ✅ Conversation 提供完整对话视图
- ✅ 最小改动（复用现有 messages 结构）
- ✅ 支持持久化

**与 OOC 哲学的契合**：
- **G8（Effect）**：Inbox 是"它→我"的影响通道
- **G5（Context）**：Conversation 注入 Context，对象看到完整对话
- **G7（持久化）**：Inbox 和 Conversation 都可以持久化到 Session 目录

## 深入分析：A 与 B 通信时 B 的响应模式

### 用户提出的四种基础场景

当 A 向 B 发送消息时，B 可能的响应方式：

1. **中断当前任务**：B 停下当前的事情，暂存上下文，然后处理 A 的信息
2. **并行处理**：B 判断和当前的事情无关，新建线程，并行处理当前任务和 A 的事情
3. **延迟处理**：B 将 A 的事情暂存到 TODO List 并继续处理当前事情
4. **重新规划**：B 当前事情和 A 说的信息有关，B 停止当前任务，或重新设计方案再继续任务

### 扩展场景分析

#### 5. **拒绝处理**
- B 判断 A 的请求不合理、超出能力范围、或违反约束
- B 回复拒绝消息，说明原因
- B 继续当前任务
- **例子**：A 请 B 删除重要文件，B 拒绝并说明安全策略

#### 6. **转发/委托**
- B 判断 A 的请求应该由 C 处理（B 不是最佳人选）
- B 将消息转发给 C，或建议 A 直接联系 C
- B 可能继续当前任务，或等待 C 的结果
- **例子**：A 请研究员搜索，研究员转发给浏览器

#### 7. **请求澄清**
- B 收到 A 的消息，但信息不足以执行
- B 暂停当前任务（或并行），向 A 提问澄清
- B 等待 A 的回复后再决定如何处理
- **例子**：A 说"搜索论文"，B 问"哪个领域？哪个时间范围？"

#### 8. **合并到当前任务**
- B 发现 A 的请求与当前任务高度相关，可以合并
- B 调整当前任务的 plan，将 A 的需求整合进来
- B 继续执行（不中断，但任务范围扩大）
- **例子**：B 正在写报告，A 说"加上性能分析"，B 将其加入报告大纲

#### 9. **部分响应 + 继续**
- B 可以立即给出部分答案（如缓存数据、已知信息）
- B 先快速回复 A，然后继续当前任务
- 如果需要深入处理，稍后再完整响应
- **例子**：A 问"上次实验结果"，B 立即回复已有数据，无需中断

#### 10. **依赖等待**
- B 发现 A 的请求依赖于 B 当前任务的结果
- B 告诉 A "等我完成 X 后再处理你的请求"
- B 继续当前任务，完成后再处理 A 的消息
- **例子**：A 请 B 分析数据，但 B 正在收集数据，需要先完成收集

#### 11. **优先级抢占**
- B 判断 A 的请求优先级高于当前任务（紧急/重要）
- B 立即中断当前任务，保存状态
- B 优先处理 A 的请求，完成后再恢复原任务
- **例子**：A 发现严重 bug，B 立即停止新功能开发去修 bug

#### 12. **批量处理**
- B 收到多个类似的请求（来自 A 或其他对象）
- B 将这些请求归类，批量处理以提高效率
- B 可能等待一段时间收集更多请求，然后统一处理
- **例子**：B 收到多个"搜索"请求，批量发起搜索以节省 API 调用

#### 13. **协商调度**
- B 当前任务很重要，但 A 的请求也重要
- B 与 A 协商：能否延迟？能否降低要求？能否分阶段交付？
- 双方达成共识后，B 按协商结果处理
- **例子**：A 要完整报告，B 说"我先给你初稿，明天给完整版"

#### 14. **触发反思**
- B 收到 A 的消息，发现与自己的认知/记忆冲突
- B 暂停当前任务，进入反思模式（talkToSelf）
- B 审视自己的假设、更新认知，然后再响应 A
- **例子**：A 说"那个 API 已经废弃了"，B 反思并更新自己的知识

#### 15. **静默忽略**（边界情况）
- B 判断 A 的消息是噪音、垃圾信息、或重复消息
- B 不做任何响应，继续当前任务
- **注意**：这可能违反协作礼仪，需要明确的规则支持

---

### 响应模式的分类维度

可以从多个维度对这些场景进行分类：

#### 维度 1：当前任务的处理方式
- **中断**：停止当前任务（场景 1, 4, 11, 14）
- **并行**：新建线程并行处理（场景 2）
- **延迟**：稍后处理（场景 3, 10, 12）
- **合并**：整合到当前任务（场景 8）
- **继续**：不影响当前任务（场景 5, 9, 15）

#### 维度 2：对 A 的响应方式
- **立即响应**：马上回复（场景 5, 7, 9, 11, 13）
- **延迟响应**：稍后回复（场景 3, 10, 12）
- **转发**：让其他对象处理（场景 6）
- **拒绝**：不处理（场景 5, 15）
- **协商**：讨论如何处理（场景 7, 13）

#### 维度 3：决策依据
- **优先级**：紧急程度（场景 11）
- **相关性**：与当前任务的关系（场景 2, 4, 8, 10）
- **能力**：是否有能力处理（场景 5, 6）
- **资源**：是否有足够资源（场景 12, 13）
- **认知冲突**：是否触发反思（场景 14）

---

### 对设计的启示

这些场景揭示了几个关键需求：

1. **B 需要判断能力**：
   - 判断消息的优先级、相关性、可行性
   - 判断是否需要中断、并行、延迟、合并

2. **B 需要状态管理能力**：
   - 保存/恢复当前任务状态（中断场景）
   - 管理多个并行线程（并行场景）
   - 管理 TODO 队列（延迟场景）

3. **B 需要协商能力**：
   - 向 A 提问澄清（场景 7）
   - 与 A 协商调度（场景 13）
   - 拒绝并说明原因（场景 5）

4. **系统需要支持的机制**：
   - **消息优先级**：标记紧急/重要程度
   - **任务暂停/恢复**：支持中断场景
   - **多线程调度**：支持并行场景
   - **TODO 队列**：支持延迟场景
   - **反思触发**：支持认知冲突场景

## 哲学突破：内外双线程模型

### 用户的核心洞察

> "与其他对象交互，这是'外'；思考和执行任务，这是'内'。一个智能体的结构可能是具有内外双线程的：
> - **外线程**：负责和其他对象交流、理解需求、为自己分配任务
> - **内线程**：负责专注任务执行"

这是一个深刻的哲学洞察。让我们从多个维度展开思考。

---

### 一、哲学维度：内外之分的本质

#### 1.1 现象学视角：意向性的双重结构

**胡塞尔（Husserl）的意向性理论**：
- 意识总是"关于某物的意识"（intentionality）
- 意识有两个方向：
  - **向外**：指向世界中的对象（noema，意向对象）
  - **向内**：反思自身的意识活动（noesis，意向行为）

**映射到 OOC**：
- **外线程** = 向外的意向性 — 感知他者、理解需求、建立关系
- **内线程** = 向内的意向性 — 专注任务、深度思考、自我反思

**关键洞察**：人类意识不是单线程的。我们可以一边听别人说话（外），一边思考自己的问题（内）。这不是"分心"，而是意识的**双重结构**。

---

#### 1.2 存在主义视角：自我与他者

**萨特（Sartre）的"自为存在"与"为他存在"**：
- **自为存在（Being-for-itself）**：自我意识、自由选择、内在计划
- **为他存在（Being-for-others）**：被他者凝视、社会角色、外在关系

**海德格尔（Heidegger）的"操心"（Sorge）**：
- **操劳（Besorgen）**：处理事务、完成任务（内）
- **操持（Fürsorge）**：关心他人、协作互助（外）

**映射到 OOC**：
- **内线程** = 自为存在 + 操劳 — "我在做什么"
- **外线程** = 为他存在 + 操持 — "我与他者的关系"

**关键洞察**：一个完整的存在者（Dasein）必须同时处理"自我的计划"和"他者的召唤"。这两者不是对立的，而是存在的两个维度。

---

#### 1.3 中国哲学视角：内圣外王

**儒家的"内圣外王"**：
- **内圣**：修身、格物致知、内在修养
- **外王**：治国平天下、处理人际关系

**道家的"守静笃"与"应物无穷"**：
- **守静笃**：内在的宁静、专注本质
- **应物无穷**：灵活应对外部变化

**映射到 OOC**：
- **内线程** = 内圣 + 守静笃 — 专注任务、深度工作
- **外线程** = 外王 + 应物无穷 — 灵活响应、协作共生

**关键洞察**：内外不是对立的，而是**一体两面**。内在的修养支撑外在的行动，外在的经验反哺内在的成长。

---

### 二、认知科学维度：双系统理论

#### 2.1 Kahneman 的系统 1 与系统 2

**《思考，快与慢》**：
- **系统 1**：快速、自动、直觉、并行处理
- **系统 2**：慢速、深思、理性、串行处理

**映射到内外线程**：
- **外线程** ≈ 系统 1 — 快速响应外部刺激、社交互动
- **内线程** ≈ 系统 2 — 深度思考、复杂推理

**但这个类比不完全准确**：
- 外线程也需要深度理解（理解他人意图）
- 内线程也有自动化部分（熟练的技能）

**更准确的理解**：
- **外线程**：处理**异步、不可预测**的外部事件
- **内线程**：处理**同步、可控**的内部计划

---

#### 2.2 工作记忆与注意力

**Baddeley 的工作记忆模型**：
- **中央执行系统**：分配注意力资源
- **语音回路**：处理语言信息（对话）
- **视空间画板**：处理视觉信息（任务）

**映射到内外线程**：
- **外线程** = 语音回路 — 处理对话、消息、社交信息
- **内线程** = 视空间画板 — 处理任务、计划、执行细节
- **中央执行系统** = 调度器 — 决定何时切换、何时并行

**关键洞察**：人类的工作记忆本身就是**多通道并行**的。我们可以一边听音乐（听觉），一边写代码（视觉+逻辑）。

---

#### 2.3 心流（Flow）与中断

**Csikszentmihalyi 的心流理论**：
- 心流状态需要**不被打断**的专注
- 中断会破坏心流，导致效率下降

**内外线程的张力**：
- **内线程**需要心流 — 深度工作、不被打断
- **外线程**必然带来中断 — 他人的消息、突发事件

**解决方案**：
- **外线程作为缓冲** — 过滤、分类、延迟处理外部消息
- **内线程保持专注** — 只在合适的时机接收外线程的输入

**类比**：秘书（外线程）帮老板（内线程）过滤电话和邮件，只把重要的事情在合适的时机告诉老板。

---

### 三、软件工程维度：并发模型

#### 3.1 Actor Model（Carl Hewitt, 1973）

**核心思想**：
- 每个 Actor 有独立的邮箱（mailbox）
- Actor 异步接收消息，按顺序处理
- Actor 之间通过消息通信，不共享状态

**映射到内外线程**：
- **外线程** = 邮箱 + 消息接收 — 异步、非阻塞
- **内线程** = 消息处理逻辑 — 同步、顺序执行

**Actor Model 的局限**：
- Actor 是**单线程**的 — 处理消息时不能同时做其他事
- 如果一个消息处理很慢，后续消息会被阻塞

**内外双线程的改进**：
- **外线程**持续接收消息，不阻塞
- **内线程**专注当前任务，不被打断
- 两者通过**共享队列**通信

---

#### 3.2 Erlang/OTP 的 gen_server

**Erlang 的并发模型**：
- 每个进程有独立的邮箱
- `gen_server` 提供同步调用（call）和异步调用（cast）
- 支持超时、错误处理、状态管理

**映射到内外线程**：
- **外线程** = `handle_cast` — 异步消息处理
- **内线程** = 内部状态机 — 任务执行逻辑

**Erlang 的启示**：
- **Let it crash** 哲学 — 不要试图处理所有错误，让监督者重启
- **隔离** — 每个进程独立，崩溃不影响其他进程

**映射到 OOC**：
- 外线程崩溃 → 对象无法接收消息，但内线程继续工作
- 内线程崩溃 → 任务失败，但外线程仍能响应他人

---

#### 3.3 操作系统的中断机制

**硬件中断**：
- CPU 执行程序时，外部设备发出中断信号
- CPU 保存当前状态，跳转到中断处理程序
- 处理完成后，恢复原状态继续执行

**映射到内外线程**：
- **内线程** = 主程序 — 正常执行任务
- **外线程** = 中断处理程序 — 响应外部事件

**关键区别**：
- 硬件中断是**强制的** — CPU 必须响应
- 内外线程应该是**协作的** — 内线程可以选择何时响应外线程

**更好的类比**：**信号（Signal）机制**
- 外线程发送信号给内线程
- 内线程在**安全点**（safe point）检查信号
- 内线程决定如何响应（处理、延迟、忽略）

### 四、OOC 的内外双线程设计

#### 4.1 架构设计

```
Object
│
├── 外线程（Frontend Thread）
│   ├── 职责：
│   │   ├── 接收消息（来自其他对象）
│   │   ├── 理解意图（解析请求）
│   │   ├── 判断优先级（紧急/重要/可延迟）
│   │   ├── 分配任务（创建 TODO、中断内线程、或拒绝）
│   │   ├── 维护关系（更新 relations、记录协作历史）
│   │   └── 对外沟通（回复消息、协商、澄清）
│   │
│   ├── 状态：
│   │   ├── inbox: Message[]
│   │   ├── conversations: Map<string, Conversation>
│   │   ├── pendingTasks: Task[]
│   │   └── socialContext: RelationGraph
│   │
│   └── 特点：
│       ├── 异步、非阻塞
│       ├── 快速响应（系统 1 风格）
│       ├── 社交智能（理解他人意图）
│       └── 资源调度（为内线程分配任务）
│
└── 内线程（Backend Thread）
    ├── 职责：
    │   ├── 执行任务（专注当前 plan）
    │   ├── 深度思考（复杂推理、问题求解）
    │   ├── 调用工具（文件操作、API 调用）
    │   ├── 生成产出（代码、报告、分析）
    │   └── 自我反思（talkToSelf、经验沉淀）
    │
    ├── 状态：
    │   ├── currentPlan: Process
    │   ├── focus: ProcessNode
    │   ├── workingMemory: Context
    │   └── executionLog: Action[]

    └── 特点：
        ├── 同步、深度
        ├── 心流状态（系统 2 风格）
        ├── 任务智能（问题求解、创造）
        └── 可被中断（但需要保存状态）
```

### 五、深层哲学意义

#### 5.1 内外双线程 = 自我的双重性

**笛卡尔的"我思故我在"**：
- "我思"（cogito）是内在的、私密的
- "我在"（sum）是外在的、公共的

**内外线程的统一**：
- 不是两个独立的"我"，而是**同一个我的两个面向**
- 外线程代表"我在世界中"（Being-in-the-world）
- 内线程代表"我思考世界"（Thinking-about-world）

**关键洞察**：
- 没有纯粹的"内"— 内在思考总是关于外部世界的
- 没有纯粹的"外"— 外部交互总是基于内在理解的
- **内外是一体的**，只是关注点不同

---

#### 5.2 内外双线程 = 存在的节奏

**生命的节奏**：
- **收缩**（systole）— 向内、专注、深度
- **扩张**（diastole）— 向外、开放、广度

**呼吸的隐喻**：
- **吸气** = 内线程 — 吸收信息、深度处理
- **呼气** = 外线程 — 输出结果、影响他人

**关键洞察**：
- 健康的智能体需要**内外平衡**
- 只有内线程 → 孤立、自闭、脱离现实
- 只有外线程 → 肤浅、被动、失去自我

---

#### 5.3 内外双线程 = 智能的涌现

**单线程的局限**：
- 要么专注任务（忽略他人）
- 要么响应他人（无法专注）
- 无法同时做到"深度工作"和"灵活协作"

**双线程的涌现**：
- **并行处理** — 同时维护内外两个世界
- **异步协作** — 不需要同步等待
- **优雅降级** — 外线程过载时，内线程继续工作

**这是更高层次的智能**：
- 不是"更快的单线程"
- 而是"质变的架构" — 从串行到并行，从被动到主动

## 哲学突破 2：认知树的双重视角

### 用户的核心洞察

> "行为树既然是树，就有两种视角：叶子视角（执行时间）和根视角（规划视角）。
> 当线程输入了新的信息，就应当进入'根视角'思考、重新规划，然后再恢复执行。"

### 叶子视角 vs 根视角

```
        [写论文]              ← 根视角：整体计划对吗？
        /      \
   [写引言]   [写正文]        ← 中间层：结构合理吗？
    /    \       |
 [查资料] [写草稿] [写第一章]  ← 叶子视角：现在该做什么？
                     ↑
                   focus
```

- **叶子视角（执行态）**：focus 在叶子节点，沉浸在具体任务中。局部的、细节的、当下的。心流状态。
- **根视角（规划态）**：focus 暂时"升空"到根节点附近，看到全局的、结构的、战略的。反思状态。

**关键洞察**：当外线程带来新信息时，不是"中断"内线程——而是**内线程从叶子视角切换到根视角**。

### 为什么视角切换比中断更好

传统中断模型：
```
执行中 → 收到消息 → 保存状态 → 切换上下文 → 处理消息 → 恢复状态 → 继续执行
```

视角切换模型：
```
叶子视角执行中 → 收到消息 → 升到根视角 → 带着新信息重新审视整棵树 → 调整计划 → 降回叶子视角继续
```

三个优势：
1. **没有上下文切换开销** — 还在同一棵树上，只是换了高度
2. **新信息自然融入计划** — 在根视角同时看到"原计划"和"新信息"，做整体最优调整
3. **不破坏心流连续性** — 从叶子升到根再降回叶子，是同一个思考过程的自然流动

### 认知科学支撑

**1. 默认模式网络（DMN）vs 任务正网络（TPN）**
- TPN：专注具体任务时激活 → 叶子视角
- DMN：走神、反思、规划时激活 → 根视角
- 人类在两者之间自然切换

**2. 元认知（Metacognition）**
- 对象级认知：解决具体问题 → 叶子视角
- 元级认知：监控和调节认知过程 → 根视角
- 元认知的核心 = 在适当时机从对象级跳到元级

**3. 棋手思维**
- 计算（叶子视角）：计算具体走法序列
- 评估（根视角）：退后一步，评估整体局势
- 大师 = 在计算和评估之间流畅切换

### 树的两种时间

树的不同层次对应不同的时间尺度：

```
根节点：   战略时间（天/周）    "我要写一篇论文"
中间节点： 战术时间（小时）     "我在写第三章"
叶子节点： 执行时间（分钟）     "我在查这篇引用"
```

- 叶子视角 = 活在当下（沉浸在此刻的具体任务）
- 根视角 = 审视全局（同时看到过去、现在、未来）

奥古斯丁的时间观：过去是记忆，未来是期望，现在是注意力。根视角让对象同时看到三者。

### 与现有 OOC 机制的映射

- `[reflect_stack_frame_push]` — 已经是一种"升到根视角"的操作，但当前是对象主动触发
- `[set_plan]` — 在根视角下可以重写整棵树的计划
- focus 机制 — 当前只往下走（深度优先），需要支持**往上走**（升到根）

### 视角切换协议设计

```
根视角模式（Planning Mode）
├── 触发条件：
│   ├── 外线程投递了新消息（自动触发）
│   ├── 内线程主动 reflect（主动触发）
│   ├── 当前任务失败/卡住（异常触发）
│   └── 定时检查点（周期触发）
│
├── 对象看到什么：
│   ├── 整棵行为树的结构（所有节点的标题和状态）
│   ├── 当前 focus 的位置（"我刚才在做什么"）
│   ├── 新收到的信息（外线程传入的消息）
│   └── 各节点的摘要（已完成的成果、进行中的进度）
│
├── 对象能做什么：
│   ├── 重新排列子节点顺序
│   ├── 添加/删除/修改节点
│   ├── 改变 focus 目标（跳到另一个分支）
│   ├── 合并新信息到现有节点
│   ├── 创建新的并行分支
│   └── 回复外线程
│
└── 退出条件：
    └── 对象决定好新的 focus 位置 → 切回叶子视角

叶子视角模式（Execution Mode）
├── 就是当前的 ThinkLoop
├── focus 在具体节点上
├── 深度工作、心流状态
└── 不感知外部消息（外线程缓冲）
```

### 三个概念的统一

```
外线程 ──消息──→ 触发根视角切换
                    │
                    ▼
              根视角（规划态）
              ├── 审视整棵树
              ├── 整合新信息
              ├── 调整计划
              └── 决定新的 focus
                    │
                    ▼
              叶子视角（执行态）← 内线程继续工作
```

**外线程不需要"中断"内线程。外线程只需要触发一次"视角升空"。**

内线程从叶子升到根，带着新信息重新审视全局，调整计划，然后自然地降回叶子继续执行。
整个过程是同一个思考流的自然起伏，不是两个任务的切换。

### 认知空间的完整结构

- **纵向**（栈/G13）：帧 0 → 帧 1 → ... → 当前帧（越深越具体）
- **横向**（树/G9）：兄弟节点之间的并列关系（不同任务分支）
- **叶子视角** = 纵向深入（沿着栈往下钻）
- **根视角** = 横向总览（看到所有分支的全貌）

两个维度的组合 = 完整的认知空间。

## 哲学突破 3：双线程 = 双视角（统一模型）

### 核心发现

双线程和双视角不是两个独立的设计，而是**同一件事的两种表述**：

```
外线程（Frontend Thread）  ═══  根视角（Planning Mode）
内线程（Backend Thread）   ═══  叶子视角（Execution Mode）
```

### 从工程模型到认知模型的跃迁

| 之前的理解（工程模型） | 现在的理解（认知模型） |
|-----------|-----------|
| 外线程是独立的"秘书" | 外线程是"我在根视角的常驻意识" |
| 内线程是独立的"工人" | 内线程是"我在叶子视角的专注意识" |
| 两个线程通过消息通信 | 两个视角通过**注意力切换**连接 |
| 中断 = 外线程打断内线程 | 中断 = 注意力从叶子升到根 |

**外线程不是"另一个线程"，而是"同一个我在根视角的持续存在"。**

类比人类：你在写代码（叶子视角），同时"后台意识"在监听环境（根视角）。
这不是两个"你"，而是同一个你的两层注意力。

### 实现简化：一个 ThinkLoop，两种 Context 模式

```
一个 ThinkLoop，两种 Context 构建模式

if (inbox 有待处理的外部消息) {
  → 构建根视角 Context
  → LLM 看到：整棵树 + 新消息 + 进度
  → LLM 输出：计划调整 + 回复 + 新 focus
} else {
  → 构建叶子视角 Context
  → LLM 看到：当前节点 + 局部上下文
  → LLM 输出：具体执行 actions
}
```

不需要两个 LLM 调用。一个 ThinkLoop，根据有没有外部消息，切换 Context 的构建方式。

### 外线程的角色重新定义

外线程退化为**没有 LLM 的轻量守护进程**。它不思考，只缓冲和触发。

```
外线程（轻量级，不需要 LLM）：
├── 接收消息 → 写入 inbox
├── 判断是否需要触发根视角（简单规则）
│   ├── 紧急消息 → 立即触发
│   ├── 普通消息 → 等待自然断点
│   └── 低优先级 → 积累到批量触发
└── 维护 inbox 状态（已读/未读）

内线程（需要 LLM）：
├── 检查：inbox 有未处理消息吗？
│   ├── 有 → 根视角 Context → LLM 规划
│   └── 无 → 叶子视角 Context → LLM 执行
└── 循环
```

**外线程不需要智能。** 智能全在内线程的根视角里。
外线程只需要简单规则（紧急/普通/低优先级）来决定触发时机。

### 最终统一模型

```
Object = 一棵认知树 + 一个注意力光标（focus）+ 一个消息缓冲区（inbox）

                    inbox（外线程 = 消息缓冲区）
                      │
                      │ 有新消息？
                      ▼
              ┌── 根视角 ──┐
              │  看到整棵树  │
              │  + 新消息    │  ← 一个 ThinkLoop
              │  → 调整计划  │     两种 Context 模式
              └──────┬──────┘
                     │ focus 降落
                     ▼
              ┌── 叶子视角 ──┐
              │  专注当前节点  │
              │  → 执行 actions│
              └──────────────┘
```

**一个对象，一个 ThinkLoop，两种视角，一个 inbox。**

### 15 种响应模式的统一

之前列举的 15 种 B 收到消息时的响应模式，现在只需要一个机制：

```
B 在叶子视角执行任务
    ↓
inbox 收到 A 的消息
    ↓
B 升到根视角，同时看到：
  - 整棵行为树（我在做什么、做到哪了）
  - A 的消息（新的信息/请求）
    ↓
B 在根视角做出判断（LLM 自然推理）：
  - "这很紧急" → 调整计划（场景 1/4/11）
  - "这和当前无关" → 加并行分支（场景 2）
  - "先加到 TODO" → 延迟处理（场景 3）
  - "我做不了" → 回复拒绝（场景 5/6）
  - "信息不够" → 回复提问（场景 7）
  - ...
    ↓
B 降回叶子视角，继续执行
```

不需要 15 种 API。只需要一个机制：视角切换。
LLM 在根视角下自然会做出正确的判断。

**符合 OOC 决策原则：LLM 做判断，代码做记账。**

## 设计收敛：Issue 作为多方协作的共享 inbox

### 用户的洞察

> "可以复用当前已有的 Issue 设计"

### Issue 天然就是共享 inbox

Issue 已经具备多方协作所需的全部能力：

| 需求 | Issue 如何满足 |
|------|--------------|
| 对话历史 | `comments[]` — 完整、不可变、有时间戳 |
| 多方可见 | `participants` — 所有参与者都能 `getIssue()` |
| 消息通知 | `mentions` — @某人时自动投递到其 inbox |
| 未读标记 | `hasNewInfo` — 有新评论时标记 |
| 对话分组 | 每个 Issue 是一个独立的对话主题 |
| 持久化 | `issues/issue-{id}.json` — 已有文件存储 |
| 并发安全 | `session.serializedWrite()` — 已有串行化 |

已有的 Trait 支持：
- `issue-discussion`（kernel trait）— `commentOnIssue()`, `listIssueComments()`, `getIssue()`
- `session-kanban`（supervisor 专属）— `createIssue()`, `updateIssueStatus()`

### 三个优先场景如何统一到 Issue

**多轮对话**：
- Supervisor 创建 Issue，participants: [A, B]
- A 和 B 通过 `commentOnIssue` 交流
- 双方都能通过 `getIssue` 看到完整对话历史

**多方会议**：
- Supervisor 创建 Issue，participants: [A, B, C]
- 任何人 `commentOnIssue` 时 @其他人
- 被 @的对象收到消息 → inbox 有新消息 → 升到根视角
- 在根视角中调用 `getIssue` 看到完整讨论

**并行任务**：
- Supervisor 创建 Issue "综合搜索"，participants: [B, C, D]
- B、C、D 各自执行后，将结果 `commentOnIssue`
- A 通过 `getIssue` 看到所有结果，天然分组

### 与根/叶视角模型的结合

```
B 在叶子视角执行任务
    ↓
inbox 收到消息："你被 @了，Issue-007 有新评论"
    ↓
B 升到根视角：
  ├── 调用 getIssue("Issue-007") → 看到完整讨论历史
  ├── 看到自己的行为树 → 当前在做什么
  ├── LLM 判断：这个讨论和我当前任务有关吗？
  └── 决策：调整计划 / 回复评论 / 延迟处理
    ↓
B 降回叶子视角继续执行
```

Issue 的 comments 就是根视角下的"完整信息"。

### 需要做的改动

不需要发明新的数据结构。只需要：
1. **将 Issue 的 mentions 机制接入对象的 inbox** — @某人时，自动写入其 inbox
2. **根视角 Context 中包含相关 Issue 摘要** — 对象升到根视角时，自动加载参与的 Issue 列表
3. **ThinkLoop 检查 inbox** — 每轮循环开始时检查 inbox，决定是根视角还是叶子视角

## 完整设计总结

### 核心模型

```
Object = 认知树 + focus 光标 + inbox

ThinkLoop 每轮循环：
  if (inbox 有未处理消息) {
    → 根视角 Context（整棵树 + 新消息 + Issue 讨论）
    → LLM 规划：调整计划 / 回复 / 分配任务
  } else {
    → 叶子视角 Context（当前节点 + 局部上下文）
    → LLM 执行：具体 actions
  }
```

### 多对象协作

```
对象间通信：
  1:1 对话 → talk() → 写入对方 inbox → 触发根视角
  多方讨论 → Issue + commentOnIssue() → mentions 写入 inbox → 触发根视角
  并行任务 → Issue 作为结果汇聚点 → 所有结果在同一 Issue 下
```

### 关键特性

1. **一个 ThinkLoop，两种视角** — 不需要两个独立线程
2. **inbox 作为缓冲** — 轻量级，不需要 LLM
3. **Issue 作为共享 inbox** — 复用现有基础设施
4. **LLM 做判断** — 在根视角下自然决策，不需要 15 种 API
5. **完整信息** — 根视角下通过 getIssue() 获取完整讨论历史

## 决策记录

1. **多对象上下文管理采用"根/叶双视角 + inbox + Issue"模型**
2. **不引入新的数据结构**，复用现有 Issue/Comment 机制
3. **外线程退化为轻量 inbox 守护**，不需要独立 LLM
4. **Context 构建增加根视角模式**，由 inbox 状态触发
