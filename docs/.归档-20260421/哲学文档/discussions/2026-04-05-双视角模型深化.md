## 已达成的共识

### 核心模型

```
Object = 认知树 + focus 光标 + inbox

ThinkLoop 每轮循环：
  if (inbox 有未处理消息) {
    → 根视角 Context（整棵树 + 新消息 + Issue 讨论）
    → LLM 规划：调整计划 / 回复 / 分配任务
  } else {
    → 叶子视角 Context（当前节点 + 局部上下文）
    → LLM 执行：具体 actions
  }
```

### 四个设计决策

1. **双线程 = 双视角**：不是两个独立线程，而是同一个 ThinkLoop 的两种 Context 构建模式
2. **外线程 = 轻量 inbox 守护**：不需要 LLM，只做消息缓冲和触发
3. **Issue = 共享 inbox**：多方协作复用现有 Issue/Comment 机制
4. **LLM 做判断**：在根视角下自然决策，不需要 15 种 API

### 多对象协作方式

```
1:1 对话  → talk() → 写入对方 inbox → 触发根视角
多方讨论  → Issue + commentOnIssue() → mentions 写入 inbox → 触发根视角
并行任务  → Issue 作为结果汇聚点
```

## 深化议题 2：inbox 的数据结构与触发规则

### inbox 消息的数据结构

```typescript
interface InboxMessage {
  id: string;
  from: string;              // 发送者
  content: string;           // 消息内容
  timestamp: number;
  source: "talk" | "issue";  // 来源：直接 talk 或 Issue mention
  issueId?: string;          // 如果来自 Issue，关联的 Issue ID
  status: "unread" | "marked";
  mark?: {
    type: "ack" | "ignore" | "todo";
    tip: string;             // mark 的解释信息（可追溯）
    markedAt: number;
  };
}
```

### mark(messageId, type, tip) — 消息确认机制

```typescript
mark(messageId: string, type: "ack" | "ignore" | "todo", tip: string)
```

| type | 含义 | 下次根视角是否展示 | 示例 tip |
|------|------|------------------|---------|
| **ack** | 已处理 | 否 | "已回复 A" / "已加入第三章计划" |
| **ignore** | 主动忽略 | 否 | "与当前任务无关" / "重复消息" |
| **todo** | 暂存待办 | 否（但 todo 列表可见） | "等第二章写完再处理" / "需要先问 C" |

**未被 mark 的消息，下次根视角仍然展示。** 保证不丢消息。

### 消息生命周期

```
到达 inbox (unread)
    │
    ▼
根视角 Context 展示
    │
    ├── mark(ack, "已回复")     → 归档
    ├── mark(ignore, "无关")    → 归档
    └── mark(todo, "等写完再说") → 进入 todo 队列
                                      │
                                      ▼
                                 条件满足时重新浮现到根视角
```

### todo 的重新浮现机制

**设计决策**：todo 不挂在消息上，而是挂在行为树节点上。

根视角下 LLM 的两种操作：
1. `mark(messageId, type, tip)` — 处理 inbox 消息
2. `addTodo(nodeId, content)` — 在行为树节点上挂 todo

典型流程：
```
B 在叶子视角写第二章
    ↓
inbox 收到 A 的消息："第一章数据有误"
    ↓
B 升到根视角，LLM 决策：
  1. mark(msg-001, "todo", "数据修正需要等第二章写完再处理")
  2. addTodo("node-写正文", "回头修正第一章数据，参考 A 的反馈")
    ↓
B 降回叶子视角，继续写第二章
    ↓
... 第二章写完，focus 推进到 "node-写正文" 层级 ...
    ↓
系统检测到 "node-写正文" 有未完成的 todo → 触发根视角
    ↓
B 升到根视角，看到 todo → 创建新子节点 [修正第一章数据]
```

todo 的浮现时机由行为树的推进自然驱动，不需要定时器或空闲检测。

### todo 的数据结构

```typescript
interface TodoItem {
  id: string;
  content: string;           // todo 内容
  sourceMessageId?: string;  // 关联的 inbox 消息（可选）
  status: "pending" | "done";
  createdAt: number;
  doneAt?: number;
}
```

todo 挂在 ProcessNode 上：
```typescript
interface ProcessNode {
  // ... 现有字段
  todos?: TodoItem[];        // 该节点的 todo 列表
}
```

### 触发规则：什么时候从叶子视角切换到根视角？

满足任一条件即触发：

1. **inbox 有 unread 消息** — 外部驱动（别人找我）
2. **当前节点（或祖先节点）有 pending todo，且子任务已全部完成** — 内部驱动（待办到期）
3. **对象主动 reflect** — 已有机制
4. **当前任务失败/卡住** — 已有机制

## 深化议题 4：多线程对话与视角的递归性

### 用户的核心洞察

> "一个行为树，允许发起多线程在不同的节点进行思考。"
> "C 想了解 B 的细节，可以 self_talk('B', '你产出的文档路径在哪里')，创建一个新线程，上下文是节点 B 所在的上下文。"
> "不同 OOC Object 之间对话时，总是创建节点为 Root 的线程。"
> "一个中间节点，对上它具有叶子视角，对下它具有根视角。"

### 视角的递归性

之前我们说"根视角"和"叶子视角"是两种模式。但这个洞察揭示了更深的结构：**视角不是二元的，而是递归的**。

```
        [Root]
        /    \
      [A]    [B]
      / \      |
    [C] [D]   [E]
```

对于节点 A：
- **对上（面向 Root）**：A 是叶子 — "我是 Root 的一个子任务"
- **对下（面向 C、D）**：A 是根 — "我管理 C 和 D 的执行"

**每个中间节点都同时是叶子和根。** 这不是两种模式的切换，而是**同一个节点的两个面**。

这和之前的"内外双线程 = 同一个我的两层注意力"完全一致——只是从二元推广到了递归。

### 节点间对话：self_talk

当 C 执行完毕，D 开始执行时，D 只能看到 C 的摘要（结构化遗忘）。但如果 D 需要 C 的细节：

```
D: self_talk("C", "你产出的文档路径在哪里")
    ↓
系统在节点 C 的上下文中创建一个新线程
    ↓
这个线程能看到 C 的完整 actions 历史
    ↓
线程回复 D："文档在 files/report.md"
    ↓
D 收到回复，继续执行
```

**self_talk 的本质**：在已完成节点的上下文中"复活"一个临时线程，用该节点的视角回答问题。

这解决了一个长期痛点：**结构化遗忘导致的信息丢失**。遗忘是必要的（G5），但有时需要"回忆"——self_talk 就是按需回忆的机制。

### 跨对象对话：总是创建 Root 线程

```
Object A 的行为树：
        [Root] ← 跨对象消息总是在这里创建线程
        /    \
      [X]    [Y]
```

当 Object B 给 A 发消息时：
- 消息到达 A 的 inbox
- 系统在 A 的 Root 节点创建新线程
- 这个线程的上下文 = Root 视角（看到整棵树的摘要）
- 线程处理消息后，可能调整行为树、回复 B、或创建新节点

**为什么是 Root？** 因为跨对象消息的处理需要全局视角——只有 Root 能看到整棵树，才能做出正确的调度决策。

### 统一模型：线程 = 在某个节点的上下文中思考

```
线程的本质：
  thread = (anchorNode, context, purpose)

  anchorNode: 线程锚定在哪个节点
  context:    该节点的上下文（对上是叶子，对下是根）
  purpose:    为什么创建这个线程

三种线程创建场景：

1. 跨对象消息 → 创建 Root 线程
   anchorNode = Root
   context = 整棵树摘要 + inbox 消息
   purpose = 处理外部消息

2. 节点间对话 → 创建目标节点线程
   self_talk("C", question)
   anchorNode = C
   context = C 的完整 actions 历史
   purpose = 按需回忆

3. 正常执行 → 就是当前的 ThinkLoop
   anchorNode = focus 节点
   context = 叶子视角
   purpose = 执行任务
```

### 中间节点的双重视角

```
        [Root]
        /    \
      [A]    [B]
      / \
    [C] [D]

节点 A 的视角：

对上（叶子视角）：
  - A 是 Root 的子任务
  - A 只需要汇报自己的进度和结果
  - A 看不到 B 的细节（只有摘要）

对下（根视角）：
  - A 管理 C 和 D 的执行
  - A 能看到 C 和 D 的状态
  - A 决定 C 和 D 的执行顺序
  - C 或 D 完成时，A 检查 todo，决定下一步
```

当 A 收到来自 Root 的消息（"进度如何？"）：
- A 以**叶子视角**回应 Root

当 A 发现 C 失败了：
- A 以**根视角**决定是重试 C 还是跳过

**同一个节点，同一个 ThinkLoop，视角由"我在和谁交互"决定。**

### 与之前设计的统一

之前的"根视角/叶子视角切换"现在变成了更通用的模型：

| 之前的理解 | 现在的理解 |
|-----------|-----------|
| 两种视角模式 | **每个节点天然具有双重视角** |
| inbox 触发根视角 | **消息到达时在对应节点创建线程** |
| 一个 ThinkLoop 两种 Context | **线程的 Context 由 anchorNode 决定** |
| 外线程 = 轻量守护 | **inbox = 消息路由到正确的 anchorNode** |

### self_talk 的 API 设计

```typescript
// 节点间对话：在目标节点的上下文中创建临时线程
self_talk(targetNodeId: string, message: string): Promise<string>

// 跨对象对话（已有）：消息路由到对方 Root
talk(targetObject: string, message: string): void

// 在当前节点创建 todo
addTodo(content: string, sourceMessageId?: string): void

// 处理 inbox 消息
mark(messageId: string, type: "ack" | "ignore" | "todo", tip: string): void
```

### 哲学意义

这个设计揭示了认知树的**分形结构**：

- 每个节点都是一个微型的"智能体"
- 对上它是执行者（叶子），对下它是管理者（根）
- 整棵树是智能体的递归嵌套

这和 OOC 的 G1（万物皆对象）形成了呼应：
- G1 说"对象是唯一建模单元"
- 现在我们发现：**行为树的每个节点也是一个微型对象**——有自己的上下文、视角、和交互能力

## 哲学突破 4：stack_push = 启动子节点线程

### 用户的核心洞察

> "一次 stack push 可以变成一个新节点的 thread 的启动。parent node 可以等待这个子节点完成，也可以继续异步发起第二个子节点，然后一并等待两个节点完成。每个节点都是独立的线程进行思考、执行。"

### 从"栈操作"到"线程启动"

之前的理解：
```
stack_push("写第一章")
  → 创建子节点，focus 移入，开始执行
  → 执行完毕，stack_pop，focus 回到父节点
  → 串行：必须等子节点完成才能继续
```

现在的理解：
```
stack_push("写第一章")
  → 创建子节点，启动一个独立线程
  → 父节点可以选择：
    a) await — 等待子线程完成（同步，和之前一样）
    b) continue — 不等待，继续发起下一个子节点（异步）
    c) await_all — 发起多个子节点后，一并等待全部完成
```

### 这统一了所有执行模式

```
串行执行（当前模式）：
  parent: push(A), await(A), push(B), await(B), push(C), await(C)

并行执行：
  parent: push(A), push(B), push(C), await_all(A, B, C)

混合执行：
  parent: push(A), await(A), push(B), push(C), await_all(B, C)
```

**不需要区分"顺序节点"和"并行节点"。** 只有一种节点，执行模式由父节点的 await 策略决定。

### 每个节点 = 独立线程

```
        [Root] ─── thread_0（根线程）
        /    \
      [A]    [B]
      │       │
   thread_1  thread_2（A 和 B 各自独立思考）
      / \
    [C] [D]
    │     │
 thread_3 thread_4（C 和 D 各自独立思考）
```

每个线程：
- 有自己的 ThinkLoop
- 有自己的 Context（由 anchorNode 决定）
- 有自己的 focus（在自己的子树内移动）
- 独立调用 LLM

父线程和子线程的关系：
- 父线程 push 子节点 → 子线程启动
- 子线程完成 → 通知父线程（类似 Promise resolve）
- 父线程可以 await 单个或 await_all 多个

### self_talk 和跨对象 talk 的统一

现在所有的"对话"都是同一个机制：**在某个节点下创建子节点，启动线程**。

```
1. stack_push("写第一章")
   → 在当前节点下创建子节点，启动线程
   → 线程执行任务

2. self_talk("C", "文档路径在哪")
   → 在节点 C 下创建子节点，启动线程
   → 线程在 C 的上下文中回答问题
   → 回答完毕，线程结束，子节点记录过程

3. talk("ObjectB", "请帮我搜索")
   → 在 ObjectB 的 Root 下创建子节点，启动线程
   → 线程在 ObjectB 的全局上下文中处理消息

三种操作的本质完全相同：创建子节点 + 启动线程。
区别只在于 anchorNode 的位置和线程的 Context。
```

### 线程的生命周期

```
线程状态机：
  created → running → finished
                │
                └→ waiting（await 子线程时）

线程结束时：
  - 过程持久化到对应的行为树节点（actions 历史）
  - 通知父线程（如果父线程在 await）
  - 线程销毁，但节点保留
```

### API 设计

```typescript
// 创建子节点并启动线程（原 stack_push 的升级）
push(title: string, options?: {
  traits?: string[];      // 子线程激活的 traits
  description?: string;   // 任务描述

// 等待单个子线程完成
await(node: NodeHandle): Promise<Result>

// 等待多个子线程全部完成

// 在已完成节点的上下文中创建临时线程（按需回忆）

// 跨对象对话（在对方 Root 下创建线程）
talk(targetObject: string, message: string): void

// 处理 inbox 消息
mark(messageId: string, type: "ack" | "ignore" | "todo", tip: string): void

// 在当前节点创建 todo
addTodo(content: string, sourceMessageId?: string): void
```

### 与 OOC 哲学的深层统一

**G13（认知栈）的重新解读**：
- 之前：认知栈 = 帧的嵌套，push/pop 是串行的
- 现在：认知栈 = 线程树，push 启动独立线程，支持并行

**G9（行为树）的重新解读**：
- 之前：行为树 = 静态的计划结构，focus 光标遍历
- 现在：行为树 = 动态的线程树，每个节点是活的线程

**G4（程序行动）的重新解读**：
- 之前：一个 ThinkLoop 串行执行
- 现在：多个 ThinkLoop 并行执行，每个节点一个

**统一效果**：
- stack_push = 启动子线程
- stack_pop = 子线程完成
- self_talk = 在已完成节点启动临时线程
- talk = 在对方 Root 启动线程
- 并行执行 = 多个子线程同时 running
- 串行执行 = 父线程 await 每个子线程

**一切都是线程。一切都是节点。**

### 议题 6：调度器与并行模型（已达成共识）

**设计决策**：所有线程真并行，无论 Object 内部还是跨 Object。

**Scheduler 简化为全局线程池**：
```
Scheduler 的工作：
  1. 收集所有 running 状态的线程
  2. 每个线程执行一轮 ThinkLoop（并行，Promise.all）
  3. 检查哪些线程 finished → 通知父线程
  4. 检查哪些父线程被唤醒（await 的子线程完成了）
  5. 重复
```

Scheduler 不需要关心"线程属于哪个 Object"。只看状态：running 就执行，waiting 就跳过，finished 就通知。

**跨 Object talk 的完整流程**：
```
A 的线程 T1: talk("B", "请搜索")
    ↓
1. A 当前节点下创建子节点 "等待 B 回复" → T1 进入 waiting
2. B 的 Root 下创建子节点 "处理 A 的请求" → 启动 T2 (running)
3. Scheduler: T1 waiting(跳过), T2 running(执行), B 的其他线程(并行执行)
4. T2 完成 → 回复写入 A 的等待节点 → T1 变为 running
5. T1 继续执行，能看到 B 的回复
```

A 和 B 的线程完全并行。A 等待 B 时，A 的其他线程继续执行。
