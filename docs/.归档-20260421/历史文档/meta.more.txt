## 一、核心哲学思考

### 1.1 从静态到动态的范式转变
    OOC 系统的本质
        将传统 Agent 的"文本 Context"升级为"活的对象生态"
        不仅是技术实现的变化，更是认知模型的革命
            传统 Agent
                Context 是死的文本
                每次对话都需要重新解析和理解
            OOC Agent
                Context 是活的对象
                具有状态、行为、记忆和演化能力

        类比
            从"函数式编程"到"面向对象编程"的跨越
            但更进一步
                对象不仅有数据和方法
                还有思维方式(bias)
                还有关系网络(relation)

### 1.2 元编程能力的哲学意义
    Object 可以"为自己编写方法"
        赋予了系统自我演化的能力
            自我认知
                Object 可以反思自己的行为模式
                发现不足
            自我优化
                Object 可以修改自己的方法
                改进执行效率
            自我扩展
                Object 可以为自己添加新的能力

        类似于人类的"元认知"能力
            不仅能思考问题
            还能思考"如何思考问题"

### 1.3 Bias 作为"思维视角"的深层含义
    Bias 不是传统意义上的"偏见"
        而是一种结构化的思维方式
            每个 Bias 代表一个特定的思考角度
                如安全审查、性能优化、用户体验
            多个 Bias 并发工作
                形成"多视角思考"
            Bias 的 when 条件
                实现了"情境感知"
                在合适的时机激活合适的思维方式

        类似于人类的"专家系统"
            面对不同问题时
            会自动切换到相应的专业思维模式

## 二、系统架构的深度分析

### 2.1 三层元类设计的合理性
    Stone（基础对象）
        定位
            最基础的对象类型，类似于"原子"
        特点
            纯数据 + 纯逻辑
            无思考能力
        适用场景
            配置对象
            数据模型
            工具类

    Flow（流程对象）
        定位
            具有思考能力和执行能力的动态过程对象
            从 Stone 派生，多继承自所属 Stone 实例和 Flow 元类
            是 OOC 系统中的核心活跃单元——所有"做事情"都通过 Flow 完成
        特点
            有状态机和生命周期
            有完整的 Bias 集合（可调用 LLM 进行思考）
            有执行能力（通过输出 Python 程序来行动）
            有聚焦能力（Focus）——可以自主拆分任务，创建子 Flow
            每个 Flow 是独立的进程/线程
        聚焦（Focus）机制
            Flow 可以通过 focus_bias 自主创建子 Flow
            Focus 是 Flow 的核心能力之一：
                评估当前任务的复杂度
                如果可分解，拆分为子任务并创建子 Flow
                子 Flow 继承父 Flow 的上下文，但拥有独立状态和生命周期
                子 Flow 完成后，结果通过 convergence_bias 收敛回父 Flow
            Flow 的派生树
                Stone 是根节点
                Flow 从 Stone 派生
                Flow 可以继续派生子 Flow，形成 Flow 派生树
                Flow 是 Stone 的私有对象，对外不可见
        适用场景
            任务流程
            工作流
            可递归分解的复合任务
            一切需要"思考+行动"的过程

    Chat（对话对象）
        定位
            Chat 是 Flow 的一种特化形式
            Chat = 由 talk() 方法创建的 Flow
            即：来自外部 caller 的对话/询问所触发的 Flow
        特点
            继承 Flow 的所有能力和 Bias（不增加额外 Bias）
            上下文来自对话历史（caller 的消息）
            支持多轮对话（对同一个 Chat 的 talk 不会重复创建 Flow）
        与普通 Flow 的区别
            触发方式: 由外部 caller 调用 talk() 创建（而非内部 focus 拆分）
            生命周期: 可持续接收后续消息（多轮），而非任务完成即结束
            上下文: 包含对话历史，而非仅任务参数
        适用场景
            外部对话交互
            跨 Stone 的协作（stoneA 向 stoneB 提问）
            用户与 Agent 的交互入口

    设计原则
        符合"复杂性递增"的原则
            Stone: 纯数据/定义，零 Bias，零开销
            Flow: 完整思考+执行能力，是核心活跃单元
            Chat: Flow 的对话特化形式，Bias 集合与 Flow 相同
        Stone 和 Flow 的关系是"静态 vs 动态"
            Stone 是定义，Flow 是过程
            Flow 从 Stone 派生，是 Stone 的"行动化身"
        Chat 和 Flow 的关系是"特化 vs 通用"
            Chat 不是"更高级的 Flow"
            Chat 是"来自对话的 Flow"——触发方式和生命周期不同，能力相同

### 2.2 Bias 的触发与检索机制
    问题
        Bias 数量爆炸
            当一个 Object 有几十甚至上百个 Bias 时
            如何高效触发？

    解决方案
        分层索引
            Root Index Bias
                Programming Bias Group
                    Go Bias
                    Python Bias
                    Java Bias
                Review Bias Group
                    Security Review Bias
                    Performance Review Bias
                    Code Style Review Bias
                Communication Bias Group
                    User Interaction Bias
                    Team Collaboration Bias

        优化思路
            向量检索
                将 Bias 的 prompt 向量化
                根据当前上下文快速检索相关 Bias
            优先级队列
                高频触发的 Bias 优先检查
            缓存机制
                缓存最近触发的 Bias
                避免重复计算

### 2.3 Relation 的活性依赖设计
    传统依赖 vs 活性依赖
        传统依赖
            A 依赖 B
            B 变化时 A 不知道
        活性依赖
            A 依赖 B
            B 变化时主动通知 A

    实现机制
        Relation 类
            包含
                from_obj: 依赖方
                to_obj: 被依赖方
                relation_type: 关系类型
                    "depends_on"
                    "collaborates_with"
                    "monitors"
            行为
                on_target_change(event)
                    当 to_obj 发生变化时
                    触发 from_obj 的相应 Bias

    应用场景
        依赖更新
            当依赖的库版本更新时
            自动触发兼容性检查
        协作通知
            当协作对象完成任务时
            自动触发下一步流程
        监控告警
            当被监控对象异常时
            自动触发应急响应

## 三、核心能力的实现路径

### 3.1 思考能力的实现
    单 Bias 思考
        流程
            获取 Bias
            构建 prompt（包含 Bias 的 prompt + self 的相关 field/method）
            调用 LLM
            返回结果

    多 Bias 并发思考
        流程
            对每个 Bias 进行思考
            收集所有结果
            综合多个 Bias 的思考结果
            返回最终决策

        综合策略
            投票机制
                多数 Bias 的结论作为最终决策
            权重机制
                不同 Bias 有不同的权重
                加权平均
            仲裁 Bias
                专门的 Bias 负责解决冲突

    递归思考（MainBrain）
        流程
            先思考"如何思考这个问题"（元思考）
            根据元思考的结果，选择合适的 Bias 组合
            用选定的 Bias 进行思考
            如果结果不满意，递归思考

        深度限制
            设置 max_depth 避免无限递归
            通常 2-3 层足够

### 3.2 记忆能力的实现
    短期记忆（工作记忆）
        存储位置
            Object 的 field 中
        特点
            随着对话进行动态更新
            对话结束后可选择性持久化

    长期记忆（持久化记忆）
        存储位置
            data.json 中
        内容
            关键事件
            重要决策
            学习到的经验
        决策机制
            通过 Bias 决定哪些信息值得长期保存

    关联记忆（Relation 记忆）
        存储位置
            Relation 网络中
        内容
            与其他 Object 的交互历史
        类比
            类似于人类的"社交记忆"

### 3.3 自我学习能力的实现
    经验积累
        流程
            分析任务执行过程
                使用 "self_reflection" Bias
                输入：task, result, feedback
            提取可复用的模式
                如果是可复用的
                    为自己编写新的方法
                    或者调整现有 Bias 的 prompt

    持续优化
        触发机制
            定期触发 "self_optimization" Bias
        优化内容
            分析自己的执行效率
            分析决策质量
            自动调整 Bias 的优先级和触发条件

### 3.4 对话能力的实现
    对话上下文管理
        包含
            历史对话记录
            当前对话状态
            相关的 field 和 method

        上下文压缩
            当对话历史过长时
            使用 Bias 进行摘要
            保留关键信息

    多轮对话
        状态维护
            在 field 中维护对话状态
            如：当前话题、待确认的信息

        意图识别
            使用 Bias 识别用户意图
            根据意图选择合适的响应策略

### 3.5 组织能力的实现
    创建子对象
        通过 world.create_object() 创建
        建立 Relation 关系

    任务分配
        根据子对象的能力
        分配合适的任务

    协作协调
        监听子对象的事件
        协调多个子对象的工作

## 四、关键技术挑战与解决方案

### 4.1 并发思考的一致性问题
    问题
        多个 Bias 并发思考时
        可能产生冲突的结论

    解决方案
        投票机制
            多数 Bias 的结论作为最终决策
        权重机制
            不同 Bias 有不同的权重
            加权平均
        仲裁 Bias
            专门的 Bias 负责解决冲突
            具有更高的权限

### 4.2 元编程的安全性问题
    问题
        Object 可以修改自己的代码
        可能引入 bug 或安全漏洞

    解决方案
        沙箱执行
            新生成的代码在沙箱中测试
            通过测试后才能正式使用
        版本控制
            保留代码的历史版本
            支持回滚
        审查 Bias
            专门的 Bias 负责审查新生成的代码
            检查安全性和正确性

### 4.3 Relation 网络的复杂度问题
    问题
        当 Object 数量增多时
        Relation 网络可能变得非常复杂

    解决方案
        分层管理
            将 Relation 分为"强依赖"和"弱依赖"
            优先处理强依赖
        定期清理
            删除不再活跃的 Relation
            避免网络过于庞大
        可视化工具
            提供 Relation 网络的可视化界面
            帮助理解和管理复杂的关系网络

### 4.4 LLM 调用成本问题
    问题
        频繁的 Bias 思考会导致大量 LLM 调用
        成本高昂

    解决方案
        缓存机制
            对于相同或相似的输入
            复用之前的结果
        Bias 优先级
            只触发高优先级的 Bias
            低优先级的 Bias 按需触发
        本地模型
            对于简单的任务
            使用本地小模型
            复杂任务才调用大模型

### 4.5 持久化性能问题
    问题
        频繁的持久化操作
        可能影响性能

    解决方案
        批量持久化
            积累一定数量的变更后
            批量写入
        异步持久化
            持久化操作在后台异步执行
            不阻塞主流程
        增量持久化
            只持久化变更的部分
            而不是整个对象

## 七、与现有系统的对比

### 7.1 vs LangChain/LlamaIndex
    LangChain/LlamaIndex
        优势
            成熟的工具链
            丰富的集成
        劣势
            静态的 Chain 结构
            缺乏自我演化能力

    OOC
        优势
            动态的对象生态
            具有自我演化能力
            更灵活的协作机制
        劣势
            需要从头构建
            生态尚不成熟

### 7.2 vs AutoGPT/BabyAGI
    AutoGPT/BabyAGI
        优势
            自主任务规划
            持续执行
        劣势
            缺乏结构化的思维方式
            难以管理复杂的协作

    OOC
        优势
            结构化的 Bias 思维
            清晰的 Relation 协作
            更好的可控性
        劣势
            需要更多的设计和配置

### 7.3 vs 传统 Agent 框架
    传统 Agent 框架
        优势
            简单直接
            易于理解
        劣势
            缺乏演化能力
            难以处理复杂场景

    OOC
        优势
            具有演化能力
            能处理复杂场景
            更接近人类的思维方式

            学习曲线较陡

## 九、关键设计决策的理由

### 9.1 为什么选择 Python 作为行动语言？
    优势
        语法简洁，易于 LLM 生成
        丰富的库生态
        动态特性支持元编程

    劣势
        性能相对较低
        类型安全性较弱

    权衡
        对于 Agent 系统
        灵活性比性能更重要
        Python 是最佳选择

### 9.2 为什么 Bias 是 field 而不是独立的 Object？
    如果 Bias 是独立的 Object
        优势
            更灵活，可以独立演化
            可以在多个 Object 之间共享
        劣势
            增加了系统复杂度
            Bias 和 Object 的边界不清晰

    如果 Bias 是 field
        优势
            Bias 和 Object 紧密绑定
            更符合"思维方式"的语义
        劣势
            Bias 的复用性较低

    权衡
        当前阶段选择 field
        未来可以考虑支持 Bias 的共享机制

### 9.3 为什么需要三层元类（Stone/Flow/Chat）？
    核心区分
        Stone vs Flow: 静态 vs 动态
            Stone 是数据和定义，不需要思考
            Flow 是过程和行动，具有完整的思考+执行能力
        Flow vs Chat: 通用 vs 对话特化
            Chat 不是"更高级的 Flow"
            Chat 是由 talk() 创建的 Flow，即来自外部对话的 Flow
            Chat 和 Flow 拥有相同的 Bias 集合和能力

    为什么不只有 Stone 和 Flow？
        Chat 作为独立元类的价值
            语义清晰：明确标识"这个 Flow 来自对话"
            生命周期不同：Chat 支持多轮对话，普通 Flow 任务完成即结束
            上下文管理不同：Chat 维护对话历史，普通 Flow 维护任务上下文
            跨 Stone 协作：Chat 是 Stone 之间对话的桥梁

    权衡
        三层设计合理
        Stone/Flow 的区分是本质性的（静态 vs 动态）
        Flow/Chat 的区分是场景性的（内部任务 vs 外部对话）

## 十、实现细节的思考

### 10.1 Bias 的 prompt 如何构建？
    基础结构
        Bias 的角色定义
            你是一个 [角色名称]
            你的职责是 [职责描述]

        可用的工具
            你可以访问以下 field: [field 列表]
            你可以调用以下 method: [method 列表]

        当前上下文
            当前任务: [task 描述]
            相关信息: [context 信息]

        输出要求
            请输出 Python 代码来完成任务
            或者输出你的思考结果

    动态调整
        根据 Bias 的历史表现
        调整 prompt 的细节

### 10.2 如何实现 Bias 的 when 条件？
    方案 1: 规则引擎
        定义规则语言
            when context.task_type == "code_generation"
            when context.has_error == True

        优势
            简单直接
        劣势
            表达能力有限

    方案 2: LLM 判断
        让 LLM 判断是否应该触发

        优势
            表达能力强
        劣势
            增加 LLM 调用成本

    方案 3: 混合方案
        简单条件用规则引擎
        复杂条件用 LLM

        优势
            平衡了性能和表达能力
        劣势
            实现复杂度较高

    推荐
        先实现方案 1
        未来扩展到方案 3

### 10.3 如何实现 Object 的持久化？
    文件结构
        每个 Object 一个目录
            code.py: 动态生成的代码
            data.json: 持久化的数据
            meta.json: 元信息（类型、创建时间等）

    持久化时机
        显式保存
            调用 self.save() 时
        自动保存
            Object 销毁时
            定期自动保存

    持久化内容
        field 的值
        Bias 的 prompt（如果被修改过）
        Relation 的信息
        历史记录（可选）

### 10.4 如何实现 Relation 的事件通知？
    事件类型
        on_create: 对象创建时
        on_update: 对象更新时
        on_delete: 对象删除时
        on_method_call: 方法调用时
        on_field_change: 字段变化时

    通知机制
        同步通知
            立即触发相关 Object 的 Bias
        异步通知
            将事件放入队列
            后台处理

    过滤机制
        不是所有事件都需要通知
        Relation 可以指定关注的事件类型

## 十五、Extension 设计

### 15.1 Extension 的本质
    Extension 是 Object
        Extension 不是外部插件，而是 OOC 世界中的一等公民 Object
        在 OOC 世界初始化时由 kernel 自动构造
        与普通 Object 的区别
            Extension 是 world 的直接子对象
            Extension 封装了与外部系统的交互能力
            Extension 通过自身的 method 向其他 Object 暴露外部能力
            Extension 创建的子对象（如 file、webpage）也是 OOC Object

    Extension 的元类
        Extension 继承自 Stone
            因为 Extension 本身不需要"思考"
            它是能力的提供者，不是决策者
            它的 method 是确定性的（打开文件、访问网页），不需要 LLM
        但 Extension 创建的对象可以是任意元类
            filesystem.create() 创建的 file 对象可以是 Stone（纯数据）
            也可以是 Chat（如果需要对文件内容进行智能分析）

### 15.2 Extension 的生命周期
    初始化阶段
        OOC 世界启动时
            kernel 读取 extension 配置
            按依赖顺序构造 Extension Object
            每个 Extension 注册到 world 的 extension registry
            Extension 执行自检（检查外部系统是否可用）

    运行阶段
        Extension 作为 world 的属性被其他 Object 访问
            world.filesystem  → filesystem Extension Object
            world.browser     → browser Extension Object
            world.terminal    → terminal Extension Object
        其他 Object 通过 Extension 的 public method 与外部系统交互

    销毁阶段
        OOC 世界关闭时
            Extension 清理所有创建的子对象
            释放外部资源（关闭文件句柄、关闭浏览器进程等）
            从 world 的 extension registry 注销

### 15.3 核心 Extension 设计

    filesystem Extension
        定位
            封装文件系统操作，将文件抽象为 OOC Object
        field
            root_path: 文件系统的根路径（受沙箱限制）
            created_files: 已创建的 file 对象列表
        method
            create(path, content="") → File Object
                在指定路径创建文件，返回 file 对象
                file 对象具有
                    field
                        path: 文件路径
                        content: 文件内容（懒加载）
                        metadata: 文件元信息（大小、修改时间等）
                    method
                        read() → str: 读取文件内容
                        write(content) → None: 写入文件内容
                        append(content) → None: 追加内容
                        delete() → None: 删除文件
            open(path) → File Object
                打开已存在的文件，返回 file 对象
            list(path) → list[File Object]
                列出目录下的文件
            exists(path) → bool
                检查文件是否存在

    browser Extension
        定位
            封装浏览器操作，将网页抽象为 OOC Object
        field
            max_pages: 最大同时打开的页面数
            opened_pages: 已打开的 webpage 对象列表
        method
            open(url) → Webpage Object
                打开指定 URL，返回 webpage 对象
                webpage 对象具有
                    field
                        url: 页面 URL
                        title: 页面标题
                        content: 页面文本内容（提取后的）
                        html: 原始 HTML（可选，按需加载）
                        links: 页面中的链接列表
                        status: 页面状态（loading/loaded/error）
                    method
                        refresh() → None: 刷新页面
                        extract_text() → str: 提取纯文本
                        extract_links() → list: 提取所有链接
                        find(selector) → str: CSS 选择器查找
                        screenshot() → bytes: 截图
                        close() → None: 关闭页面
            search(query) → list[Webpage Object]
                搜索引擎查询，返回结果页面列表

    terminal Extension
        定位
            封装终端/命令行操作
        field
            work_dir: 工作目录
            history: 命令执行历史
        method
            run(command) → Result Object
                执行命令，返回结果对象
                result 对象具有
                    field
                        stdout: 标准输出
                        stderr: 标准错误
                        exit_code: 退出码
                        duration: 执行耗时

### 15.4 Extension 创建的对象的特殊性
    与外部系统的绑定
        Extension 创建的对象（file、webpage 等）与外部资源绑定
        外部资源变化时，对象状态需要同步
            方案 1: 懒加载——每次访问时重新读取
            方案 2: 监听——注册文件系统/网络事件，主动更新
            方案 3: 混合——常用数据缓存，定期刷新

    生命周期与外部资源
        对象销毁时，需要决定是否同时销毁外部资源
            file 对象销毁 ≠ 删除文件（除非显式调用 delete）
            webpage 对象销毁 = 关闭浏览器标签页
        外部资源消失时，对象需要优雅处理
            文件被外部删除 → file 对象标记为 invalid
            网页无法访问 → webpage 对象标记为 error

    Relation 与 Extension 对象
        Extension 创建的对象自动与 Extension 建立 "created_by" relation
        其他 Object 可以与 Extension 对象建立 relation
            例如：researcher 对象与多个 webpage 对象建立 "references" relation

### 15.5 Extension 的扩展机制
    自定义 Extension
        用户可以编写自定义 Extension
        需要遵循 Extension 接口规范
            必须继承自 Stone
            必须实现 init() 和 cleanup() method
            创建的子对象必须是合法的 OOC Object

    Extension 之间的协作
        Extension 之间可以建立 relation
            例如：browser 下载文件时，可以委托 filesystem 创建 file 对象
        Extension 可以组合使用
            例如：terminal.run("curl ...") 的结果可以通过 filesystem.create() 保存

    Extension 的权限控制
        每个 Extension 有明确的权限边界
            filesystem: 只能访问 root_path 下的文件
            browser: 可以配置允许/禁止的域名
            terminal: 可以配置允许/禁止的命令
        权限由 world 在初始化时配置
        Extension 的 method 在执行前检查权限
