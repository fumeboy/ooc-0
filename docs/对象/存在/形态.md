# 形态 — Stone 与 Flow

> G2：对象有两种存在状态，如同物质的势能与动能。

## 核心区分

```
Stone（石头）                    Flow（流）
──────────                       ──────────
静态 / 潜能态                    动态 / 现实态
能力已定义，未激活               ThinkLoop 正在运行
"可以思考的东西"                 "正在思考的东西"
不会主动做任何事                 持续思考、行动、改变
```

**Stone 就像一块刻了字的石头**：信息在那里，但石头不会自己读出来。
**Flow 是 Stone 被一个具体任务唤醒后的活体**。

## 具体差异

| 维度 | Stone | Flow |
|---|---|---|
| 存储位置 | `stones/{name}/` | `flows/{sessionId}/objects/{name}/` |
| 标记文件 | `.stone` | `.flow` |
| 生命周期 | 跨越所有任务，永久存在 | 仅当前 Session 有效 |
| 思考能力 | 无（不调用 LLM） | 有（ThinkLoop 主动调用 LLM） |
| 可否被调用 | 是（作为工具/数据） | 是（通过 talk） |
| 可否自主行动 | 否 | 是 |
| 数量限制 | 一个 name 一个 Stone | 一个 Stone 可派生多个 Flow |

## 为什么是"一个 Stone 多个 Flow"

一个对象可能同时处理多个任务：

```
stones/alan/             ← 唯一的 Alan Stone
  ├── readme.md          ← 唯一的身份定义
  └── traits/

flows/sess_A/objects/alan/    ← Session A 中的 Alan Flow（处理任务 A）
flows/sess_B/objects/alan/    ← Session B 中的 Alan Flow（处理任务 B）
flows/sess_C/objects/alan/    ← Session C 中的 Alan Flow（处理任务 C）
```

三个 Flow **共享同一个 Stone 的身份和能力**，但各自有独立的运行时状态（线程树、inbox、当前思考）。这让对象可以并发处理多个任务，而不混淆各任务的上下文。

## Stone 和 Flow 不是"低级 vs 高级"

很多人第一直觉是 "Flow 比 Stone 更强大"。这是错的。

正确的对比是：**哪种形态适合当前场景**。
- 一个"文件系统操作器"对象 → Stone 就够了。它只需要被调用、返回结果，不需要自主思考
- 一个"研究员"对象 → 需要 Flow。它需要理解任务、主动探索、持续推进

一个对象**不需要思考就做到自己该做的**，就应该保持为 Stone。**只有需要自主推进任务时才派生 Flow**。

## 对象视角：Self / Session / SelfMeta

Stone 和 Flow 是系统内部术语。对**对象自身**而言，它感知到的是三个概念：

| 系统术语 | 对象视角 | 含义 |
|---|---|---|
| Stone 持久化目录 | **Self**（自我） | "我是谁"——跨任务的持久身份 |
| Flow 持久化目录 | **Session**（此刻） | "我现在在做什么"——当前任务的工作空间 |
| `_selfmeta` Flow | **SelfMeta** | "自我的管理者"——维护 Self 长期数据的常驻 Flow |

三者的权限关系：
- **Session 只能写自己的目录**。不能直接修改 Self（身份）
- **Self 只能通过 SelfMeta 修改**。对象向 SelfMeta 发消息 → SelfMeta 决定是否沉淀到 Self
- **SelfMeta 是同一个对象的常驻 Flow**，共享 Stone 身份，但拥有写 Self 目录的特权

这个设计的哲学意义：**沉淀不是机械的数据搬运，而是一次自我对话**。
详见 [../成长/反思机制/](../成长/反思机制/)。

## Flow 状态机

一个 Flow 在其生命周期中会经过以下状态：

```
running → waiting → pausing → finished / failed
```

- **running** — 正在执行 ThinkLoop 循环
- **waiting** — 等待外部输入（用户消息 / 子线程完成 / inbox 消息）
- **pausing** — 人机协作检查点（详见 `认知/context/pause.md`）
- **finished** — 任务完成
- **failed** — 任务失败
