# 能力 — 知道怎么做

> 沉淀的第二层。从具体知识抽象出可复用的模式。

## 含义

**能力** = 可复用的做事方式。"遇到 X 就用 Y 方法"——这是一个**规则**或**模板**。

## 载体：trait

能力的物理载体是 **trait**（G3）：

```
stones/alan/traits/my-debug-flow/
├── TRAIT.md            ← 这个能力是什么、怎么用
└── methods.ts          ← 具体方法实现（可选）
```

trait 的 readme 包含"能力说明"，methods 包含"可被调用的实现"。

## 能力的形成路径

### 路径 1：从知识手动提炼

对象（或 Sophia 设计师）发现某类知识反复出现，手动写成 trait：

```
观察：多次看到 alan 的知识里有"LLM 会在 try/catch 里吞错误"的记录
  ↓
提炼：这是一条"做事规则"
  ↓
写 trait：
  stones/alan/traits/llm-debug-hygiene/
  TRAIT.md 里说明何时该避免 try/catch、何时必须
```

### 路径 2：通过 reflect() 请求创建

```
对象运行过程中：
reflect("请沉淀为 trait：每次 LLM 调用失败后，必须保留原始错误栈")
  ↓
SelfMeta 审视：
  - 这是一个可复用的规则吗？（yes）
  - 是否与现有 trait 冲突？（no）
  - 创建新 trait 还是添加到现有？
  ↓
写入 trait（或扩展现有 trait 的 readme）
```

## 能力的激活：按需

与知识不同——**能力不是每轮都在 Context**。它只在需要时激活：

```
对象调用 open(type=command, command=X)
  → 如果有 trait command_binding X，激活该 trait
  → trait 的 readme 进入 Context
```

或：

```
对象调用 open(type=trait, name=my-ability)
  → 手动激活
```

详见 [../../结构/trait/渐进式激活.md](../../结构/trait/渐进式激活.md)。

## 能力 vs 直觉

```
能力：在需要时才出现
直觉：始终在 Context 里
```

区别：**激活策略**（trait 的 `when` 字段）：

- `when: never`（能力）— 按需激活
- `when: always`（直觉）— 始终激活

一个能力被反复证明"每次都重要"，它就应该从 `never` 升级为 `always`——成为直觉。

详见 [直觉.md](直觉.md)。

## 好能力 vs 坏能力

### 好能力的特征

- **明确触发条件**：何时适用？（用 command_binding 或显式描述）
- **具体可操作**：不是"要小心"而是"具体做 A、B、C"
- **最小边界**：不越界去影响其他场景

### 坏能力的反例

```markdown
# 不好的 trait
一般来说，遇到问题应该想清楚再做。
```

这太抽象，LLM 看了也不知道具体该做什么——实际上和没有一样。

```markdown
# 更好的 trait
遇到程序执行错误时：
1. 读完整 stack trace（不要只看第一行）
2. 定位到具体行号
3. 形成假设
4. 做最小改动验证
```

具体、可操作、有边界。

## 能力的验证

新沉淀的能力需要验证。验证不通过的能力应该回退：

```
沉淀新 trait → 使用若干次 → 监测效果
  ↓
效果好 → 保留（可能升级为直觉）
效果差 → 修改 readme 或移除 trait
```

验证通常通过**用例验证**（`工程管理/验证/用例/`）或**体验测试**（Bruce 测试）。

## 能力的组合

多个能力可以组合：

```
alan 激活了：
  - kernel/computable
  - kernel/talkable
  - alan-custom/llm-debug-hygiene
  - alan-custom/communication-style

→ 形成"alan"这个对象的独特"处理方式"
```

组合是 G3 的核心——能力通过叠加形成复合行为。
