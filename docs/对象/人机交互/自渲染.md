# Views — UI 与方法的统一表达（G11）

> 对象不是"被 UI 设计"的，它**自己画自己**。
> 2026-04-21 起，自渲染从 `ui/index.tsx` 升级为 `views/{viewName}/` 三件套 Views 机制，
> 让"UI"和"能力"共用同一套 Trait 基础设施。

## 核心概念：View 是 kind=view 的 Trait

View 本质上就是一个 `kind: view` 的 Trait。它与普通 trait 共享：

- 同一个 Loader（loader.loadTrait 支持 VIEW.md）
- 同一个 MethodRegistry（双通道 llm_methods / ui_methods）
- 同一套 namespace 和 traitId 规则（traitId = `self:{viewName}`）
- 可声明 command_binding（LLM 可以激活 view 的 readme）

独有的：

- `frontend.tsx` — Vite 动态 import 渲染的 React 组件（必须存在，loader 强制校验）
- `ui_methods` — 暴露给前端 HTTP 调用的方法（`POST /api/.../call_method`）

## 路径

### Stone 级

```
stones/{name}/views/{viewName}/
├── VIEW.md          ── 元数据（namespace=self, kind=view）
├── frontend.tsx     ── 默认导出 React 组件
└── backend.ts       ── 可选，ui_methods / llm_methods
```

跨 session 共享。默认视图通常叫 `main`（`views/main/`）。

### Flow 级

```
flows/{sid}/objects/{name}/views/{viewName}/
├── VIEW.md
├── frontend.tsx
└── backend.ts
```

Session 内产出的"报告"、"反馈表单"、"交互展示"等，通常放在 Flow 级（随 session 生命周期）。

Flow 级 views 优先级高于 Stone 级（同 viewName 时覆盖）。

## VIEW.md frontmatter

```yaml
---
namespace: self           # 必填，固定 self（views 只存在于对象内）
name: main                # view 名称（= viewName）
kind: view                # 可选（文件名是 VIEW.md 时自动置为 view）
type: how_to_interact     # 通常是交互类型
when: never               # 默认 never，或 always（与普通 trait 规则一致）
description: View 的一行描述
command_binding:          # 可选。声明该 view 在哪些 command 下被激活
  commands: ["return", "talk"]
deps: []
---

# View 的 readme（激活时注入 Context）

描述 view 的用途、何时产出、如何被用户看见。
```

## frontend.tsx 契约

```tsx
import React, { useState } from "react";

interface ViewProps {
  sessionId?: string;       // Flow 级上下文存在，Stone 级为空字符串
  objectName?: string;      // 对象名（view 的所有者）
  callMethod?: (traitId: string, method: string, args: object) => Promise<unknown>;
  // 其余 props 由 DynamicUI 透传
}

export default function MyView({ sessionId, objectName, callMethod }: ViewProps) {
  const [done, setDone] = useState(false);

  const submit = async () => {
    if (!callMethod) return;
    const result = await callMethod("self:my", "submit", { value: 42 });
    setDone(true);
  };

  return <button onClick={submit}>{done ? "已提交" : "提交"}</button>;
}
```

**callMethod** 由 DynamicUI 自动注入（仅当 props 含 sessionId + objectName）：
- 调用 `POST /api/flows/:sid/objects/:name/call_method`
- 白名单：只能调目标对象自己的 view（`self:` namespace + `kind=view`）的 `ui_methods`
- 方法执行支持 `ctx.notifyThread(message)` 向根线程 inbox 写消息 + 复活 done 线程

## backend.ts 契约

```typescript
import type { TraitMethod } from "@kernel/types";

/** UI 通道：前端通过 POST /call_method 调用 */
export const ui_methods: Record<string, TraitMethod> = {
  submit: {
    description: "用户提交表单",
    params: [
      { name: "value", type: "number", description: "", required: true },
    ],
    fn: async (ctx, { value }) => {
      ctx.setData("lastSubmission", value);
      ctx.notifyThread?.(`[UI] 用户提交 value=${value}`);
      return { ok: true, value };
    },
  },
};

/** LLM 通道：LLM 沙箱通过 callMethod(traitId, method, args) 调用 */
export const llm_methods: Record<string, TraitMethod> = {};
```

## 方法上下文（MethodContext）

两个通道的方法都接收 `MethodContext`：

| 字段 | 类型 | 说明 |
|---|---|---|
| data | `Record<string, unknown>` | Stone 数据快照 |
| getData(key) | fn | 读取 data |
| setData(key, value) | fn | 写入 stone.data 并持久化（HTTP 通道）|
| print(...args) | fn | 日志输出 |
| sessionId | string | 当前 session id |
| filesDir | string | `flows/{sid}/objects/{name}/files/` |
| rootDir | string | user repo 根 |
| selfDir | string | `stones/{name}/` |
| stoneName | string | 对象名 |
| notifyThread?(content, opts?) | fn | HTTP 通道独有：写根线程 inbox + 复活 done 线程 |

## 动态加载

前端 `DynamicUI` 组件：

```tsx
<DynamicUI
  importPath={`@stones/${name}/views/main/frontend.tsx`}
  componentProps={{ sessionId, objectName }}
/>
```

- 支持 `@stones/...` 和 `@flows/...` 两种别名路径
- Vite 动态 import（@vite-ignore 绕过静态分析）
- 自动注入 callMethod 闭包（当 componentProps 含 sessionId+objectName 且未显式传 callMethod）
- 渲染失败：通知对象（talkTo）+ 降级错误提示

## 失败降级

- 文件不存在（404）→ "信息待产出..."
- 文件加载/渲染错误 → 红色错误提示 + 自动通知对象修复
- Error Boundary 兜底防止白屏

## 有 views 的对象

前端：

- 对象有任一 view（`views/*/frontend.tsx`）→ 默认展示 View Tab
- 默认加载 `main` view，若无则取第一个（按字母序）
- 没有 views 的对象使用通用视图（ObjectDetail / FlowView 的其他 tabs）

## 典型用例

### 1. Supervisor Session Dashboard

`stones/supervisor/views/main/`：总览看板（替代旧 SessionKanban 的 UI Tab）。

### 2. 任务报告 + 反馈表单

`flows/{sid}/objects/supervisor/views/task-report-2026-04-21/`：
- frontend.tsx：展示任务结果 + 反馈表单
- backend.ts：ui_methods.submitFeedback → setData + notifyThread

### 3. 数据可视化

`stones/analytics/views/main/`：stone.data 读入，用 LineChart / BarChart 渲染。

## ooc:// 协议

views 通过 `ooc://view/{相对路径}` 引用：

- Stone 级：`ooc://view/stones/{name}/views/{viewName}/`
- Flow 级：`ooc://view/flows/{sid}/objects/{name}/views/{viewName}/`
- 尾部斜杠代表 view 目录本身（默认指向 frontend.tsx）

在 return/talk 消息中输出 `[navigate]` 卡片可引导用户打开 view：

```
[navigate title="反馈表单" description="请提交你对本次结果的反馈"]
ooc://view/flows/s_xyz/objects/supervisor/views/feedback-2026/
[/navigate]
```

## 组件安全性

与 2026-04-21 前一致：views 运行在前端主 JS 上下文中（无 iframe 沙箱）。
假设"对象的 view 代码可信"——来自 Session/知情用户/对象自身创作。
未来可能加入可选的 iframe 沙箱模式。

## 源码锚点

| 概念 | 实现 |

| ViewRegistry 分发 | `kernel/web/src/router/registrations.tsx` |

| ooc://view/ 解析 | `kernel/web/src/lib/ooc-url.ts` |

## 与基因的关联

- **G11**（UI 即面孔）— 本章核心
- **G3**（能力即自我立法）— view 是 kind=view 的 trait
- **G6**（关系即网络）— namespace 显式标记"能力来源"
- **G10**（Effect 是对外唯一通道）— HTTP call_method 是"用户 → 对象 data"的 Effect 入口
