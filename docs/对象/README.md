# 对象 — 本体层

> 这一层回答的问题是：**OOC 系统是什么**。
> 对象是 OOC 的唯一建模单元（G1）。整个本体层就是"对象"这一概念在 6 个维度上的展开。

## 六个维度

| 维度 | 问题 | 目录 | 主要基因 |
|---|---|---|---|
| **存在** | 对象如何"在"？ | [存在/](存在/) | G2, G7 |
| **结构** | 对象由什么组成？ | [结构/](结构/) | G1, G3, G6 |
| **认知** | 对象如何"知"？ | [认知/](认知/) | G4, G5, G13 |
| **合作** | 对象如何"做"与"管"？ | [合作/](合作/) | G8, G9, G10 |
| **成长** | 对象如何"变"？ | [成长/](成长/) | G12, G5 |
| **人机交互** | 对象如何"显"？ | [人机交互/](人机交互/) | G11 |

## 为什么是这六个维度

这不是任意的划分。它来自 `哲学/统一性.md` 的核心等式：

```
Object = 存在 ∩ 认知 ∩ 行动 ∩ 成长
```

- **存在** + **结构** 共同回答"对象是什么"（存在的位置 + 内部组成）
- **认知** 回答"对象如何独立思考"
- **合作** 回答"对象如何行动和与他者协作"（行动 + 看板的合并）
- **成长** 回答"对象如何改变自己"
- **人机交互** 回答"对象如何被人类看见"——这是统一性等式的外延

## 阅读路径

**按启动顺序**（对象生命周期视角）：存在 → 结构 → 认知 → 合作 → 成长 → 人机交互

**按抽象层次**（从底向上）：存在（物理）→ 结构（组件）→ 认知（思考）→ 合作（交互）→ 成长（演化）→ 人机交互（界面）

**按代码模块**（工程视角）：
- `kernel/src/persistence/` → 存在
- `kernel/src/stone/`, `kernel/src/trait/` → 结构
- `kernel/src/thread/` → 认知 + 成长
- `kernel/src/world/`, `kernel/src/kanban/` → 合作
- `kernel/web/` → 人机交互

## 文档写作约定

1. **每个子领域都有 README.md** 作为导航
2. **与源码的对应关系**写在各文档的 "源码锚点" 小节
3. **跨领域概念**（如 reflect 既在认知又在成长）采用"主归属 + 交叉引用"：主归属放在最本质的层，其他层用 `@ref` 引用
4. **实现细节与哲学理由分开**：文档主体讲"是什么 + 为什么"，实现细节单独成节或引用源码
