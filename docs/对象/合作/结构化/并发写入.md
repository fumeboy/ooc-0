# 并发写入 — per-session 串行化队列

> 同一个 Session 下可能有多方同时写看板文件。通过串行化队列保证一致性。

## 三个写入方

一个 Session 的 `issues/` 和 `tasks/` 文件可能被**三方**同时写：

| 写入方 | 通过什么 |
|---|---|
| **Supervisor** | `kernel/traits/session-kanban/` |
| **其他对象** | `kernel/traits/talkable/issue-discussion/` |
| **用户** | 后端 API（如 `POST /api/session/{sid}/issues/{id}/comments`）|

如果没有保护，可能发生：

```
A 读 issue-001.json
B 读 issue-001.json
A 在 comments 中加一条
A 写回
B 在 comments 中加一条（基于之前读的版本）
B 写回  ← A 的 comment 丢失
```

## serializedWrite 机制

Session 级别维护一个**写入队列**，保证对同一文件的读-改-写原子化：

```typescript
// kernel/src/world/session.ts
class Session {
  private writeQueues = new Map<string, Promise<void>>();

  async serializedWrite(filePath: string, fn: () => Promise<void>): Promise<void> {
    const prev = this.writeQueues.get(filePath) ?? Promise.resolve();
    const next = prev.then(async () => {
      await fn();
    });
    this.writeQueues.set(filePath, next);
    await next;
  }
}
```

### 使用

```typescript
await session.serializedWrite(`issues/issue-${id}.json`, async () => {
  const data = await readFile(...);           // 读
  data.comments.push(newComment);              // 改
  await writeFile(..., JSON.stringify(data));  // 写
});
```

这三步在队列的**一次回合**中原子化——其他并发写入会排队等待。

## per-session 粒度

每个 Session 有自己的 writeQueues。跨 Session 的写入互不影响——这是合理的，看板是 Session 级的，不同 Session 没有共享文件。

## 性能考虑

串行化看起来"慢"，但实际：

- 同一文件的并发写**必须**串行化（数据正确性 > 速度）
- 不同文件的并发写**不**相互等待（writeQueues 按 filePath 分开）
- 看板写入频率通常不高（一分钟几十次），串行完全够用

## 用户 API 的并发保护

后端 API 也走 serializedWrite：

```typescript
app.post("/api/session/:sid/issues/:id/comments", async (req, res) => {
  await session.serializedWrite(`issues/issue-${req.params.id}.json`, async () => {
    // 读-改-写
  });
  res.json({ ok: true });
});
```

保证即使用户点快速连按，也不会丢评论。

## index.json 的同步更新

修改 issue-{id}.json 时，通常也要更新 index.json（如 updatedAt 字段）。两者是**不同文件**，但应该一起更新：

```typescript
await session.serializedWrite(`issues/index.json`, async () => {
  // 先读 index.json
  const index = await readFile(...);
  const entry = index.find(e => e.id === id);
  entry.updatedAt = now();
  await writeFile(..., index);
});

await session.serializedWrite(`issues/issue-${id}.json`, async () => {
  // ...
});
```

两个队列各自串行化，但两者之间的顺序**无保证**。如果需要跨文件原子性，要在单一 serializedWrite 里操作（用双写或事务语义）。实践上看板数据不要求这种强一致性。

## 失败处理

如果 serializedWrite 的 fn 抛错：

- 错误向上抛（调用方处理）
- 队列继续（下一个写入正常执行）
- 文件保持抛错前的状态（如果抛错在读后、写前，文件未改；如果抛错在写途中，可能半写——用原子写入函数避免）

## 原子写入

writeFile 本身是原子的（通过 tmpfile + rename）：

```typescript
async function atomicWrite(path, content) {
  const tmpPath = path + ".tmp";
  await fs.writeFile(tmpPath, content);
  await fs.rename(tmpPath, path);
}
```

保证不会出现"半个文件"——要么旧内容，要么新内容。

## 与基因的关联

- **G10**（行动记录不可变）— 并发保护保证 append-only 语义
- **G7**（目录即存在）— 文件是对象的物理存在，写入一致性不能妥协
