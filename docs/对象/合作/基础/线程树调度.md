# 线程树调度 — 完整机制

> ThreadScheduler 管理线程执行顺序、等待、死锁、跨对象协作。
> 本文档是合作维度的**写侧**：从"多 Flow 协同"角度描述调度。
> 认知侧的简要版本见 [../../认知/线程树/调度.md](../../认知/线程树/调度.md)。

## 调度模型：单线程循环

每个对象的 Flow 有自己的调度循环：

```
while (flow 活跃) {
  tick:
    for each running thread in flow:
      执行一轮 ThinkLoop（engine.runThreadIteration）
    检查 waiting 线程唤醒
    检查 done 线程复活（有新 inbox 消息时）
    检查死锁
  sleep(tick interval)
}
```

## 跨对象协作

一个 Session 可能有多个 Flow（多对象）。调度器如何协调？

### 每个对象一个调度器

```
Session
  ├── Flow (A) → ScheduleLoop A
  ├── Flow (B) → ScheduleLoop B
  └── Flow (C) → ScheduleLoop C
```

每个 Flow 独立跑 tick。并发执行，不互相阻塞。

### 跨对象消息的投递

A 的线程 `talk(B, message)`：

```
A.thread.actions push tool_use
  → 消息写入 flows/{sid}/objects/B/threads/{tid}/thread.json 的 inbox
  → 如果 B 的目标 thread 在 waiting，标记可唤醒
  → B 的 ScheduleLoop 下一 tick 时检测到，唤醒 thread
  → B 开始处理消息
```

整个过程是**异步**的——A 不等 B。如果 A 需要等，用 talk_sync（本质是 talk + wait）。

详见 [../消息/跨对象协作.md](../消息/跨对象协作.md)。

## 自动创建 Flow

A talk 给 B，但 B 的 Flow 在当前 Session 不存在？

**自动创建**：

```
A.thread talk("B", ...)
  → collaboration.ts 检查 B 的 Flow
  → 不存在 → 创建 flows/{sid}/objects/B/，初始化 threads.json
  → 创建 B 的根线程，将消息放入其 inbox
  → B 的 ScheduleLoop 开始运行
```

这让对象之间**天然可协作**——只要知道名字，就能 talk。

## 等待机制

线程 wait 后，ThreadScheduler 在每 tick 检查唤醒条件：

### 等待子线程

```typescript
if (thread.status === "waiting" && thread.waitFor) {
  const children = thread.waitFor.map(id => threads.get(id));
  if (children.every(c => c.status === "done" || c.status === "failed")) {
    thread.status = "running";  // 唤醒
  }
}
```

### 等待 inbox 消息

```typescript
if (thread.status === "waiting" && !thread.waitFor) {
  if (thread.inbox.some(m => !m.marked)) {
    thread.status = "running";
  }
}
```

## 死锁检测

如果**所有** running 线程都在 waiting 且没有外部事件能改变（例如所有等待的子线程也 waiting）：

```
A.thread waiting for B.thread
B.thread waiting for A.thread
  → 死锁
```

ThreadScheduler 的检测逻辑：

```typescript
function detectDeadlock(flow) {
  const running = flow.threads.filter(t => t.status === "running");
  const waiting = flow.threads.filter(t => t.status === "waiting");

  if (running.length === 0 && waiting.length > 0) {
    // 没有 running，只有 waiting
    // 检查是否所有 waiting 的条件都不可能满足
    if (noEventCanWakeAny(waiting)) {
      return true;
    }
  }
  return false;
}
```

死锁时，**强制唤醒**最久 waiting 的线程，并注入 `deadlock_notice`：

```
[deadlock notice]
检测到死锁。你和其他线程互相等待，没有新事件能打破。
请考虑：
1. 放弃某个等待（close 对应 form）
2. 主动向某个对象发消息打破僵局
3. return 当前线程（如果能继续的话）
```

## Session 终止条件

### 根线程 done / failed

**对象级**：A 对象的根线程 done → A 的 Flow 结束（.flow 标记删除）。

**Session 级**：Supervisor 判断"整体任务完成"→ Session 终止，所有 Flow 清理。

### 用户强制终止

用户通过 UI / API 终止 Session → 所有 Flow 停止 tick，标记清理。

### 超时（可选）

Session 可以设置超时时间，到期自动终止（防止僵尸 Flow）。

## 调度的公平性

- **线程级**：同一 Flow 内多线程，按 `lastExecutedAt` 排序，最久未执行优先
- **Flow 级**：多个 Flow 并发 tick，互相不阻塞（各自独立循环）
- **优先级**：当前无显式 priority 字段，都走 FIFO 公平调度

## SSE 事件

调度过程中产生的事件通过 SSE 推送：

| 事件 | 触发 |
|---|---|
| `flow:start` | Flow 被创建 |
| `thread:start` | 线程创建 |
| `thread:revived` | done → running |
| `thread:returned` | running → done |
| `thread:failed` | any → failed |
| `stream:action` | 每个 action 写入时 |
| `stream:thought` | thinking 流式推送 |
