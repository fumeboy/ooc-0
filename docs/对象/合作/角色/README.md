# 角色 — 合作中的特殊主体

> 大部分对象地位平等：都有 Stone、都能 talk、都能创建子线程。
> 但有些对象因为**合作中的位置**而拥有特殊身份——最典型的是 **Supervisor**。

## 目前一个文档

- [supervisor.md](supervisor.md) — 全局代理，Session 总协调者

## 为什么需要"特殊角色"

如果完全平等，谁接收用户的第一条消息？谁管理 Session 看板？谁判断任务整体完成？

**答**：需要一个"默认路由 + 总协调"的角色。这就是 Supervisor。

## 设计原则

**特殊角色不是特殊对象**——它们仍然是 Stone、仍然有 readme、仍然通过 trait 定义能力。只是：

- 系统的某些默认路径指向它们
- 它们激活了一些"专属 trait"（如 session-kanban）
- 它们的 readme 通常更详细（因为承担了更多职责）

## 为什么不多做特殊角色

以 OOC 的理念，应该**尽量少**有特殊角色——因为每增加一个特殊性都是对"数据即对象"的让步。

当前只有 Supervisor 是强特殊。其他对象（filesystem、library 等）虽然常用，但都是**普通对象**——没有系统级特权，只是用得多。
