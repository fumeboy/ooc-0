# 社交网络 — 从局部关系到全局图

> 所有对象的关系汇聚成一张有向图。但没有人（包括 World）能全知地看到它。

## 从局部到全局

每个对象的 relations 是局部的：

```
alan.relations   → [bruce, filesystem, browser]
bruce.relations  → [alan, filesystem]
supervisor.relations → [alan, bruce, world]
...
```

**扫描所有对象的 readme.md**，就能重建全局关系图：

```
supervisor
    ├── alan
    │   ├── bruce
    │   ├── filesystem
    │   └── browser
    └── bruce
        ├── alan
        └── filesystem
```

但——**谁来做这个扫描**？答案是：**系统级工具**（registry.ts），而**不是**某个对象。对象只能看到自己的局部视图。

## World 不是全知节点

World 是所有对象的根容器，但它也是一个对象（有 readme，有 relations）：

```yaml
# world 的 readme.md
relations:
  - name: supervisor
  - name: library
  - name: alan
```

World 的 relations 只有它直接持有的对象。

**为什么这样设计？**
因为 OOC 的核心原则之一是"**没有全知节点**"。如果 World 能看到所有对象的 relations，它就成了一个神——这违反 G6。

## 发现新对象的路径

如果对象 A 想和对象 C 沟通，但 A.relations 里没有 C，怎么办？

### 方式 1：通过共同邻居

```
A.relations has B
B.relations has C
A 先 talk(B, "帮我联系 C")
→ B 把消息转发给 C
```

这让"社交网络"真正像社交——通过中介介绍认识。

### 方式 2：通过 World

World 作为根容器，通常 relations 里有很多"系统级对象"（supervisor、library 等）。任何对象的 relations 里通常都有 supervisor 或 world：

```
A → supervisor (supervisor.relations has C)
A 先 talk(supervisor, "我需要和 C 沟通")
→ supervisor 介绍 C
```

### 方式 3：通过名字直接 talk

如果对象 A 通过某种方式（如看到 C 的名字出现在历史 actions 中）知道 C 的存在，可以：

```
talk("C", "...")
```

系统会根据 name 查找 C 的 Stone（跨 Session 可解析），建立临时连接。**注意**：这不会自动把 C 加入 A.relations——A 需要主动"认识"C 并通过 SelfMeta 更新关系。

## 前端展示：关系图

前端可以展示对象的社交图谱（通过扫描 readme.md）：

```
左侧栏 → 对象列表（World 的 relations）
对象详情页 → 本对象的 relations 列表 + ObjectAvatar 头像
命令面板 → 按名字搜索所有对象
```

详见 [../../人机交互/](../../人机交互/)。

## 关系的三种形态

| 形态 | 含义 | 是否固定 |
|---|---|---|
| **直接关系** | 在 readme.relations 里声明 | 持久 |
| **间接关系** | 经由中介可达 | 动态（取决于中介的 relations） |
| **动态关系** | 通过 name 直接 talk（无 relation） | 临时（仅本次对话有效） |

## 局部视图的优势

1. **加对象零成本** — 不用更新全局索引
2. **对象可移植** — 一个对象的 relations 自带，迁移到新项目时不依赖别人
3. **松耦合** — 对象之间不互相知道"谁引用了我"

## 局部视图的代价

1. **发现新对象需要协作** — 不能"直接查全图"
2. **孤立关系不自动清理** — 被引用方消失了，引用方不知道

但这些代价是 G6 的必然推论——选择了局部知识，就要接受"发现是协作的"这一点。
