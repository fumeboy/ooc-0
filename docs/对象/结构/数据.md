# 数据 — 对象的工作记忆

> 数据是动态键值对。对象的**工作记忆**——随任务变化，不定义对象"是什么"，而是"当前在处理什么"。

## 两种粒度

| 粒度 | 位置 | 生命周期 | 例子 |
|---|---|---|---|
| **Stone 级** | `stones/{name}/data.json` | 跨任务持久 | `preferred_model`, `last_reviewed_at` |
| **Flow 级** | `flows/{sid}/objects/{name}/data.json` | 当前 Session | `current_step`, `local_cache` |

两者格式相同，都是 JSON 键值对：

```json
{
  "current_project": "thread-tree 重构",
  "progress": 0.75,
  "notes": ["2026-04-20 修复 inbox 竞态", "2026-04-21 完成集成测试"]
}
```

## 数据不是身份

OOC 刻意把身份（readme.md）和数据（data.json）分开：

- **身份** = "我是谁"——相对稳定、描述性
- **数据** = "我当前在处理什么"——快速变化、状态性

理由：**身份改变是哲学事件**（需要反思、沉淀），**数据改变是日常操作**（可以自由修改）。如果都混在一起，就无法区分"临时状态"和"长期承诺"。

## 写入 API

对象通过 trait 暴露的方法修改 data.json：

```typescript
// 通过 computable trait 的 setData 指令
setData(key: string, value: any)
getData(key: string): any
```

**Stone 级 data**：只有 SelfMeta 能写。
**Flow 级 data**：当前 Flow 的 ThinkLoop 可以直接写。

## 典型用法

### 1. 任务进度跟踪

```json
{
  "current_task": "thread-tree-phase-5",
  "steps_done": 3,
  "steps_total": 5,
  "blockers": ["等 Sophia 确认 G13 更新"]
}
```

### 2. 缓存与中间结果

```json
{
  "search_results_cache": { "AI safety": [...] },
  "last_calc": { "sum": 42, "ts": "..." }
}
```

### 3. 配置与偏好

```json
{
  "preferred_model": "glm-5.1",
  "thinking_budget": 8000,
  "language": "zh-CN"
}
```

## 数据的"幻觉陷阱"

data.json 是 LLM 读写的——存在 **"对象自己骗自己"** 的风险：

- LLM 可能写入"任务已完成: true"，但实际没完成
- LLM 可能假装某个中间结果已缓存，实际没存

这是 G13 提到的 hallucination 风险。缓解手段：
- 关键状态用 trait 方法（有代码验证）而非直接 setData
- 可验证的状态放 memory.md 或 actions（有历史痕迹）
- 对 data.json 的写入也记录到 actions（可追溯）

详见 `认知/thinkloop/engine.md` 中 "三层幻觉防线" 节。

## 数据与 Context 的关系

data.json 并**不自动**进入 Context。它需要对象显式 `getData(key)` 后，结果才进入当前轮次的 actions，再通过 actions 注入 Context。

这是刻意的：Context 容量有限，不是所有数据都值得每轮注入。让对象自己决定"现在需要什么数据"。
