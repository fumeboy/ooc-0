# 方法注册 — MethodRegistry 与沙箱调用

> Trait 暴露给 LLM/前端的不仅有知识（TRAIT.md），还有**方法**（可调用的函数）。
> 2026-04-21 起，方法分 LLM/UI 两个严格隔离的通道。

## 方法的两种通道

每个 trait 可在 `index.ts`（或 view 的 `backend.ts`）导出两张方法表：

```typescript
import type { TraitMethod } from "@kernel/types";

/** LLM 通道：LLM 沙箱可调用 */
export const llm_methods: Record<string, TraitMethod> = {
  readFile: {
    description: "读取文件内容",
    params: [
      { name: "path", type: "string", description: "文件路径", required: true },
    ],
    fn: async (ctx, { path }) => {
      return /* ... */;
    },
  },
};

/** UI 通道：前端 HTTP 可调用 */
export const ui_methods: Record<string, TraitMethod> = {
  submit: {
    description: "用户提交表单",
    params: [{ name: "value", type: "number", description: "", required: true }],
    fn: async (ctx, { value }) => {
      ctx.setData("submitted", value);
      ctx.notifyThread?.(`[UI] 用户提交 value=${value}`);
      return { ok: true };
    },
  },
};
```

**所有方法的参数都是对象**（`fn(ctx, argsObj)`，非位置参数）。

## 双通道严格隔离

| 通道 | 注册来源 | 可调用入口 | 不可调用 |
|---|---|---|---|
| llm | `llm_methods` | 沙箱 `callMethod(traitId, method, args)` | `ui_methods`、其他对象的 methods |
| ui | `ui_methods` | HTTP `POST /api/flows/:sid/objects/:name/call_method` | `llm_methods`、kernel/library traits、其他对象的 methods |

HTTP UI 通道额外白名单：
- traitId 必须 `self:` namespace（kernel/library 不可被前端调用）
- 目标 trait 必须 `kind: view`（普通 self trait 也不可被前端调用）

## MethodRegistry（三元键）

所有方法在 trait 加载时注册到 MethodRegistry，key 是 **(traitId, methodName, channel)** 三元组：

```typescript
type Channel = "llm" | "ui";

class MethodRegistry {
  register(traitId: string, methodName: string, def: TraitMethod, channel: Channel): void;
  get(traitId: string, methodName: string, channel: Channel): RegisteredMethod | undefined;
  getUiMethod(traitId, methodName): RegisteredMethod | undefined; // channel=ui 的快捷
  registerAll(traits: TraitDefinition[]): void;
  buildSandboxMethods(ctx: MethodContext, _stoneName): { callMethod };
}
```

## 沙箱调用协议（LLM 通道）

沙箱只暴露 **callMethod** 单一函数：

```typescript
// sandbox API（在 program 代码中）
await callMethod(traitId, methodName, args);
```

例：

```typescript
// 完整 traitId
await callMethod("kernel:computable", "readFile", { path: "foo.md" });

// 省略 namespace（按 self → kernel → library 顺序查找）
await callMethod("computable", "readFile", { path: "foo.md" });

// self view 的 LLM 方法（若 view 也声明了 llm_methods）
await callMethod("self:report", "parseReport", { path: "report.md" });
```

**取消了以下旧调用方式**（硬迁移，无兼容）：
- 扁平命名（`readFile(...)`）
- 两段式命名（`computable.readFile(...)`）
- 位置参数调用

## HTTP 调用协议（UI 通道）

前端通过 api/client 的 `callMethod`：

```typescript
import { callMethod } from "@ooc/api/client";

await callMethod(sessionId, objectName, "self:main", "submit", { value: 42 });
```

后端实现：`POST /api/flows/:sid/objects/:name/call_method`
请求体 `{ traitId, method, args }`，响应 `{ success: true, data: { result } }`。

DynamicUI 组件会自动注入 `callMethod` 闭包到 View 的 React 组件 props（当 componentProps 含 sessionId+objectName）。

## MethodContext（注入到 fn 的第一个参数）

```typescript
interface MethodContext {
  readonly data: Record<string, unknown>;        // stone 数据快照
  getData(key: string): unknown;
  setData(key: string, value: unknown): void;    // 写入 stone.data（HTTP 通道会持久化）
  print(...args: unknown[]): void;
  readonly sessionId: string;
  readonly filesDir: string;                     // flows/{sid}/objects/{name}/files/
  readonly rootDir: string;                      // user repo 根
  readonly selfDir: string;                      // stones/{name}/
  readonly stoneName: string;

  // HTTP 通道独有（LLM 沙箱中通常 undefined）
  notifyThread?(content: string, opts?: { from?: string }): void;
}
```

### notifyThread 语义

UI 方法通过 `ctx.notifyThread(msg)` 向对象的**根线程 inbox** 写一条 system 消息：

- 写入后若根线程状态为 `done`，自动复活为 `running`（ThreadsTree.writeInbox 内置机制）
- 复活时服务器非阻塞触发 `world.resumeFlow(objectName, sessionId)` 启动 scheduler 迭代
- 是否调用 notifyThread 完全由方法内部决定（UI 改 data 可以不通知，也可以通知）

这是"用户 → 对象 data 改变 + 对象被唤醒继续思考"的完整回路。

## 方法与指令的区别

| 维度 | 指令（command） | 方法（method） |
|---|---|---|
| 触发 | 通过 `open/submit` tool call | sandbox 中 `await callMethod(...)` 或 HTTP `/call_method` |
| 粒度 | 粗粒度（program、talk、return） | 细粒度（readFile、createIssue、setData 等） |
| 层级 | LLM 直接交互 | 在 program 执行过程中调用 / 前端交互 |
| 激活 | `open(type=command)` 驱动 command_binding | 方法始终注册，按 activatedTraits 过滤 llm 通道可见性 |

## 安全约束

### 沙箱隔离（LLM 通道）

- 沙箱变量：`self_dir`、`world_dir`、`filesDir` 等只读路径
- 基础 API：`readFile`/`writeFile`/`print`/`getData`/`setData` 等原生注入（非 trait 方法）
- trait 方法只能通过 `callMethod` 调用（无扁平命名）

### 白名单（UI 通道）

HTTP 端点强制白名单：
- `self:` namespace only
- `kind: view` only
- `ui_methods` only
- view 必须属于 URL 参数指向的对象

### 作用域

LLM 沙箱中，方法的可见性按**当前线程激活的 traits** 过滤（computeScopeChain）。
HTTP 端点不受激活状态影响（view 的 ui_methods 始终可调，这是 view 的契约）。

## 跨对象方法调用（可选扩展）

通过 `talkable/cross_object` trait，LLM 可以调用其他对象的 public methods：

```typescript
// 在沙箱中
const result = await talk_sync("B", { method: "someMethod", args: {...} });
```

本质是一次 talk_sync（线程树 await 机制），对方收到后解析 method + args，执行，返回结果。

详见 `kernel/traits/talkable/cross_object/TRAIT.md`。
