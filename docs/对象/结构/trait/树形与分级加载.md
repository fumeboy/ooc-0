# 树形结构与 Progressive Disclosure

> trait 支持任意深度的树形嵌套。三层加载策略让 Context 只注入"当前必需"的内容，避免巨大的系统提示词。

## 树形嵌套

一个 trait 可以有子 trait：

```
kernel/computable                            ← 父
├── kernel/computable/program_api            ← 子（完整 API 参考）
├── kernel/computable/file_ops               ← 子（文件操作详说）
├── kernel/computable/file_search            ← 子（glob/grep 详说）
├── kernel/computable/shell_exec             ← 子（exec/sh 详说）
├── kernel/computable/web_search             ← 子（互联网搜索）
└── kernel/computable/testable               ← 子（测试能力）
```

父 trait 的 TRAIT.md 是**概览性**的：讲清楚"这个能力是什么、主要 API 签名"。
子 trait 的 TRAIT.md 是**细节性**的：展开具体 API 的参数、示例、边界条件。

## Progressive Disclosure — 三层加载

```
Level 1 ── 精简注入
  父 trait 的精简 TRAIT.md
  always-on trait 采用此级别

Level 2 ── 子 trait 描述可见
  当父 trait 被激活时，其子 trait 的一行 description 对 LLM 可见
  LLM 知道"有哪些进一步细节可以加载"

Level 3 ── 按需激活
  open(type=trait) 或 command_binding 触发
  子 trait 的完整 TRAIT.md 注入 Context
```

## 举例：file_ops 的加载流程

**场景**：对象想写一个文件。

1. **初始状态**
   - `kernel/base` 已加载（Level 1）：对象知道有 open/submit/close
   - 其他 trait 未加载

2. **开启 program 指令**
   ```
   open(type=command, command=program)
   ```
   - `kernel/computable` 的 `command_binding: [program]` 触发
   - `kernel/computable` 被激活（Level 1 注入）
   - 同时 `kernel/computable` 的子 trait 描述可见（Level 2）：
     ```
     子能力：
     - file_ops: 文件读写、目录列表、原子写入
     - file_search: glob/grep 搜索
     - shell_exec: 执行 shell 命令
     - web_search: 搜索互联网
     - testable: 运行测试
     ```

3. **LLM 决定要写文件**
   - 看到 file_ops 描述，决定激活完整内容
   - 调用 `open(type=trait, name=kernel/computable/file_ops)`
   - file_ops 的完整 TRAIT.md 注入 Context（Level 3）
   - 现在 LLM 看到详细的 writeFile / appendFile API

4. **LLM 写完文件**
   - `submit` 执行完成
   - `close(type=trait, name=kernel/computable/file_ops)`（或由 FormManager 自动 close）
   - file_ops 被卸载（Context 释放空间）

5. **close(type=command, command=program)**
   - `kernel/computable` 被卸载
   - 子 trait 描述也一并消失

## 为什么要这样设计

### 问题：单一 system prompt 太长

传统 Agent 把所有工具说明、所有能力描述、所有格式约定都写在一个 system prompt 里。结果是：

- 每次调用 LLM 都发送几万 token
- 很多内容与当前任务无关
- LLM 注意力被稀释

### 解法：按需加载

OOC 把"能力说明"拆成树形，只在需要时加载叶子。一般一次思考只有 2-5 个 trait 处于 Level 3 激活状态。

### 约束：层级不能太深

实践经验：**不超过 3 层**（父 / 子 / 孙）。再深就会让对象在"该激活哪个子孙 trait"上花太多时间。

## 实现机制

### 加载时建树

`kernel/src/trait/loader.ts` 遍历 trait 目录时：

```
foreach directory containing TRAIT.md:
  path = directory path
  trait.name = path relative to traits root
  if parent directory also has TRAIT.md:
    trait.parent = parent trait name
    parent.children.push(trait.name)
```

### Context 构建时分级

`kernel/src/thread/context-builder.ts`：

```
for each trait in activeTraits:
  if trait is active (Level 3):
    inject trait.readme           # 完整内容
  elif trait.parent is active (Level 2):
    inject "- trait.name: trait.description"  # 一行描述
```

### 激活/卸载

`kernel/src/thread/tree.ts`：
- `activateTrait(name)` — refcount++，首次激活时记入线程的 activatedTraits
- `deactivateTrait(name)` — refcount--，归零时从 activatedTraits 移除

同一 trait 可被多个 form 持有（refcount），保证 close 不会过早卸载正在使用的 trait。

## 源码锚点

| 概念 | 实现 |
|---|---|
| 加载建树 | `kernel/src/trait/loader.ts` |
| 分级注入 | `kernel/src/thread/context-builder.ts` |
| 激活 refcount | `kernel/src/thread/tree.ts` |

## 与基因的关联

- **G3**（trait 是自我定义）— 树形结构支持能力的层次化组织
- **G5**（Context 即世界）— 分级加载是 Context 容量管理的核心手段
- **G13**（线程树即运行模型）— trait 激活的作用域由线程树 scope chain 决定
