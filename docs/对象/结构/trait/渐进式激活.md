# 渐进式激活 — activates_on.paths 驱动的按需加载

> Trait 不是"全部始终加载"。大部分 trait 在需要时才激活，完成后即卸载，最小化 Context 占用。

## 核心机制：activates_on.paths（原 command_binding，2026-04-26 更名）

每个 trait 可以声明它关联的**路径**：

```yaml
# kernel/computable/TRAIT.md
---
name: computable
activates_on:
  paths: [program]
when: never
---
```

意思是：当对象 `open(type=command, command=program)` 时，`kernel/computable` **自动激活**。

## 激活链路

```
open(type=command, command=program)
         ↓
   FormManager.begin(formId, command=program)
         ↓
   collectCommandTraits("program")
      → 遍历所有 trait
      → 找到 activates_on.paths 包含 "program" 的
      → 返回 [kernel/computable]
         ↓
   activateTrait("kernel/computable")
         ↓
   线程树 activatedTraits 更新
         ↓
   下一轮 Context 构建时，kernel/computable.readme 注入
```

## 卸载链路

```
submit(formId, args)           或           close(formId)
         ↓                                        ↓
   FormManager.submit()                    FormManager.cancel()
         ↓                                        ↓
   form 完成 / 取消
         ↓
   for each trait activated by this form:
     deactivateTrait(trait_name)
         ↓
   refcount--
         ↓
   if refcount == 0:
     从 activatedTraits 移除
```

## Refcount 机制

多个 form 可能激活同一个 trait。例如两个并发 program 指令：

```
form-1: open(command=program) → activateTrait(computable), refcount=1
form-2: open(command=program) → activateTrait(computable), refcount=2
form-1: submit  → deactivate, refcount=1  （computable 还在）
form-2: submit  → deactivate, refcount=0  （computable 被卸载）
```

这避免"刚激活就因其他 form 结束被卸载"的竞态。

## 手动激活：open(type=trait)

除了 activates_on.paths 驱动，LLM 也可以**主动**激活 trait：

```
open(type=trait, name="kernel/computable/file_ops")
```

这会激活 `file_ops` 子 trait（进入 Level 3），让完整 API 文档注入 Context。

close 时同样通过 FormManager 管理。

## 多种激活方式

| 方式 | 触发 | 典型用法 |
|---|---|---|
| **always 激活** | 对象 readme 中 `traits: [...]` 且 trait.when=always | 基座能力 |
| **command 驱动** | `open(type=command, command=X)` 触发 activates_on.paths | 能力层主要入口 |
| **直接激活** | `open(type=trait, name=...)` | 激活子 trait / 单独激活某能力 |
| **skill 激活** | `open(type=skill, name=...)` | 激活 skill（非 trait，但机制类似） |

## 三层加载的配合

渐进式激活 + 分级加载（[树形与分级加载.md](树形与分级加载.md)）配合：

```
Time 0:
  kernel/base 已 always 激活 (Level 1)
  Context 中有：base 的简短说明

Time 1: open(command=program)
  kernel/computable 激活 (Level 1)
  其子 trait 描述可见 (Level 2)
  Context 中新增：computable 概览 + 子 trait 列表

Time 2: open(type=trait, name=kernel/computable/file_ops)
  file_ops 激活 (Level 3)
  Context 中新增：file_ops 完整文档

Time 3: submit 完成 file_ops 相关操作
  close(type=trait, name=kernel/computable/file_ops)
  file_ops 被卸载
  Context 收缩：file_ops 完整文档消失，其他不变

Time 4: 另一个 open(type=trait, name=kernel/computable/shell_exec)
  shell_exec 激活 (Level 3)
  Context 中新增：shell_exec 完整文档
```

## 关键设计选择：为什么不预加载全部

### 选择 A：预加载全部能力
- 优点：LLM 一眼看到所有可能性
- 缺点：Context 巨大（可能数万 token），注意力被稀释

### 选择 B：全部按需加载（当前选择）
- 优点：Context 精简，LLM 专注于当前任务相关内容
- 缺点：LLM 需要记住"有哪些能力可以激活"——通过基座的 open 机制 + Level 2 的子 trait 描述解决

## FormManager 的作用

FormManager 是**激活生命周期的守护者**：

- 跟踪当前线程有哪些 form 处于 active 状态
- 每个 form 关联了哪些 trait
- form 结束（submit 或 cancel）时自动调用 deactivateTrait
- 防止内存泄漏（trait 永不卸载）

详见 [../../认知/指令系统/form-manager.md](../../认知/指令系统/form-manager.md)。

## 与基因的关联

- **G3**（trait 是自我定义）— 按需激活让自我定义动态化
- **G5**（Context 即世界）— 按需加载是 Context 容量管理的核心
- **G13**（线程树即运行模型）— trait 激活范围 = 线程的作用域链
