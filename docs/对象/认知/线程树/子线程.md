# 子线程

> 通过 `think(context="fork")` 在线程树下创建新节点。子线程独立执行，完成后 return 结果给父线程。

> 2026-04-22 起 `think` 统一了 `create_sub_thread` / `continue_sub_thread`——
> 同一语义通过 `context="fork"` 或 `context="continue"` 切换，旧命令已直接删除（不做兼容层）。

## 为什么需要子线程

### 保持焦点

当前线程在做 A 任务，中途发现需要做 B 子任务。如果在同一线程做 B，B 的思考会污染 A 的 Context。

**解法**：开子线程做 B。B 完成后只把摘要返回给 A。A 不看 B 的具体过程。

### 作用域隔离

子任务可能需要**不同的 trait 作用域**。比如 A 线程在写代码，需要临时做一次代码审查——审查要激活 reviewable，但审查完就该卸载。

**解法**：开子线程做审查。reviewable 在子线程激活，子线程 return 后自动卸载。

### 并行

多个独立子任务可以并行开子线程。父线程 `await_all` 等全部完成。

## 创建子线程（think fork）

```
open(type=command, command=think, description="搜索相关文档")
submit(form_id, {
  title: "搜索相关文档",               // 同时作为子线程名
  context: "fork",                     // 必填
  msg: "在 docs/ 下找与 X 相关的 md 文件，返回前 5 个最相关",
  traits: ["kernel/computable"],       // 可选：新子线程的 trait 列表
})
→ 返回 thread_id: "th_xxx"
```

## 参数

| 参数 | 含义 | 必填 |
|---|---|---|
| `title` | 子线程标题（tool call 的行动标题；同时是子线程名） | 是 |
| `context` | 必须为 `"fork"` | 是 |
| `msg` | 消息内容，作为子线程首轮 inbox + description | 是 |
| `threadId` | 以哪个线程为父（省略则以当前线程为父） | 可选 |
| `traits` | 子线程静态 trait 列表 | 可选 |

## 生命周期

```
父线程 think(fork)
    ↓
 创建子节点 (status=running)
    ↓
 Scheduler 开始调度子线程
    ↓
 子线程独立执行（新的 ThinkLoop 循环）
    ↓
 子线程 return
    ↓
 子线程 status=done
 return summary 写入父线程 inbox
    ↓
 父线程收到 inbox 消息（如果在 wait，则恢复 running）
```

## 作用域继承

子线程**继承父线程的 trait 激活状态**（沿 scope chain 向上收集）：

```
父: activatedTraits = {base, talkable, computable}
子: activatedTraits = {base, talkable, computable}（scope chain 继承）
```

子线程可以直接用这些能力，不需要重新 open。
若要额外引入子线程专属 trait，使用 `traits` 参数。

## 与 talk 的关系

`think` 和 `talk` 的区别：

| 维度 | think | talk |
|---|---|---|
| 目标 | 同一对象的线程 | 另一个对象的线程 |
| 参数 | `{msg, threadId?, context}` | `{target, msg, threadId?, context}` |
| 作用域 | 继承父（scope chain） | 目标对象的独立作用域 |
| 通信 | 父子直接（通过 inbox） | 跨对象通信 |
| 场景 | 任务拆分 | 能力调用 / 协作 |

两个指令**参数一致**（除 target），只是语义上对应不同的作用域。

什么时候用哪个？

- 任务在**当前对象的能力范围内** → `think`
- 任务需要**另一个对象的专业能力** → `talk`

## 等待子线程

```
// 等待单个
await(thread_id="th_xxx")

// 等待多个（全部完成）
await_all(thread_ids=["th_xxx", "th_yyy"])
```

`await` 让父线程进入 waiting。子线程完成后自动唤醒。

## 向子线程追加消息（think continue）

向已创建的子线程**追加消息**（替代旧 `continue_sub_thread`）：

```
open(type=command, command=think, description="补充信息")
submit(form_id, {
  title: "补充 Y 的检查要求",
  context: "continue",
  threadId: "th_xxx",
  msg: "继续：再检查 Y"
})
```

`context="continue"` 时 `threadId` 必填（engine 会校验）。
如果子线程已 done，自动**复活**（详见 [线程复活.md](线程复活.md)）。

## 子线程的 Context

子线程第一轮思考时的 Context：

- **whoAmI**：同对象 readme（同父）
- **parentExpectation**：子线程的 description（来自 think(fork) 时传入的 msg 或 description）
- **process**：父线程渲染快照（通过 inject action 注入）
- **inbox**：首条消息（msg 投递）
- **activeForms**：空
- **directory**：同对象目录
- **instructions + knowledge**：scope chain 继承
- **childrenSummary**：空
