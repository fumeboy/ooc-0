# 节点状态

## 四个状态

| 状态 | 含义 | 典型转换 |
|---|---|---|
| **running** | 正在执行 ThinkLoop | → waiting（wait 子线程/消息）<br>→ done（return）<br>→ failed |
| **waiting** | 等待外部信号 | → running（收到信号）|
| **done** | 已完成（root 线程 done 表示任务结束） | → running（线程复活） |
| **failed** | 失败（根因 + 堆栈记录在 actions 最后一条） | （终态，除非手动 reset） |

## 状态转换图

```
         ┌─────── running ────────┐
         │          ↑             │
         ↓          │             ↓
      waiting ──────┘           done
         │          ↑             │
         │          │             ↓
         └──────────┴────────  failed
                 新消息触发复活
```

- **running → waiting**：调用 wait / talk_sync 等待子线程或消息
- **waiting → running**：子线程完成、消息到达、pause 恢复
- **running → done**：调用 return
- **done → running**：**线程复活**（新消息进入 inbox 时自动）
- **任何 → failed**：未捕获错误、强制终止

## running

线程当前轮次被 Scheduler 调度，正在执行 ThinkLoop：
- 构建 Context
- 调用 LLM
- 处理 tool calls

一次 ThinkLoop 后，状态可能保持 running（下一轮继续）或切换。

## waiting

线程暂停，释放调度权。常见原因：

1. **等待子线程**：`wait([child_thread_id])` — 直到子线程 return 后恢复
2. **等待消息**：`wait()` — 任意新消息到达后恢复
3. **talk_sync 隐式 wait**：发送 talk_sync 后自动 waiting
4. **pause 检查点**：人工暂停（状态为 pausing，细分自 waiting）

## done

线程已 return。完成形态：
- 根线程 done → 整个任务完成
- 子线程 done → return summary 写入父线程 inbox，父线程复活或收到新消息

**done 不是永恒的**——收到新 inbox 消息会自动复活。

## failed

线程遇到未处理的异常终止。`thread.json.actions` 最后一条记录错误信息：

```json
{
  "type": "error",
  "message": "Uncaught: ReferenceError...",
  "stack": "...",
  "ts": "..."
}
```

failed 状态**不会自动复活**——需要人工介入或父线程重试。

## 状态与 Context 的关系

Context 构建**只对 running 线程**进行。waiting 的线程不主动构建新 Context——等被唤醒时才构建。

done / failed 的线程不参与调度。

## 状态转换的事件

| 事件 | 触发 |
|---|---|
| `thread_started` | 新线程创建 |
| `thread_revived` | done → running |
| `thread_returned` | running → done |
| `thread_failed` | 任何 → failed |
| `thread_waiting` | running → waiting |
| `thread_resumed` | waiting → running |

这些事件通过 SSE 推送给前端，用于实时更新线程树视图。

## 与基因的关联

- **G9**（线程树）— 节点状态是线程树的生命语义
- **G13**（线程树即运行模型）— 状态机驱动运行时
