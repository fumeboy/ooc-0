# 线程树 — 思考的作用域结构

> G13：对象的运行时 = 一棵线程树。每个节点 = 一个线程 = 一层认知作用域。
> 这替代了旧的"认知栈 + 行为树"，统一为单一数据结构。

## 核心结构

```
根线程（由用户消息或 talk 创建）
  ├── 子线程 A（独立处理一个子任务）
  │   ├── 孙线程 A1
  │   └── 孙线程 A2
  └── 子线程 B
```

每个节点是一个**线程**，有自己的：
- `actions`（本线程的思考/行动历史）
- `inbox`（本线程的未读消息）
- `activatedTraits`（本线程激活的 trait）
- `status`（running / waiting / done / failed）

## 与旧架构的对比

### 旧：认知栈 + 行为树

```
认知栈：线性，push/pop，后入先出
行为树：分开维护，plan 在 process.json，状态独立
```

问题：
- 同一个"思考"分散在两个结构里
- 子任务创建和 context 切换逻辑复杂
- 难以表达"线程可以被唤醒"

### 新：线程树统一

```
线程树：单一数据结构
  - 节点 = 认知作用域
  - 父子关系 = 子任务委派
  - 状态 = 线程是否活跃
```

好处：
- 单一真相源
- 作用域链清晰（沿树向上）
- 线程可反复唤醒（revival）

## 关键概念

### Scope Chain（作用域链）

从当前节点沿树**向上**收集。决定哪些 trait、哪些知识、哪些记忆对当前线程可见。

详见 [../context/scope-chain.md](../context/scope-chain.md)。

### 线程复活

`done` 线程**不是一次性**的——任何时候有新的 inbox 消息进来，自动恢复为 `running`。

这让线程像一个"认知通道"——**对象在同一个话题上与对方的所有对话都在同一棵线程**。

详见 [线程复活.md](线程复活.md)。

### 根节点的厚度 = 智慧

**新手**：要做一件事需要开很多子线程（每一步都要单独规划）。
**专家**：根线程已经内化了大量经验，同样的事只需要很浅的树。

这在工程上体现为：对象的 `readme` + `traits` 沉淀越丰富，线程树越浅。这就是 G12（经验沉淀）在运行时的可观测体现。
