# 调度（读侧） — ThreadScheduler 的基本策略

> 本文档讲**认知视角**的调度：每轮选哪个线程执行。
> 完整的调度机制、死锁检测、跨对象协作见 [../../合作/基础/线程树调度.md](../../合作/基础/线程树调度.md)。

## 基本循环

```
while (有 running 线程) {
  for each running thread:
    执行一轮 ThinkLoop
  检查 waiting 线程是否可唤醒
  检查死锁
}
```

## 每轮只选一个线程

ThreadScheduler 采用**轮询制**：每次 tick 只调度**一个** running 线程执行一轮。

理由：
- 避免单个对象独占资源
- 让子线程和父线程都能推进
- 保持调度公平性

## 线程状态与调度的关系

| 状态 | 是否被调度 |
|---|---|
| **running** | 是 |
| **waiting** | 否（但监测唤醒条件） |
| **done** | 否（但监测复活触发） |
| **failed** | 否 |
| **pausing** | 否（等待人工恢复） |

## 为什么线程间要轮询而非同步

一个 Session 可能有多个 Flow（多对象），每个 Flow 可能有多个线程。如果某个线程卡住（如 LLM 调用超时），不应阻塞其他线程。

轮询保证：**每个活跃线程都有推进机会**。

## 等待机制

线程通过 `wait` 进入 waiting。ThreadScheduler 检查条件：

```typescript
for each waiting thread:
  if 等待子线程完成:
    if 子线程 done:
      thread.status = running  // 唤醒
  if 等待 inbox 消息:
    if inbox.unread.length > 0:
      thread.status = running

  if 死锁（详见合作/基础/线程树调度.md）:
    强制唤醒并注入 deadlock_notice
```

## 调度顺序：深度优先 vs 广度优先

OOC 调度器倾向**公平性**而非"某个策略"：

- 不是严格深度优先（不会一直钻到最深的叶子）
- 不是严格广度优先（不会总是所有兄弟先跑一轮）
- 而是**按线程的 lastExecutedAt 时间戳排序**，最久未执行的优先

这让深层子线程有机会推进，同时保持兄弟间的协同。

## 终止条件

### 单 Flow 终止

根线程 `done` 或 `failed` → 该 Flow 执行结束。

### Session 终止

Supervisor 判断"任务整体完成"或用户 signal 终止 → Session 结束，所有 Flow 的 .flow 标记被清除。
