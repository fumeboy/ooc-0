# 线程复活 — done 不是终点

> done 线程收到任何 inbox 消息时，**自动恢复**为 running。
> 线程不是一次性执行单元，而是可**反复唤醒**的认知通道。

## 核心机制

```
thread status: done
    ↓
 新消息写入 thread.inbox
    ↓
 系统检测：status=done 且 inbox 有 unread
    ↓
 status: done → running
 revivalCount += 1
 Context 注入 revival_notice
```

## revivalCount

每个线程有 `revivalCount` 字段，记录被复活次数：

```json
{
  "id": "th_xxx",
  "status": "running",
  "revivalCount": 3,
  // ...
}
```

初始为 0；每次 done → running 复活时 +1。

## revival_notice 注入

复活时，Engine 会在下一轮 Context 中注入 **revival_notice**：

```
[revival notice]
你之前已 return 这个线程。现在又收到了新消息。你可以：
1. 继续完成工作（从新消息看是否需要追加行动）
2. 再次 return（如果新消息不需要行动）
3. 如果是错误的唤醒，可以 mark(ignore) 消息后 return

这是第 3 次复活。
```

这让 LLM 理解"我以为结束了，但有新情况"——不会把当前轮当作全新任务处理。

## 为什么要复活机制

### 持续对话

用户和对象之间通常不是一问一答就结束——会来回多轮。如果每次对话都是**新线程**：

```
用户说 A → 线程 1 处理 A，return
用户说 B（跟 A 相关） → 线程 2 处理 B，但线程 2 不知道 A 的上下文！
```

**解法**：同一个"话题"共享一棵线程树。用户 B 消息写入原线程 inbox，触发复活——线程 2 还是线程 1，保留了全部历史。

### 异步协作

子线程 return 后，父线程可能**很久以后**才收到新的 talk 消息。如果子线程永远 done：

```
子线程 return
...（几小时后）...
新情况需要子线程再做点事，但已 done
```

**解法**：复活机制让子线程在新情况到达时重新 running。`do(context="continue")` 就是利用这一点。

### 行动单元 vs 认知通道

传统思维里，"线程"是一段代码执行单元——跑完就结束。
OOC 里，"线程"是**话题的认知通道**——只要话题还活着，线程就可以继续。

这个哲学转变是 G13 的核心之一。

## 触发复活的消息来源

1. **talk**：其他对象发消息进来（`talk(target, msg, threadId=X, context="continue")` 投递到对方线程 X）
2. **do(continue)**：父线程向已 done 子线程追加消息（替代旧 `continue_sub_thread`）
3. **system**：系统通知（如告知某文件变更）

所有消息进入 inbox 的路径都可能触发复活。

## 什么不触发复活

- **ack/ignore/todo 标记操作**：不是新消息，不复活
- **failed 线程**：不复活（失败是终态，除非手动 reset）
- **子线程 done 但无新消息**：不自动复活（done = 阶段性完成）

## 复活后的 Context 构建

复活后的第一轮 Context：

- **whoAmI**：同 Stone readme
- **parentExpectation**：同创建时的 description
- **processEvents**：**之前 done 时的 events 全部保留**（不是空）
- **inbox**：当前未读消息（含触发复活的那条）
- **revival_notice**：注入上述提示
- **revivalCount**：注入实际次数

对象看到的是"完整历史 + 新情况"，可以无缝继续。

## 复活的代价

每次复活都需要构建 Context。如果线程复活数十次，`process` 会非常长——**三层记忆压缩**（详见 [../context/三层记忆.md](../context/三层记忆.md)）在这时起作用：
- recent 只保留最近 N 轮
- 早期 events 被压缩或归档

所以频繁复活的线程**Context 实际大小相对可控**。
