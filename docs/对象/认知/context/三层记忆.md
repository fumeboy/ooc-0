# 三层记忆

> Context 容量有限。对象的历史被压缩为三层：永久（long-term）、会话（session）、近期（recent）。

## 三层结构

| 层次 | 存储 | 生命周期 | 特点 |
|---|---|---|---|
| **long-term** | `stones/{name}/readme.md` + `memory.md` | 永久 | 对象主动沉淀，跨所有任务可见 |
| **session** | `flows/{sid}/objects/{name}/memory.md` | 当前 Session | Flow 可直接写，任务结束消散 |
| **recent** | 当前线程 `thread.json.actions` | 线程级 | 最近 N 轮完整保留 |

## 为什么分三层

### 问题：actions 历史无法无限增长

一个复杂任务可能跑几百轮。每轮的 `thought + tool_use + result` 如果全部注入 Context，几万 token 就用完了。

### 分层策略

```
  任务开始时    ──→  recent 少，session 空，long-term 沉淀较少
  任务进行中    ──→  recent 增长到阈值，开始压缩中段进 session memory
  任务结束时    ──→  session memory 汇成摘要，值得的进 long-term
```

- **Recent** 保留最近 N 轮（通常 5-20 轮）完整
- 更老的 actions → 压缩为**中期摘要**
- 更更老的 → **远期摘要** 或直接归入 session memory

## 具体机制

### Recent：完整保留

```
thread.json.actions 数组
  [action_1, action_2, ..., action_N]
```

最近 N 轮直接以完整 `ThreadAction` 形式注入 Context.process。

### 中期：压缩为摘要

N+1 轮到某个"压缩边界"之间的 actions，被压缩为：

```
## 中期历史（第 11-30 轮摘要）
- 搜索了 3 份文档
- 尝试了 2 次 talk_sync，第 2 次成功
- ...
```

压缩由 Engine 在 Context 构建时按需进行（not by storing a compressed version，但逻辑一致）。

### 远期：极简归档

更早的 actions 被压缩为一两句话，或直接标记为"归档，详见 session memory"。

### Session Memory

Flow 可以主动写 `memory.md`：

```markdown
# Session 记忆 — Alan

## 2026-04-21 10:30
用户让我评估 thread-tree 集成。发现 issue-discussion trait 的位置有问题。
```

这段内容在 Context 构建时作为 `long-term` 的**补充**（或作为独立字段）注入。

### Long-term Memory

Stone 级的 `memory.md` 只能通过反思写入：

```
talk({ target: "super", content: "请记住：在 thread-tree 架构中，issue-discussion 挂在 talkable 下" })
  ↓
SelfMeta 审视 → 决定写 memory.md
```

下次 Alan 的任何任务开始时，这段记忆进入 Context 的 whoAmI/knowledge 附近。

## Context 字段归属

| 层次 | 进入 Context 的位置 |
|---|---|
| long-term | whoAmI（readme 正文）+ 独立的 memory 段 |
| session | Context 内独立段（Flow memory） |
| recent | process 字段 |

## 压缩策略（Exp 012 验证）

OOC 采用 **"三区压缩"** 策略：

- **近期**（recent）：保留最近 5 轮完整
- **中期**（mid）：保留最多 15 条摘要
- **远期**（far）：保留 20 条极简归档

超出阈值时，最旧的条目往更外层移动。

## 与遗忘的关系

三层记忆本质上是**遗忘机制**——让不相关的信息退场，为当前任务腾出空间。

G5 说"遗忘是智能的基础设施"——不是丢失，是让不相关退场。

详见 [../../成长/遗忘.md](../../成长/遗忘.md)。

## 对象主动管理记忆

对象可以通过以下方式主动管理自己的记忆：

```typescript
// 写会话记忆
await updateSessionMemory("关键发现：X 导致 Y");

// 写长期记忆（需通过向 SuperFlow 发消息）
await talk({ target: "super", content: "请记住：X 导致 Y 是因为 Z" });

// 写 Flow Summary（任务结束时的一句话摘要）
await updateFlowSummary("本次会话讨论了 A，结论是 B");
```

这让对象的"记忆"不是被动压缩的结果，而是**主动选择**的产物。
