# Context 的九大组成

> 每次 LLM 调用发送的 Context 由 9 个字段组成。每个字段有明确的来源和语义。

## 总览

```
ContextData = {
  whoAmI,              // 我是谁
  instructions,        // 作为 OOC 对象的基础规则
  knowledge,           // 当前激活的具体能力
  parentExpectation,   // 父线程对我的期望
  processEvents,       // 我在做什么（events 历史）
  inbox,               // 别人对我说了什么（未读消息）
  activeForms,         // 我正在进行的操作
  directory,           // 我拥有什么（目录 + relations）
  childrenSummary,     // 子线程完成情况
}
```

## 1. whoAmI — 我是谁

**来源**：`stones/{name}/readme.md` 的正文部分

**内容**：对象的完整自我描述——目标、风格、知识背景、偏好

**容量**：无硬限制（通常几百到几千字）

**典型用法**：
```
# 关于我自己
我是 Alan Kay，OOC 项目的 Supervisor。
我具有丰富的哲学、逻辑学...

# 我的终极目标
我正在打造世界上最聪明的 Agent 系统...
```

## 2. instructions — 系统指令

**来源**：激活的 **kernel trait** 的 TRAIT.md（`type: how_to_think` 或 `how_to_interact`）

**内容**：作为一个 OOC 对象的基础规则，包括：
- base 的四原语用法（`open/submit/close/wait`）
- 当前激活的能力层 trait 的指令（如 computable 的代码执行规则）
- 反思与验证约束（reflective、verifiable）

**特点**：系统级的"做事的规则"，而非"具体知识"。

## 3. knowledge — 知识窗口

**来源**：激活的 **library / user trait** 的 TRAIT.md

**内容**：具体能力的知识——API 参考、域知识、操作手册

**与 instructions 的区别**：
- instructions = 系统级规则（OOC 对象怎么思考）
- knowledge = 领域知识（具体怎么用某个能力）

例：激活了 `library/browser` 后，knowledge 里有浏览器操作的详细说明。

## 4. parentExpectation — 父线程的期望

**来源**：当前线程树节点的 `title` + `description` 字段

**内容**：父线程创建此线程时传入的任务描述。子线程读到这个就知道"我被要求做什么"。

**根线程**的 parentExpectation 是**用户消息**本身。

## 5. processEvents — 我在做什么

**来源**：当前线程 `thread.json` 的 `events` 数组

**内容**：本线程内历次思考、工具调用、结果的完整序列

**三层压缩**（详见 [三层记忆.md](三层记忆.md)）：
- 最近 N 轮完整保留
- 中段压缩（events 摘要）
- 远段归入 session memory

## 6. inbox — 未读消息

**来源**：当前线程 `thread.json` 的 `inbox`，过滤出 unread

**内容**：其他对象通过 talk 发来的消息，系统通知，错误报告

**字段**：每条消息含 `messageId`、`from`、`content`、`marked`（ack/ignore/todo/null）

**展示原则**：inbox 不过滤已标记消息——让上下文保持完整。对象用 `mark` 参数主动标记，避免重复处理。

详见 [../../合作/消息/](../../合作/消息/)。

## 7. activeForms — 正在进行的操作

**来源**：当前线程的 FormManager 状态

**内容**：当前 open 但未 submit / close 的 form 列表：

```
活跃表单：
  - form_id: f_001
    type: command
    command: program
    description: "写入配置文件"
    opened_at: "..."
```

**作用**：提醒 LLM"你还有未完成的 open，要不要 submit 或 close"。

详见 [../指令系统/form-manager.md](../指令系统/form-manager.md)。

## 8. directory — 我拥有什么

**来源**：
- 对象的持久化目录内容（files/、自定义 trait、memory.md 等）
- readme.md 的 `relations` 字段

**内容**：
```
你的目录：
  files/
    - report-2026-04.md
    - data.csv
  traits/
    - my-workflow

你的关系：
  - filesystem: 可以读写文件
  - browser: 可以搜索互联网
```

**作用**：让对象知道"我手边有哪些资源"（文件、自定义能力、关系）。

## 9. childrenSummary — 子线程完成情况

**来源**：当前线程树节点的子节点摘要

**内容**：每个子线程的状态 + return 摘要：

```
子线程：
  - [done] 搜索相关文档: 找到 3 份相关文档，详见 inbox
  - [running] 分析实验数据: ...
  - [failed] 编译检查: 类型错误在 line 42
```

**作用**：父线程不需要读子线程的完整 events 就知道"他们做得怎样了"。

## 为什么是这 9 个

这 9 个字段覆盖了 LLM 生成下一步 tool call **所需的全部信息维度**：

- **我是谁**（whoAmI）+ **怎么做事**（instructions + knowledge）= 主体
- **要我做什么**（parentExpectation）+ **做到哪了**（process）= 目标 + 进度
- **外界发生了什么**（inbox）+ **我启动了什么**（activeForms）+ **我的助手做到哪了**（childrenSummary）= 事件
- **我手边有什么**（directory）= 资源

少了任何一个，LLM 都可能做出不合理的选择。

## 组装流程

`kernel/src/thread/context-builder.ts` 的主函数：

```typescript
async function buildContext(threadId: string): Promise<ContextData> {
  return {
    whoAmI: await buildWhoAmI(obj),
    instructions: await buildInstructions(activatedTraits.filter(isKernel)),
    knowledge: await buildKnowledge(activatedTraits.filter(isNonKernel)),
    parentExpectation: await buildExpectation(threadId),
    processEvents: await buildProcessEvents(threadId),
    inbox: await buildInbox(threadId),
    activeForms: await buildActiveForms(threadId),
    directory: await buildDirectory(obj),
    childrenSummary: await buildChildrenSummary(threadId),
  };
}
```
