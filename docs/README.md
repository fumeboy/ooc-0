# OOC 文档

> Object-Oriented Context — 一种 AI 智能体架构：把上下文组织为「活的对象生态」。

## 三分结构

OOC 的文档按三层组织。每一层回答一个不同层次的问题。

```
docs/
├── meta.md          ← 全局入口：概念总树 + 工程子树（先从这里读）
├── 哲学/            ← 元层 · 思想侧：为什么这样设计
├── 对象/            ← 本体层：OOC 系统是什么
└── 工程管理/        ← 元层 · 实践侧：我们如何做 OOC
```

| 层 | 内容 | 读者视角 |
|---|---|---|
| **哲学/** | 13 条基因 + 12 条涌现能力 + 两个循环 + 统一性 | 想理解"为什么是这样" |
| **对象/** | 对象的 6 个维度（存在/结构/认知/合作/成长/人机交互） | 想理解"系统是怎样构造的" |
| **工程管理/** | 目标/组织/流程/规范/验证 | 想理解"我们是怎么做出来的" |

## 阅读路径

**第一次接触 OOC**：`meta.md` → `哲学/README.md` → `哲学/genes/` → `对象/README.md`

**参与开发**：`meta.md` → `对象/` 中与工作相关的分支 → `工程管理/规范/` → `工程管理/流程/`

**做哲学设计**：`哲学/` 全读 → `对象/成长/` → `哲学/discussions/`

**体验验证**：`工程管理/验证/用例/` → 对照 `对象/人机交互/`

## 三个元层的对称

OOC 在文档结构上有意呈现一种对称：

```
哲学/  ← 关于思想的思想     （"为什么"）
         ↓
对象/  ← 系统本身           （"是什么"）
         ↑
工程管理/  ← 关于设计的设计  （"怎么做"）
```

这不是偶然。OOC 的核心主张是"**用 OOC 的方式做 OOC**"——文档结构本身也是一个对象，它有身份（meta.md）、结构（三层）、成长（演进）。

## 文档写作约定

详见 `工程管理/规范/文档规范.md`。关键点：

- 目录对应概念树的节点，文件对应叶子
- 跨领域引用用 `@ref path` 或 `[[链接]]` 语法
- 每个目录至少有一份 README.md 作为导航
- 代码 ↔ 文档必须对齐：如果改了代码但没改文档，这个 commit 不完整
