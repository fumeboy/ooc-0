# Q: 为什么让对象自己编写 UI？

## 问题

传统架构中，前端是独立的，需要为每种数据类型设计展示方式。

问题：
- 前端开发者需要理解所有数据类型的语义
- 新增数据类型需要修改前端代码
- 展示逻辑与业务逻辑分离，难以保持同步

## 决策

让对象决定自己如何被看见（G11）。

- 对象最了解自己的数据结构和功能
- UI 是对象「表达自己」的一种方式
- 通过 trait 告诉对象如何编写 UI，而不是替对象写 UI

类比：浏览器不知道每个网站长什么样，但网站可以通过 HTML/CSS 告诉浏览器。

## 新模型适配

旧模型中 UI 能力通过 `ui_template` bias 提供。
新模型中，UI 能力通过 `ui_template` kernel trait 提供（readme.md 是编写指导，index.ts 提供 writeUI() 方法）。
UI 文件本身存储在对象的 `ui/index.tsx`——它是对象的顶层组成部分（面孔），
与 readme.md（身份）、data.json（状态）平级，不是某个 trait 的附属品。
前端通过扫描 `objects/*/ui/index.tsx` 动态加载。
