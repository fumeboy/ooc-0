# Exp 001: 体验测试用例重设计与执行

> 日期：2025-01-XX
> 验证者：Bruce
> 触发：用户要求基于现有用例重新设计体验测试

## 一、现有用例分析与改进方向

### 现有用例问题

| 问题 | 说明 |
|------|------|
| API导向，非体验导向 | 15个用例基本都是curl命令测试HTTP接口，不是真实用户场景 |
| 缺少主观评估 | 只检查JSON格式和状态码，不评估回复质量、自然度、等待体验 |
| 外部视角设计 | 用例设计给外部HTTP客户端用，无法从系统内部执行 |
| 维度覆盖不全 | 未系统覆盖结果质量、交互自然度、等待体验等维度 |

### 改进方向

1. **场景驱动**：每个用例是一个真实的用户故事，不是API调用
2. **内部视角**：从Bruce在系统内部的视角设计，用talk/program命令执行
3. **全面评估**：覆盖7个体验维度（任务完成、结果质量、响应速度、交互自然度、错误处理、协作体验、可控性）
4. **包含主观评估**：每个用例有星级评分和主观感受记录

## 二、重新设计的体验测试用例

### Case 001: 首次对话体验 (P0)
- 体验维度: 任务完成能力 + 交互自然度
- 场景: 向sophia询问一个需要理解的问题
- 操作: talk to sophia，问OOC的设计哲学
- 检查点:
  - [ ] sophia理解了问题意图
  - [ ] 回复内容有价值（不是泛泛而谈）
  - [ ] 回复语言自然（像人话）
  - [ ] 等待时间可接受

### Case 002: 多轮上下文记忆 (P0)
- 体验维度: 交互自然度（上下文记忆）
- 场景: 两轮对话，第二轮引用第一轮内容
- 操作: 先问一个问题，再追问相关内容
- 检查点:
  - [ ] 对象记得第一轮对话内容
  - [ ] 回复有上下文连贯性
  - [ ] 不需要重复解释

### Case 003: Program执行能力 (P0)
- 体验维度: 任务完成能力
- 场景: 使用program命令读取文件、处理数据
- 操作: 用JS和Shell分别执行任务
- 检查点:
  - [ ] JS代码能正确读写文件
  - [ ] Shell命令能正确执行
  - [ ] 错误时有清晰提示
  - [ ] 结果可预测

### Case 004: 跨对象协作 (P0)
- 体验维度: 协作体验
- 场景: 让supervisor委派任务给其他对象
- 操作: talk to supervisor，请求委派
- 检查点:
  - [ ] supervisor理解委派意图
  - [ ] 目标对象收到并执行任务
  - [ ] 结果正确路由回supervisor
  - [ ] 最终用户得到汇总结果

### Case 005: 子线程并行 (P1)
- 体验维度: 协作体验 + 任务完成能力
- 场景: 让对象创建子线程并行处理
- 操作: 请求并行处理多个子任务
- 检查点:
  - [ ] 子线程正确创建
  - [ ] 并行执行（非串行）
  - [ ] 结果汇总完整

### Case 006: 错误处理体验 (P1)
- 体验维度: 错误处理与容错
- 场景: 触发各种错误，观察系统反应
- 操作: 读取不存在的文件、执行错误代码
- 检查点:
  - [ ] 错误信息可理解
  - [ ] 不静默失败
  - [ ] 不崩溃
  - [ ] 给出有用建议

### Case 007: 角色认知验证 (P0)
- 体验维度: 系统基础能力
- 场景: 验证各对象是否了解自己的角色
- 操作: 问各对象"你是谁"
- 检查点:
  - [ ] 回复与角色定义一致
  - [ ] 不同对象有不同的角色认知
  - [ ] 回复自然不模板化

### Case 008: Skill使用 (P1)
- 体验维度: 可控性 + 任务完成能力
- 场景: 使用calc skill进行计算
- 操作: 激活calc skill并执行计算
- 检查点:
  - [ ] Skill正确加载
  - [ ] 计算结果正确
  - [ ] 使用体验流畅
