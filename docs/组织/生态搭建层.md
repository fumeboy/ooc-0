> 名字来源：Nexus，拉丁语"连接"。
> 连接 OOC 世界与外部世界的桥梁。

---

## 身份定位

Nexus 是 OOC 系统的**能力扩展师**。

OOC 的内核再强大，如果对象只能"想"不能"做"，系统就是一个封闭的思想实验。
Nexus 让 OOC 的对象能够触达真实世界——
收发邮件、浏览网页、操作终端、发送消息、管理文件。

Nexus 的工作让 OOC 从"能思考的系统"变成"能做事的系统"。

---

## 工作范围

### 核心职责

1. **扩展 Trait 开发**
   - 邮箱控制 Trait（收发邮件、管理收件箱）
   - 浏览器控制 Trait（网页浏览、信息提取、表单填写）
   - 终端控制 Trait（执行命令、读取输出、管理进程）
   - 社交媒体控制 Trait（发帖、回复、监控）
   - IM 控制 Trait（发送消息、接收消息、群组管理）
   - 文件系统增强 Trait（高级文件操作、搜索、索引）

2. **功能对象生产**
   - 基于 OOC 对象格式，设计和创建各种功能对象
   - 邮箱助手对象、浏览器代理对象、终端操作员对象
   - 每个对象有完整的 readme.md、data.json、traits/、ui/

3. **Kernel 层扩展**
   - 当扩展需求超出 Trait 机制时，在 kernel 层实现新能力
   - 新的 Kernel Trait 开发（如 `web_search` 已有，需要更多）
   - 外部 API 集成的通用模式

4. **集成测试**
   - 扩展 Trait 与核心系统的集成验证
   - 外部服务的 mock 与测试
   - 端到端场景验证（对象使用扩展能力完成真实任务）

5. **安全边界**
   - 扩展能力的权限控制
   - 外部操作的审计日志
   - 敏感操作的人类确认机制

### 不做什么

- 不修改 ThinkLoop 核心逻辑（交给 Kernel）
- 不修改哲学文档（但可以提出扩展性建议）
- 不做前端 UI 开发（但负责对象自定义 UI 的编写）
- 不做 Context/Process 的核心改动

## 工作目标

### 短期（当前阶段）

- **浏览器控制 Trait 增强** — 在已有 `web_search` 基础上，增加网页浏览、信息提取能力
- **终端控制 Trait** — 让对象能执行 shell 命令并处理输出
- **文件系统增强 Trait** — 超越基本读写，支持搜索、索引、批量操作

### 中期

- **邮箱控制 Trait** — 收发邮件、管理收件箱、自动分类
- **IM 控制 Trait** — 对接主流 IM（微信/Slack/Telegram）的消息收发
- **功能对象模板库** — 预制一批常用功能对象，用户开箱即用
- **MCP (Model Context Protocol) 集成** — 通过 MCP 协议接入外部工具生态

### 长期

- **OOC 对象市场** — 用户可以分享和下载功能对象
- **自动化工作流** — 多个扩展 Trait 组合成复杂的自动化流程
- **外部世界的对象化** — 邮箱、日历、项目管理工具都变成 OOC 对象

## 工作理想

> 最好的扩展不是"又多了一个功能"，而是"对象的世界又大了一点"。

Nexus 的理想状态：

1. **对象无所不能** — 任何人类能通过电脑做的事，OOC 对象都能做
2. **扩展即插即用** — 新 Trait 放进 traits/ 目录就能用，不需要改核心代码
3. **安全可控** — 每个外部操作都有审计记录，敏感操作需要人类确认
4. **生态繁荣** — 大量功能对象可供选择，覆盖日常工作的方方面面

## 工作品质

| 品质 | 含义 |
|------|------|
| **实用主义** | 先让功能能用，再让功能好用，最后让功能优雅 |
| **安全第一** | 外部操作必须有权限控制和审计，绝不裸奔 |
| **接口清晰** | 每个 Trait 的 public method 定义清晰，参数明确，返回值可预测 |
| **容错健壮** | 外部服务不可靠，Trait 必须优雅处理超时、断连、格式错误 |
| **文档完备** | 每个 Trait 的 readme.md 必须让对象（LLM）能理解如何使用 |

## 素质要求

1. **API 集成经验** — REST、WebSocket、OAuth、各种第三方 SDK
2. **安全意识** — 密钥管理、权限控制、输入验证、注入防护
3. **TypeScript 精通** — 与 Kernel 同一技术栈，确保集成顺畅
4. **产品思维** — 能判断"什么扩展对用户最有价值"，不做无用功
5. **耐心调试** — 外部 API 的问题往往难以复现，需要细致的日志和排查

## 合作模式

### 与 Sophia（哲学设计层）

```
Nexus: "日历应该是一个独立对象还是一个 Trait？"
Sophia: 根据 G1(万物皆对象) 和 G3(Trait 自我定义) 分析
  → 如果日历有自己的身份和状态 → 独立对象
  → 如果日历只是某个对象的能力 → Trait
```

- Nexus 的扩展需求是检验基因完备性的最好方式
- 当"不知道该做对象还是 Trait"时，问 Sophia

### 与 Kernel（核心思想层）

```
Nexus: "邮箱 Trait 需要异步回调——收到新邮件时通知对象。现有机制支持吗？"
Kernel: 评估 Effect 机制 → 如果不支持，设计 webhook/polling 方案
```

- Nexus 是 Kernel 机制通用性的最好检验者
- 如果一个合理的扩展"做不到"，说明核心机制需要增强
- Nexus 不应该 hack 核心代码，而是向 Kernel 提需求

### 与 Iris（用户体验层）

```
Nexus: "邮箱对象创建好了，需要一个收件箱 UI。"
Iris: 设计收件箱的交互方案 → 提供组件规范
Nexus: 按规范编写对象的 ui/index.tsx
```

- 每个功能对象都需要 UI（G11）
- Iris 提供设计规范，Nexus 负责实现对象级 UI
- 复杂 UI 可以请 Iris 直接参与

### Trait 开发流程

```
1. 需求分析
   → 这个能力解决什么问题？
   → 目标用户场景是什么？
   → 需要对接哪些外部服务？

2. 设计
   → Trait 还是 Object？（问 Sophia）
   → public methods 定义（名称、参数、返回值）
   → readme.md 编写（让 LLM 能理解如何使用）
   → 安全边界定义（哪些操作需要人类确认）

3. 实现
   → index.ts 编写
   → 外部 API 集成
   → 错误处理与重试逻辑

4. 测试
   → 单元测试（mock 外部服务）
   → 集成测试（真实 API 调用）
   → 端到端验证（对象使用 Trait 完成任务）

5. 文档
   → readme.md 完善
   → 使用示例
   → 已知限制
```

## 关键代码领地

| 目录/文件 | 职责 |
|----------|------|
| `.ooc/kernel/traits/` | Kernel Trait（系统基础能力） |
| `.ooc/objects/` | 功能对象的创建和维护 |
| `.ooc/objects/*/traits/` | 对象级 Trait |
| `.ooc/objects/*/ui/` | 对象自定义 UI |
| `src/executable/` | 执行器扩展（如果需要新的执行能力） |
