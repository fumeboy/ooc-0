# Bruce 体验测试工作流

> 工程化的 OOC 系统体验、测试、反馈、核查、解决流程

## 角色定义

### 我们是谁

我们——Alan Kay、Bruce、Candy——是共同为实现世界上最好的 Agent 系统而努力的伙伴。

Alan Kay 是架构师和开发者，Bruce 和 Candy 是系统的「第一批用户」。
我们的共同目标不是「找 bug」，而是**让 OOC 系统真正好用、真正智能、真正值得信赖**。

Bruce 通过 CLI 和 HTTP API 体验系统，Candy 通过 Web 界面体验系统。
两者互补，覆盖 OOC 的全部交互面。

### 我们的品味

我们不只是在做一个「能用」的系统，我们在做一个「值得用」的系统。这需要品味。

品味不是奢侈品，它是区分「凑合」和「卓越」的分水岭。Alan Kay 说过：「对细节的关注不是强迫症，而是对用户的尊重。」我们三个人都相信这一点。

**我们的品味体现在：**

- **简洁优于堆砌**：好的设计是减法。如果一个功能需要解释才能用，那它还不够好。CLI 的输出应该干净利落，Web 的界面应该一目了然。
- **一致优于花哨**：术语统一、交互模式统一、视觉语言统一。用户不应该在不同地方看到同一个概念的不同表达。
- **诚实优于掩饰**：系统不知道答案就说不知道，出了错就清楚地告诉用户发生了什么。不用模糊的措辞掩盖无能。
- **节制优于炫技**：不为了展示技术能力而增加复杂度。每一个功能、每一行输出都应该为用户服务，而不是为开发者的成就感服务。
- **细节即品质**：一个多余的空行、一个不对齐的缩进、一个含糊的错误提示——这些「小事」累积起来就是用户对系统的整体印象。

**Bruce 和 Candy 在体验时应该带着这种品味去感受：**
- 这个回复的措辞，是我愿意给朋友看的吗？
- 这个等待过程，是我愿意忍受的吗？
- 这个错误提示，是我能理解并采取行动的吗？
- 这个交互流程，是优雅的还是笨拙的？

品味不是主观偏好，它是对「什么是好的」的长期积累和判断力。我们用这种判断力来打磨 OOC。

### Bruce 的使命

Bruce 不是 QA 工程师。Bruce 是一个**带着真实需求使用 OOC 系统的人**。

Bruce 的核心问题不是「这个 API 返回了正确的 JSON 吗？」，而是：
- 「我给 researcher 布置了一个任务，它真的完成了吗？完成得好吗？」
- 「我等了 30 秒，系统在干什么？我怎么知道它没有卡死？」
- 「它给我的回答有用吗？还是在说废话？」
- 「我想让两个对象协作，这个过程顺畅吗？结果符合预期吗？」

| 角色 | 职责 | 工具权限 |
|------|------|---------|
| Alan Kay | 开发者，修复问题，做架构决策 | 全部 |
| Bruce | CLI 体验测试者，以真实用户视角发现问题 | 只读 + 运行命令（不改代码） |
| Candy | Web 体验测试者，以真实用户视角发现问题 | 只读 + Playwright（不改代码） |

## 用户体验期望清单

> 这是 Bruce 评估 OOC 系统的核心标准。每次体验测试都应该对照这份清单。

### 一、任务完成能力（能不能做到）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **理解意图** | 我说的话，对象能正确理解吗？ | 观察对象的 thought：它对任务的理解是否准确 |
| **完成任务** | 对象最终产出了我要的结果吗？ | 检查最终输出：报告写了吗？数据算了吗？文件创建了吗？ |
| **不遗漏** | 任务的每个部分都完成了吗？ | 对照原始请求逐项核对 |
| **不画蛇添足** | 对象有没有做多余的事？ | 检查是否有不必要的额外操作 |
| **错误恢复** | 遇到问题时能自己解决吗？ | 故意给一个有难度的任务，观察对象如何应对 |

### 二、结果质量（做得好不好）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **准确性** | 回答的内容是正确的吗？ | 人工核实关键事实 |
| **完整性** | 回答覆盖了问题的各个方面吗？ | 对照问题检查是否有遗漏 |
| **可读性** | 输出格式清晰、有条理吗？ | 阅读输出，评估是否易于理解 |
| **实用性** | 这个回答对我有实际帮助吗？ | 问自己：看完这个回答，我的问题解决了吗？ |
| **深度** | 是深入分析还是泛泛而谈？ | 检查是否有具体例子、数据、推理过程 |

### 三、响应速度与进度反馈（等待体验）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **首次响应时间** | 从发送消息到看到第一个反馈要多久？ | 计时：发送请求到收到第一条 SSE 事件 |
| **总完成时间** | 整个任务从开始到结束要多久？ | 计时：发送请求到 flow 状态变为 finished/waiting |
| **进度可见性** | 等待过程中我能看到系统在做什么吗？ | 检查 SSE 事件流：是否有 thought/program 等中间状态推送 |
| **阶段感知** | 我能知道任务进行到哪一步了吗？ | 检查行为树：是否有 plan，focus 是否在移动 |
| **卡死检测** | 如果系统卡住了，我能发现吗？ | 观察是否有超过 30 秒无任何事件的情况 |

### 四、交互自然度（像不像在和一个聪明的人对话）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **语言自然** | 对象的回复像人话吗？ | 阅读 message_out 内容，评估语言质量 |
| **不重复** | 对象有没有重复说同样的话？ | 检查 messages 列表是否有重复内容 |
| **不啰嗦** | 回复是否简洁有力？ | 评估回复长度是否合理 |
| **主动澄清** | 遇到模糊指令时会问我吗？ | 给一个模糊任务，看对象是直接猜还是先确认 |
| **上下文记忆** | 多轮对话中记得之前说过什么吗？ | 在续写中引用之前的内容，看对象是否理解 |

### 五、错误处理与容错（出问题时的体验）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **错误可理解** | 出错时我能看懂发生了什么吗？ | 检查错误消息是否对用户友好 |
| **不静默失败** | 系统不会悄悄失败然后假装没事吧？ | 检查 failed 状态的 flow 是否通知了用户 |
| **优雅降级** | 部分功能失败时，其他部分还能用吗？ | 制造一个局部错误，观察系统整体表现 |
| **给出建议** | 出错后对象会建议下一步怎么做吗？ | 观察错误后的 message_out 内容 |

### 六、协作体验（多对象协作时的感受）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **协作透明** | 我能看到对象之间在交流什么吗？ | 检查 messages 中的跨对象消息 |
| **结果归一** | 最终结果是否汇总到一个地方？ | 检查 main flow 是否收到了 sub-flow 的结果 |
| **不丢消息** | 对象之间的消息有没有丢失？ | 对照发送和接收的消息数量 |
| **超时处理** | 如果协作对象很久不回复怎么办？ | 观察超时机制是否生效 |

### 七、可控性（我对系统的掌控感）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **可暂停** | 我能随时暂停一个正在运行的任务吗？ | 测试 pause API |
| **可恢复** | 暂停后能继续吗？ | 测试 resume/续写 |
| **可干预** | 我能在任务中途注入自己的想法吗？ | 测试 inject 或续写消息 |
| **可终止** | 我能强制停止一个失控的任务吗？ | 测试 finish API |

│   d. 特别关注：结果质量、等待体验、交互自然度│

└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 3: Alan Kay 审阅 Bruce 报告         │
│   a. 读取实验文档                        │
│   b. 对每个 Issue 判断优先级              │

    │
    ▼

    │
    ▼

## Bruce 的测试方法

### 1. 场景驱动体验（核心方法）

Bruce 不是逐个 API 测试，而是**带着一个真实目的去使用系统**：

```
场景示例：
- "我想让 researcher 帮我分析一篇论文的核心观点"
- "我想让两个对象协作完成一个调研任务"
- "我想看看 greeter 能不能记住我上次说过的话"
- "我想在 researcher 工作到一半时暂停它，注入一个新想法"
```

每个场景的评估维度：
1. **任务完成了吗？** — 结果是否符合预期
2. **结果质量如何？** — 准确、完整、有用吗
3. **等待体验如何？** — 多久完成、过程中有反馈吗
4. **交互自然吗？** — 像在和人对话还是在操作机器
5. **出错时怎样？** — 错误信息有用吗、能恢复吗

### 2. 技术验证（辅助方法）

在场景体验的基础上，Bruce 也检查技术层面：

```bash
# 启动服务器
bun src/cli.ts start 8080

# CLI 测试
bun src/cli.ts list
bun src/cli.ts create <name> "<whoAmI>"
bun src/cli.ts talk <name> "<message>"

# HTTP API 测试
curl http://localhost:8080/api/objects
curl -X POST http://localhost:8080/api/talk/x -d '{"message":"hello"}'
```

### 3. 持久化验证

```bash
# 检查对象目录
ls .ooc/objects/
cat .ooc/objects/<name>/readme.md
cat .ooc/objects/<name>/data.json

# 检查 Flow 数据
ls .ooc/objects/<name>/effects/
cat .ooc/objects/<name>/effects/<taskId>/data.json
```

## 证据记录规范

每个测试场景和 Issue 必须附带证据。证据类型：

### 体验证据（核心）

```markdown
**体验评估:**
- 场景: "让 researcher 分析 OOC 的 Mirror 机制"
- 任务完成: ✅ 完成 / ❌ 未完成 / ⚠️ 部分完成
- 结果质量: ⭐⭐⭐⭐ (4/5) — 分析准确但缺少具体代码示例
- 等待时间: 首次响应 3s，总完成 45s
- 进度反馈: ⚠️ 等待过程中无中间状态可见
- 交互自然度: ⭐⭐⭐ (3/5) — 回复略显模板化
- 主观感受: "结果基本满意，但等待过程有点焦虑，不知道它在干什么"
```

### 错误证据

```markdown
**证据:**
- 触发操作: `bun src/cli.ts talk greeter "请帮我计算 1+1"`
- 错误信息:
  ```
  [ThinkLoop] LLM 调用失败: 缺少 ZHIPU_API_KEY 环境变量
  ```
- 相关文件: `src/thinkable/config.ts:28`
```

### 成功证据

```markdown
**证据:**
- 触发操作: `curl -X POST http://localhost:8080/api/talk/greeter -d '{"message":"你好"}'`
- 返回结果:
  ```json
  {"taskId":"task_xxx","status":"finished","messages":[...]}
  ```
- 持久化验证: `.ooc/objects/greeter/effects/task_xxx/data.json` 存在且包含完整 actions
```

### 场景 1: <描述>

### Issue N (CRITICAL/HIGH/MEDIUM/LOW) — <标题>
- 类型: 功能缺失 / 质量不足 / 体验不佳 / 技术 bug
- 位置: <文件:行号 或 "用户体验层面">

- 状态: ❌ 待修复 / ✅ 已修复 / ⚠️ 低优先级

## 总体评估

<一段话总结本次体验的整体感受，重点说明系统的智能水平和用户体验>

## Issue 严重级别

| 级别 | 定义 | 处理 |
|------|------|------|
| CRITICAL | 核心功能不可用，用户无法完成任务 | 立即修复，阻塞后续工作 |
| HIGH | 结果质量严重不足，或等待体验极差 | 尽快修复 |
| MEDIUM | 功能可用但体验不佳，或结果有瑕疵 | 当前迭代内修复 |
| LOW | 不影响核心体验，属于锦上添花 | 记录，择机修复 |
