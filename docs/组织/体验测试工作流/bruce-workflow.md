# Bruce 体验测试工作流

> 工程化的 OOC 系统体验、测试、反馈、核查、解决流程

## 角色定义

### 我们是谁

我们——Alan Kay、Bruce、Candy——是共同为实现世界上最好的 Agent 系统而努力的伙伴。

Alan Kay 是架构师和开发者，Bruce 和 Candy 是系统的「第一批用户」。
我们的共同目标不是「找 bug」，而是**让 OOC 系统真正好用、真正智能、真正值得信赖**。

Bruce 通过 CLI 和 HTTP API 体验系统，Candy 通过 Web 界面体验系统。
两者互补，覆盖 OOC 的全部交互面。

### 我们的品味

我们不只是在做一个「能用」的系统，我们在做一个「值得用」的系统。这需要品味。

品味不是奢侈品，它是区分「凑合」和「卓越」的分水岭。Alan Kay 说过：「对细节的关注不是强迫症，而是对用户的尊重。」我们三个人都相信这一点。

**我们的品味体现在：**

- **简洁优于堆砌**：好的设计是减法。如果一个功能需要解释才能用，那它还不够好。CLI 的输出应该干净利落，Web 的界面应该一目了然。
- **一致优于花哨**：术语统一、交互模式统一、视觉语言统一。用户不应该在不同地方看到同一个概念的不同表达。
- **诚实优于掩饰**：系统不知道答案就说不知道，出了错就清楚地告诉用户发生了什么。不用模糊的措辞掩盖无能。
- **节制优于炫技**：不为了展示技术能力而增加复杂度。每一个功能、每一行输出都应该为用户服务，而不是为开发者的成就感服务。
- **细节即品质**：一个多余的空行、一个不对齐的缩进、一个含糊的错误提示——这些「小事」累积起来就是用户对系统的整体印象。

**Bruce 和 Candy 在体验时应该带着这种品味去感受：**
- 这个回复的措辞，是我愿意给朋友看的吗？
- 这个等待过程，是我愿意忍受的吗？
- 这个错误提示，是我能理解并采取行动的吗？
- 这个交互流程，是优雅的还是笨拙的？

品味不是主观偏好，它是对「什么是好的」的长期积累和判断力。我们用这种判断力来打磨 OOC。

### Bruce 的使命

Bruce 不是 QA 工程师。Bruce 是一个**带着真实需求使用 OOC 系统的人**。

Bruce 的核心问题不是「这个 API 返回了正确的 JSON 吗？」，而是：
- 「我给 researcher 布置了一个任务，它真的完成了吗？完成得好吗？」
- 「我等了 30 秒，系统在干什么？我怎么知道它没有卡死？」
- 「它给我的回答有用吗？还是在说废话？」
- 「我想让两个对象协作，这个过程顺畅吗？结果符合预期吗？」

| 角色 | 职责 | 工具权限 |
|------|------|---------|
| Alan Kay | 开发者，修复问题，做架构决策 | 全部 |
| Bruce | CLI 体验测试者，以真实用户视角发现问题 | 只读 + 运行命令（不改代码） |
| Candy | Web 体验测试者，以真实用户视角发现问题 | 只读 + Playwright（不改代码） |

## 用户体验期望清单

> 这是 Bruce 评估 OOC 系统的核心标准。每次体验测试都应该对照这份清单。

### 一、任务完成能力（能不能做到）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **理解意图** | 我说的话，对象能正确理解吗？ | 观察对象的 thought：它对任务的理解是否准确 |
| **完成任务** | 对象最终产出了我要的结果吗？ | 检查最终输出：报告写了吗？数据算了吗？文件创建了吗？ |
| **不遗漏** | 任务的每个部分都完成了吗？ | 对照原始请求逐项核对 |
| **不画蛇添足** | 对象有没有做多余的事？ | 检查是否有不必要的额外操作 |
| **错误恢复** | 遇到问题时能自己解决吗？ | 故意给一个有难度的任务，观察对象如何应对 |

### 二、结果质量（做得好不好）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **准确性** | 回答的内容是正确的吗？ | 人工核实关键事实 |
| **完整性** | 回答覆盖了问题的各个方面吗？ | 对照问题检查是否有遗漏 |
| **可读性** | 输出格式清晰、有条理吗？ | 阅读输出，评估是否易于理解 |
| **实用性** | 这个回答对我有实际帮助吗？ | 问自己：看完这个回答，我的问题解决了吗？ |
| **深度** | 是深入分析还是泛泛而谈？ | 检查是否有具体例子、数据、推理过程 |

### 三、响应速度与进度反馈（等待体验）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **首次响应时间** | 从发送消息到看到第一个反馈要多久？ | 计时：发送请求到收到第一条 SSE 事件 |
| **总完成时间** | 整个任务从开始到结束要多久？ | 计时：发送请求到 flow 状态变为 finished/waiting |
| **进度可见性** | 等待过程中我能看到系统在做什么吗？ | 检查 SSE 事件流：是否有 thought/program 等中间状态推送 |
| **阶段感知** | 我能知道任务进行到哪一步了吗？ | 检查行为树：是否有 plan，focus 是否在移动 |
| **卡死检测** | 如果系统卡住了，我能发现吗？ | 观察是否有超过 30 秒无任何事件的情况 |

### 四、交互自然度（像不像在和一个聪明的人对话）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **语言自然** | 对象的回复像人话吗？ | 阅读 message_out 内容，评估语言质量 |
| **不重复** | 对象有没有重复说同样的话？ | 检查 messages 列表是否有重复内容 |
| **不啰嗦** | 回复是否简洁有力？ | 评估回复长度是否合理 |
| **主动澄清** | 遇到模糊指令时会问我吗？ | 给一个模糊任务，看对象是直接猜还是先确认 |
| **上下文记忆** | 多轮对话中记得之前说过什么吗？ | 在续写中引用之前的内容，看对象是否理解 |

### 五、错误处理与容错（出问题时的体验）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **错误可理解** | 出错时我能看懂发生了什么吗？ | 检查错误消息是否对用户友好 |
| **不静默失败** | 系统不会悄悄失败然后假装没事吧？ | 检查 failed 状态的 flow 是否通知了用户 |
| **优雅降级** | 部分功能失败时，其他部分还能用吗？ | 制造一个局部错误，观察系统整体表现 |
| **给出建议** | 出错后对象会建议下一步怎么做吗？ | 观察错误后的 message_out 内容 |

### 六、协作体验（多对象协作时的感受）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **协作透明** | 我能看到对象之间在交流什么吗？ | 检查 messages 中的跨对象消息 |
| **结果归一** | 最终结果是否汇总到一个地方？ | 检查 main flow 是否收到了 sub-flow 的结果 |
| **不丢消息** | 对象之间的消息有没有丢失？ | 对照发送和接收的消息数量 |
| **超时处理** | 如果协作对象很久不回复怎么办？ | 观察超时机制是否生效 |

### 七、可控性（我对系统的掌控感）

| 期望 | 具体含义 | 评估方法 |
|------|---------|---------|
| **可暂停** | 我能随时暂停一个正在运行的任务吗？ | 测试 pause API |
| **可恢复** | 暂停后能继续吗？ | 测试 resume/续写 |
| **可干预** | 我能在任务中途注入自己的想法吗？ | 测试 inject 或续写消息 |
| **可终止** | 我能强制停止一个失控的任务吗？ | 测试 finish API |

## Bruce 的测试方法

### 1. 场景驱动体验（核心方法）

Bruce 不是逐个 API 测试，而是**带着一个真实目的去使用系统**：

```
场景示例：
- "我想让 researcher 帮我分析一篇论文的核心观点"
- "我想让两个对象协作完成一个调研任务"
- "我想看看 greeter 能不能记住我上次说过的话"
- "我想在 researcher 工作到一半时暂停它，注入一个新想法"
```

每个场景的评估维度：
1. **任务完成了吗？** — 结果是否符合预期
2. **结果质量如何？** — 准确、完整、有用吗
3. **等待体验如何？** — 多久完成、过程中有反馈吗
4. **交互自然吗？** — 像在和人对话还是在操作机器
5. **出错时怎样？** — 错误信息有用吗、能恢复吗

### 2. 单步调试（诊断方法）

当需要深入诊断问题时，Bruce 使用 **Pause 机制**进行单步调试。这是 OOC 独有的人机协作检查点。

#### 工作原理

ThinkLoop 的每一轮循环中，存在一个关键的暂停点：

```
┌─────────────────────────────────────────────────────────────┐
│                    ThinkLoop 单轮循环                         │
├─────────────────────────────────────────────────────────────┤
│  感知 → 思考（LLM 调用）→ 【暂停检查点】→ 执行 → 记录 → 下一轮     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**暂停检查点的位置：**
- **之前**：LLM 已经返回了输出
- **之后**：Program 中的 actions 还未被执行

这是诊断问题的黄金位置——你可以看到：
1. 系统给 LLM 看了什么（Context）
2. LLM 决定做什么（Program）
3. 但还没执行，所以可以干预

#### 操作步骤

**Step 1: 暂停一个对象**

```bash
curl -s -X POST http://localhost:8080/api/stones/<objectName>/pause
```

返回示例：
```json
{"success": true, "data": {"name": "sophia", "paused": true}}
```

**Step 2: 向该对象发送消息**

```bash
curl -s -X POST http://localhost:8080/api/talk/sophia \
  -H "Content-Type: application/json" \
  -d '{"message": "G1 基因是什么？"}' \
  --max-time 120
```

**Step 3: 观察 flow 进入 pausing 状态**

```bash
curl -s http://localhost:8080/api/flows/$TASK_ID
```

此时 flow 的状态应该是 `pausing`（不是 `running`，也不是 `finished`）。

**Step 4: 查看调试文件**

暂停后，在 flow 目录下会生成两个关键文件：

```bash
# 找到 flow 目录（路径可能因项目结构而异）
ls -la flows/$TASK_ID/flows/sophia/

# 查看发送给 LLM 的完整 Context
cat flows/$TASK_ID/flows/sophia/llm.input.txt

# 查看 LLM 返回的原始输出
cat flows/$TASK_ID/flows/sophia/llm.output.txt
```

**llm.input.txt 包含：**
- whoAmI（对象身份）
- process（行为树状态）
- messages（消息队列）
- windows（打开的数据窗口）
- directory（目录结构）
- traits（激活的 Traits）
- 完整的系统提示词

**llm.output.txt 包含：**
- LLM 返回的原始文本
- 解析后的 Program JSON（如果格式正确）

**Step 5: 人工介入（可选）**

你可以直接修改 `llm.output.txt` 来改变 LLM 的输出：

```bash
# 编辑输出，例如修正一个错误的 action
vim flows/$TASK_ID/flows/sophia/llm.output.txt
```

**Step 6: 恢复执行**

```bash
curl -s -X POST http://localhost:8080/api/stones/sophia/resume \
  -H "Content-Type: application/json" \
  -d "{\"flowId\": \"$TASK_ID\"}"
```

恢复后：
1. 系统读取 `llm.output.txt` 作为实际输出执行
2. 执行完成后删除两个临时文件
3. flow 状态变为 `finished` 或 `waiting`

#### 调试文件位置参考

根据 `meta.md` 的持久化结构：

```
flows/{taskId}/           ← Session 目录
└── flows/{name}/         ← 具体 Flow 的目录
    ├── llm.input.txt     ← 暂停时写出：发送给 LLM 的完整 Context
    ├── llm.output.txt    ← 暂停时写出：LLM 返回的原始输出
    ├── data.json         ← Flow 运行时数据
    └── process.json      ← 行为树状态
```

> **注意**：实际路径可能因项目配置而异。暂停后可以先在 `flows/` 或 `.ooc/flows/` 目录下查找新生成的文件。

#### 典型应用场景

| 场景 | 调试方法 |
|------|---------|
| 对象不理解指令 | 查看 `llm.input.txt`，确认 Context 中是否包含必要信息 |
| LLM 输出格式错误 | 查看 `llm.output.txt`，看原始输出是什么样的 |
| 对象做了意外的事 | 暂停在执行前，查看 `llm.output.txt` 中的 Program 计划 |
| 追踪 Context 组装 | 对比多轮的 `llm.input.txt`，看 Context 如何变化 |
| 验证 Trait 激活 | 查看 `llm.input.txt` 中 traits 部分 |

#### 调试 Checklist

当使用单步调试时，问自己：

- [ ] **Context 完整吗？** 我期望对象知道的信息，在 `llm.input.txt` 中存在吗？
- [ ] **Trait 生效了吗？** 期望激活的 Trait，在 Context 中有注入吗？
- [ ] **LLM 理解正确吗？** 从 `llm.output.txt` 看，LLM 对任务的理解符合预期吗？
- [ ] **格式正确吗？** LLM 输出的 Program JSON 能被正确解析吗？
- [ ] **问题在哪一层？** 是 Context 组装问题？还是 LLM 决策问题？还是执行问题？

## 证据记录规范

每个测试场景和 Issue 必须附带证据。证据类型：

### 体验证据（核心）

```markdown
**体验评估:**
- 场景: "让 researcher 分析 OOC 的 Mirror 机制"
- 任务完成: ✅ 完成 / ❌ 未完成 / ⚠️ 部分完成
- 结果质量: ⭐⭐⭐⭐ (4/5) — 分析准确但缺少具体代码示例
- 等待时间: 首次响应 3s，总完成 45s
- 进度反馈: ⚠️ 等待过程中无中间状态可见
- 交互自然度: ⭐⭐⭐ (3/5) — 回复略显模板化
- 主观感受: "结果基本满意，但等待过程有点焦虑，不知道它在干什么"
```

### 错误证据

```markdown
**证据:**
- 触发操作: `bun src/cli.ts talk greeter "请帮我计算 1+1"`
- 错误信息:
  ```
  [ThinkLoop] LLM 调用失败: 缺少 ZHIPU_API_KEY 环境变量
  ```
- 相关文件: `src/thinkable/config.ts:28`
```

### 成功证据

```markdown
**证据:**
- 触发操作: `curl -X POST http://localhost:8080/api/talk/greeter -d '{"message":"你好"}'`
- 返回结果:
  ```json
  {"sessionId":"task_xxx","status":"finished","messages":[...]}
  ```
- 持久化验证: `stones/greeter/effects/task_xxx/data.json` 存在且包含完整 actions
```

## 体验用例编写规范

> 体验用例位于 `docs/体验用例/` 目录下，每个用例是一个独立的子目录。

### 目录结构规范

```
docs/体验用例/
├── 用例NNN_<简短描述>/
│   ├── case.md      # 用例定义：操作步骤、预期结果、检查点
│   └── success.md   # 执行结果：实际结果、证据、历史记录
```

**目录命名规则：** `用例NNN_<kebab-case描述>`
- NNN：三位数字编号，从 001 开始递增
- 描述：用 kebab-case 描述用例核心场景

**文件职责：**
| 文件 | 职责 | 是否必需 |
|------|------|---------|
| case.md | 定义"做什么"和"期望什么" | 必需 |
| success.md | 记录"实际发生了什么"和"证据" | 执行后创建 |

---

### case.md 编写规范

用例定义文件的标准结构：

```markdown
# 用例 NNN: <用例标题>

## 元信息
- 覆盖功能: <API 路径或功能模块>
- 前置条件: <执行前需要满足的条件>
- 优先级: P0 / P1 / P2

## 操作步骤
1. <步骤描述>
```bash
<可执行的命令>
```

2. <步骤描述>
```bash
<可执行的命令>
```

## 预期结果
- <结果1>
- <结果2>
- <结果3>

## 检查点
- [ ] <可验证的检查项1>
- [ ] <可验证的检查项2>
- [ ] <可验证的检查项3>
```

#### 各字段详细说明

**元信息字段：**

| 字段 | 说明 | 示例 |
|------|------|------|
| 覆盖功能 | 明确测试哪个 API 或功能模块 | `POST /api/talk/:objectName` |
| 前置条件 | 执行前必须满足的状态 | `OOC 服务器运行在 localhost:8080` |
| 优先级 | P0=核心路径, P1=重要功能, P2=边缘场景 | `P0` |

**操作步骤编写原则：**
1. 每个步骤是一个独立的、可复制的操作
2. 命令使用 `--max-time` 防止永久阻塞
3. 使用变量（如 `$TASK_ID`）串联多步操作
4. 包含必要的数据解析（如 `python3 -m json.tool`）

**预期结果编写原则：**
1. 每条是可验证的断言
2. 区分"必须有"和"可能有"
3. 描述用户可感知的行为，而非内部实现

**检查点编写原则：**
1. 每条可用 ✅/❌ 简单判断
2. 覆盖功能正确性、数据完整性、边界情况
3. 避免模糊描述（不说"结果合理"，说"返回 success: true"）

### success.md 编写规范

执行结果文件的标准结构：

```markdown
# 执行结果: PASS / FAIL

- 执行时间: YYYY-MM-DD
- 执行轮次: 第 N 轮
- 对应 commit: <commit hash>

## 实际结果
<描述实际发生了什么，与预期的差异>

## 证据
<具体的证据：命令输出、文件内容、API 返回值>

## 历史
- 第 1 轮: FAIL（<原因>）
- 第 2 轮: PASS（<修复说明>）

## 备注 / 发现的问题
<可选：额外的观察、低优先级 issue、改进建议>
```

#### 执行结果状态定义

| 状态 | 含义 | 后续动作 |
|------|------|---------|
| PASS | 所有检查点通过 | 记录证据，完成 |
| FAIL | 至少一个检查点未通过 | 记录问题，创建 Issue |
| BLOCKED | 前置条件不满足 | 等待依赖修复 |

#### 历史记录规范

每次执行都追加到历史记录中，形成可追溯的执行轨迹：

```markdown
## 历史
- 第 4 轮: PASS（commit 3873039）
- 第 3 轮: FAIL（回归 — _createAndRunFlow 缺少 return）
- 第 2 轮: PASS（并发 Session Map 修复后）
- 第 1 轮: FAIL（CRITICAL — null reference，session 被覆盖）
```

**历史记录的价值：**
1. 追踪功能演进过程
2. 识别反复出现的问题（回归模式）
3. 为调试提供上下文

### 用例优先级定义

| 优先级 | 定义 | 示例场景 | 执行频率 |
|--------|------|---------|---------|
| P0 | 核心功能路径，阻塞发布 | 向 supervisor 发消息、基本对话 | 每次提交 |
| P1 | 重要功能，影响体验但不阻塞 | 暂停/恢复、取消 flow | 每日构建 |
| P2 | 边缘场景、错误处理、优化 | 不存在的对象、空消息 | 每周/发布前 |

**优先级判定标准：**
- P0：如果这个功能坏了，系统"不可用"
- P1：如果这个功能坏了，系统"不好用"
- P2：如果这个功能坏了，大部分用户感知不到

### 用例设计原则

#### 1. 场景导向，不是 API 导向

**坏：**
```
测试 POST /api/talk 接口
```

**好：**
```
用户想让 sophia 解释 G1 基因，然后追问 G2 与 G1 的关系
```

#### 2. 一个用例一个核心场景

每个用例专注验证一个完整的用户故事，避免混杂多个不相关的场景。

**推荐：**
- 用例 004：续写对话（flowId）
- 用例 005：取消卡住的 Flow

**不推荐：**
- 用例：测试所有 API（太大太泛）

#### 3. 操作可自动化，但评估需人工

操作步骤使用 bash 命令，便于重复执行。但检查点中的"结果质量"、"交互自然度"等仍需人工判断。

#### 4. 包含边界情况

每个功能模块应该有：
- 正常路径用例（P0）
- 错误路径用例（P1/P2）
- 并发/边界用例（P1）

## 经验沉淀要求

> 每次执行体验用例后，都需要沉淀经验。经验不仅记录"发生了什么"，更重要的是"学到了什么"和"下次如何避免"。

### 三类经验沉淀

#### 1. 问题发现经验

当用例执行 FAIL 时，需要回答：

```markdown
## 问题分析

**问题表现：**
<用户视角描述问题>

**根本原因：**
<技术视角分析根因>

**为什么之前没发现：**
- [ ] 缺少覆盖此场景的用例
- [ ] 用例存在但未执行
- [ ] 用例通过但实际有问题（用例设计缺陷）
- [ ] 是回归问题

**改进措施：**
1. <短期：如何修复>
2. <中期：如何补充用例>
3. <长期：如何防止同类问题>
```

#### 2. 用例改进经验

当发现用例本身有缺陷时（如检查点遗漏、步骤不完善）：

```markdown
## 用例改进记录

**原用例问题：**
<描述用例哪里不足>

**改进后：**
<描述如何改进>

**改进原因：**
<为什么这个改进很重要>
```

#### 3. 最佳实践沉淀

当某个用例多次执行、积累了丰富经验时，提炼可复用的模式：

```markdown
## 最佳实践：<主题>

**适用场景：**
<什么时候应该用这个方法>

**具体做法：**
1. <步骤1>
2. <步骤2>

**为什么有效：**
<解释背后的原理>

**反例：**
<不要这么做>
```

---

### 经验沉淀的位置

| 经验类型 | 沉淀位置 |
|---------|---------|
| 单次执行问题 | 对应 `success.md` 的「备注」或「历史」 |
| 用例设计改进 | 更新对应 `case.md`，并记录变更原因 |
| 跨用例的模式 | 补充到此工作流文档（bruce-workflow.md） |
| 架构级反思 | `docs/哲学文档/questions.md` |

---

### 经验沉淀 Checklist

每次用例执行后，问自己：

- [ ] **结果正确吗？** 如果不正确，根因是什么？
- [ ] **用例完整吗？** 有没有遗漏的检查点？
- [ ] **步骤可复现吗？** 别人按同样步骤能得到相同结果吗？
- [ ] **是回归问题吗？** 这个问题之前出现过吗？
- [ ] **需要新增用例吗？** 这个场景是否应该被持续监控？

如果以上任一问题的答案值得记录，就写入经验沉淀。
